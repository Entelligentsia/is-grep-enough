# L4 reference answer key — the template/layout lookup subsystem (the "layout lookup rules")

Pinned source: `experiment/repos/hugo` (SHA d15baf53…, Go). All cites re-verified.

A bounded cluster in `tpl/tplimpl/` (with the page-side descriptor builder in `hugolib/page.go`). This is
the inside of the `LookupPagesLayout` black box that L3's `resolveTemplate` delegates to. Several
cooperating pieces, not a single path: descriptor/query construction → the store tree → path-walk scoring →
baseof resolution.

## Required spine (gradeable core — must hit all four pieces)

### Piece 1 — query / descriptor construction (page side)

  `tplimpl.TemplateDescriptor{Kind, LayoutFromUser, OutputFormat, MediaType, IsPlainText}`

- `TemplateStore.findBestMatchWalkPath` `templatestore.go:858`: `s.treeMain.WalkPath(k1, ...)`

## Common misses / wrong turns

- Treating it as one function (`LookupPagesLayout`) rather than a cluster of four cooperating pieces.
- Stopping at the page-side `resolveTemplate` (`page.go:630`) and never entering `TemplateStore` internals —
  that is L3's view, not L4.
- Confusing `CategoryLayout` (the overlay/layout pass) with `CategoryBaseof` (the base-template pass) — they
  are two related but distinct passes (`LookupPagesLayout` does layout, then `findBestMatchBaseof` does
  baseof).
- Missing the `weight.distance` (path proximity) component — scoring is *descriptor match plus path
  distance*, not descriptor alone (`templatestore.go:877`, `:344`).
- Pointing at the `output.Format`/layout-name string conventions instead of the actual scoring machinery.

## Completeness scale

- **Full:** all four pieces — descriptor/query construction (page.go side), the store tree +
  `LookupPagesLayout` orchestration, the path-walk scoring (`findBestMatchWalkPath` + `compareDescriptors` +
  `weight.distance`), and the baseof pass (`findBestMatchBaseof`) — with cites.
- **Partial:** 2-3 of the four pieces, or the store + entry without the scoring internals or baseof.
- **Miss:** only `resolveTemplate`, or only naming "layout lookup rules" without the machinery.

## Model answer

When a page is rendered it builds a query: `pageOutput.GetInternalTemplateBasePathAndDescriptor`
(`hugolib/page.go:616`) fills a `TemplateDescriptor{Kind, LayoutFromUser, OutputFormat, MediaType,
IsPlainText}` (`:621-626`) and `pageState.resolveTemplate` (`:630`) wraps it in a
`TemplateQuery{Category: CategoryLayout, Sites, Desc}` (`:638-643`) and calls
`TemplateStore.LookupPagesLayout(q)` (`:645`). The store (`tpl/tplimpl/templatestore.go:434`) keeps every
template in `treeMain` keyed by path with a `nodeKey` carrying category + `TemplateDescriptor`; categories are
`CategoryLayout`/`CategoryBaseof`/… (`:65-72`). `LookupPagesLayout` (`:584`) initializes the query, derives a
path key, and runs `findBestMatchWalkPath` (`:858`), which walks `treeMain.WalkPath` and, for each candidate
matching the category, scores it with `descriptorHandler.compareDescriptors` (`templatedescriptor.go:66` →
`doCompare` `:100`) and adds a `weight.distance` = path-slash proximity (`:877`), keeping the best in a
`bestMatch` (`:2016`, `weight` `:2143`). If the best template has a base template (`!noBaseOf`),
`TemplInfo.findBestMatchBaseof` (`:328`) walks `baseVariants` and scores candidates with
`CategoryBaseof` (`:342`, distance `:344`) to pick the base, producing a `TemplWithBaseApplied`
(overlay+base+final). The chosen `TemplInfo` is returned to the renderer. So the "layout lookup rules" are
realized as descriptor construction → tree path-walk with weighted descriptor+distance scoring → a separate
baseof pass, all organized in the `TemplateStore`.
