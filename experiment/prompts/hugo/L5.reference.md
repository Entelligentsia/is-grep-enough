# L5 reference answer key — incremental rebuild via file-watch + identity tracking

Pinned source: `experiment/repos/hugo` (SHA d15baf53…, Go). All cites re-verified.

A cross-cutting flow threading the filesystem watcher, the `WhatChanged` change-bag, the `identity` package,
`resource.Staler`/`MemCache` invalidation, template refresh, and the partial assemble/render phases. This is
the **rebuild** branch of `process` (the `len(events) > 0` path), distinct from L3's full-build branch.

## Common misses / wrong turns

- Stopping at event classification and never reaching the identity-based invalidation + stale-marking.
- Missing the feedback into `assemble` (the `needsPagesAssembly` gate that decides full vs partial
  reassembly) — the prompt says "how that feeds back into the partial assemble and render phases."
- Conflating `processPartialFileEvents` (the per-fs-event classifier) with `processPartialRebuildChanges`
  (the remote/programmatic-change path).
- Crediting the debug `WalkIdentitiesDeep` log (`content_map_page.go:921`) as the invalidation mechanism —
  the real invalidation is `MarkStale` (`:943`) + `MemCache.ClearOnRebuild` (`:962`) + the evicted-identity
  drain/`identity.NewFinder.Contains` pass (`:977-1005`).
- Treating `WhatChanged` as a plain list of files — it carries identity keys, structural sentinels, and a
  coarse `GenghisKhan` fallback.
- Forgetting that template changes need an explicit `TemplateStore.RefreshFiles` pass — stale-marking pages
  does not reload templates.

## Completeness scale

- **Full:** the whole flow — event intake → per-component classification into identity changes →
  `WhatChanged` bundling → onChange notify + `resolveAndClearStateForIdentities` (`MarkStale` +
  `MemCache.ClearOnRebuild` + evicted-identity drain/`identity.NewFinder`) → template/i18n refresh branches
  → feedback into `assemble` (`needsPagesAssembly` gate) and `render`, with cites.
- **Partial:** covers classification + invalidation but misses the assemble feedback or the template refresh
  branch, or credits the debug `WalkIdentitiesDeep` as invalidation.
- **Miss:** only "there is a watcher," or only naming "hugo rebuilds changed files" without the
  identity/stale mechanism.

## Model answer

When `hugo server` is running and a file changes, the watcher hands `fsnotify.Event`s to
`HugoSites.processPartialFileEvents` (`hugo_sites_build.go:865`). It filters/trims the events, resolves each
to a component path, and — in `handleChange` (`:967`) — classifies per source component into a
`changes []identity.Identity` list: content changes call `pageTrees.collectAndMarkStaleIdentities`
(`content_map_page.go:143`) and set `needsPagesAssemble` (`hugo_sites_build.go:999`); layout changes set
`tmplChanged` and (for new `_markup`/`shortcode` templates) append the coarse `identity.GenghisKhan`
(`:1047`, `identity/identity.go:35`); data/i18n changes append `siteidentities.Data` / `GenghisKhan`
(`:1073`/`:1080`); deletes append structural add/remove sentinels (`identity.go:37-38`,
`:948`/`:951`) and predicate identities for removed dirs (`:1120`). The changes are bundled into a
`WhatChanged` (`site.go:1133`) via `changed.Add(changes...)` and assigned to `config.WhatChanged`
(`hugo_sites_build.go:1177-1182`). After notifying `OnChangeListeners` (`:1201`),
`resolveAndClearStateForIdentities` (`content_map_page.go:900`) marks every changed `resource.Staler` stale
(`MarkStale()` `:943`, `resources/resource/resourcetypes.go:220`), GCs the dynacache
(`MemCache.ClearOnRebuild` `:962`), then drains evicted identities and pulls in related ones with
`identity.NewFinder` + `depsFinder.Contains` (`:977-1005`), collapsing to `GenghisKhan` on mass eviction
(`:1005`). If templates changed, `TemplateStore.RefreshFiles` reloads the affected template files
(`:1212`); if i18n changed, `Deps` are recompiled (`:1235`). Control then returns to `Build`'s phase loop:
`assemble` (`:302`) checks `WhatChanged.needsPagesAssembly` (`:306`) to decide between a full re-assembly
and a targeted invalidation, after which `render` (`:399`) re-renders the affected pages. So a file event
becomes identity changes → stale + cache invalidation → template refresh → partial assemble/render,
gluing the watcher, the `identity` system, the resource cache, and the build phases together.
