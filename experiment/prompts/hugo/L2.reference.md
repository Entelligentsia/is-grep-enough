# L2 reference answer key — the build-eligibility gate (`shouldBuild`) + its callers

Pinned source: `experiment/repos/hugo` (SHA d15baf53…, Go). All cites re-verified.

One focal predicate plus two real assembly-time callers that react to it differently. A star, not a path.

## Required spine (gradeable core — must hit all)

1. **Focal: `(s *Site) shouldBuild(p page.Page) bool`** — `hugolib/site.go:1763`. Composes an inner kind gate
   `s.conf.IsKindEnabled(p.Kind())` (`site.go:1764`; returns false if the kind is disabled) with the date/draft
   helper `shouldBuild(...)` (`site.go:1767`).
2. **Helper: `shouldBuild(buildFuture, buildExpired, buildDrafts, Draft, publishDate, expiryDate) bool`** —
   `hugolib/site.go:1771`. The load-bearing rules:
   - drafts: `if !(buildDrafts || !Draft)` return false (`site.go:1774-1776`) — a draft page is built only when
     the site's `buildDrafts` flag is on;
   - future: `hnow := htime.Now()` (`site.go:1777`); `if !buildFuture && !publishDate.IsZero() &&
     publishDate.After(hnow)` return false (`site.go:1778-1780`);
   - expired: `if !buildExpired && !expiryDate.IsZero() && expiryDate.Before(hnow)` return false
     (`site.go:1781-1783`);
   - else true (`site.go:1784`).
   Note the three site-config flags (`buildDrafts`/`buildFuture`/`buildExpired`, from `s.Conf`) are distinct
   from the three page-supplied values (`Draft`, `PublishDate`, `ExpiryDate`).
3. **Caller A — page creation walk, `content_map_page_assembler.go:276`** — when building a page from its meta
   source, `if !site.shouldBuild(p)` → `(&p.m.pageConfig.Build).Disable()` (`:278`) then
   `switch p.Kind()` (`:279`): for `KindHome`/`KindSection`/`KindTaxonomy` the page is *kept* (comment "We need
   to keep these for the structure", `:281-282`); for every other kind it is *dropped* by appending its path to
   `a.droppedPages` (`:283-291`). So a failing structural kind stays as scaffolding; a failing regular page is
   discarded.
4. **Caller B — taxonomy-term "dates" pass, `content_map_page_assembler.go:1088`** — for a `KindTerm` page,
   `if !p.s.shouldBuild(p)` (`:1088`) → `sa.s.pageMap.treePages.Delete(s)` (`:1089`) and
   `sa.s.pageMap.treeTaxonomyEntries.DeletePrefix(paths.AddTrailingSlash(s))` (`:1090`); the term is
   deleted from the pages tree *and* its whole taxonomy-entry subtree is prefix-pruned. If it *passes*, the
   `else` branch walks the term's taxonomy entries to propagate date info up the tree (`:1091-1101`). The
   two callers thus do materially different things with the same boolean.

## Common misses / wrong turns

- Pointing only at `IsKindEnabled` and ignoring the draft/future/expiry date comparisons — the date logic is
  the load-bearing part of the focal and the prompt names "draft / future-dated / expired."
- Stopping at the focal definition and not reading the two callers' differing reactions — the gradeable
  synthesis is exactly "what the different callers do when a page fails it."
- Conflating the two callers' "drop" semantics: caller A *keeps* structure kinds (home/section/taxonomy)
  even when failing; caller B *deletes* the failing term and prunes its taxonomy subtree.
- Treating `buildDrafts`/`buildFuture`/`buildExpired` as page fields — they are site-config flags; only
  `Draft`/`PublishDate`/`ExpiryDate` come from the page.

## Model answer

Hugo's per-page build-eligibility gate is `(s *Site) shouldBuild` (`hugolib/site.go:1763`). It first checks the
kind is enabled via `s.conf.IsKindEnabled(p.Kind())` (`:1764`), then delegates to the helper `shouldBuild`
(`:1771`), which applies three rules against the site config flags `buildDrafts`/`buildFuture`/`buildExpired`
and the page's `Draft`/`PublishDate`/`ExpiryDate`: drafts are dropped unless `buildDrafts` (`:1774-1776`);
pages with a future `PublishDate` are dropped unless `buildFuture` (`:1778-1780`); pages with a past
`ExpiryDate` are dropped unless `buildExpired` (`:1781-1783`). Two assembly sites consult it. During page
creation (`content_map_page_assembler.go:276`) a failing page has its `Build` config disabled and is then
either kept — for `KindHome`/`KindSection`/`KindTaxonomy`, which are retained as structural scaffolding
(`:281-282`) — or recorded in `a.droppedPages` for any other kind (`:283-291`). During the taxonomy-term
"dates" pass (`content_map_page_assembler.go:1088`) a failing `KindTerm` is deleted from `treePages` and its
`treeTaxonomyEntries` subtree is prefix-pruned (`:1089-1090`); a passing term instead walks its entries to
propagate dates upward (`:1092-1104`). So the same boolean triggers keep-vs-drop at creation and
delete-and-prune for terms.
