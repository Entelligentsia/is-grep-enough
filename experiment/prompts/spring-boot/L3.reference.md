# L3 reference answer key — `SpringApplication.run` startup sequence (ordered chain across files)

Pinned source: `experiment/repos/spring-boot` (SHA e16c5d01…). All cites re-verified.

A single ordered chain threaded through `SpringApplication.java`, the composite
`SpringApplicationRunListeners.java`, the built-in `EventPublishingRunListener.java`, the
`EnvironmentPostProcessorApplicationListener.java`, and the runner interfaces. Order is gradeable.

## Required spine (gradeable core — must hit all, in order)

1. **Entry `run`** — `SpringApplication.java:304` (`public ConfigurableApplicationContext run(String...)`).
   It enables the shutdown hook (if configured), builds the bootstrap context, and obtains the run
   listeners.
2. **Obtain run listeners + `starting`** — `getRunListeners(args)` (`SpringApplication.java:312`, def
   `:453`) builds a `SpringApplicationRunListeners` composite from `SpringApplicationRunListener`
   factories; then `listeners.starting(bootstrapContext, mainApplicationClass)` (`:313`). The
   composite fans out via `doWithListeners` — `starting` at `SpringApplicationRunListeners.java:55`;
   the built-in `EventPublishingRunListener.starting` (`EventPublishingRunListener.java:75`)
   publishes `ApplicationStartingEvent` (`:76`).
3. **`prepareEnvironment` + `environmentPrepared` (the cross-file hop to env post-processors)** —
   `prepareEnvironment(...)` called at `SpringApplication.java:316` (def `:350`); it creates the
   environment (`getOrCreateEnvironment` `:353`), attaches `ConfigurationPropertySources` (`:355`),
   then `listeners.environmentPrepared(bootstrapContext, environment)` (`:356`). Composite
   `environmentPrepared` (`SpringApplicationRunListeners.java:64`) → `EventPublishingRunListener
   .environmentPrepared` (`EventPublishingRunListener.java:80`) publishes
   `ApplicationEnvironmentPreparedEvent` (`:83`). That event is consumed by
   `EnvironmentPostProcessorApplicationListener.onApplicationEvent` (`:118`) →
   `onApplicationEnvironmentPreparedEvent` (`:130`) which loops each
   `EnvironmentPostProcessor.postProcessEnvironment(environment, application)` (`:136-137`). Control
   returns to `prepareEnvironment`, which re-attaches `ConfigurationPropertySources` (`:366`) and
   returns the environment.
4. **`printBanner`** — `SpringApplication.java:317`.
5. **`createApplicationContext`** — `SpringApplication.java:318` (def `:579`), delegating to
   `ApplicationContextFactory.create(webApplicationType)`.
6. **`prepareContext`** — `SpringApplication.java:320` (def `:380`). Sets the environment, calls
   `applyInitializers(context)`, `listeners.contextPrepared(context)` (composite
   `SpringApplicationRunListeners.java:69` → `EventPublishingRunListener.contextPrepared`
   `EventPublishingRunListener.java:87`, publishing `ApplicationContextInitializedEvent` `:88`),
   registers singleton beans (`springApplicationArguments`, banner), then `load(context, sources)`
   and `listeners.contextLoaded(context)` (composite `:73` →
   `EventPublishingRunListener.contextLoaded` `:92`, publishing `ApplicationPreparedEvent` `:99`).
7. **`refreshContext`** — `SpringApplication.java:321` (def `:441`): registers the context with the
   shutdown hook (`:442`) then `refresh(context)` (`:443`) — the actual Spring container refresh.
8. **`afterRefresh`** — `SpringApplication.java:322` (def `:764`, an overridable no-op hook).
9. **`started`** — `listeners.started(context, timeTakenToStarted)` (`SpringApplication.java:327`);
   composite `SpringApplicationRunListeners.java:77` → `EventPublishingRunListener.started`
   (`:103`) publishes `ApplicationStartedEvent` (`:104`) on the context.
10. **`callRunners`** — `SpringApplication.java:328` (def `:767`): collects beans of type `Runner`,
    orders them, and `callRunner(runner, args)` (`:786`) dispatches to `ApplicationRunner.run(args)`
    (`:788`, interface `ApplicationRunner.java:33`/`run` `:40`) and/or
    `CommandLineRunner.run(args.getSourceArgs())` (`:791`, interface `CommandLineRunner.java:36`/
    `run` `:43`).
11. **`ready`** — `listeners.ready(context, startup.ready())` (`SpringApplication.java:335`);
    composite `SpringApplicationRunListeners.java:81` → `EventPublishingRunListener.ready`
    (`:109`) publishes `ApplicationReadyEvent` (`:110`). Failure at any guarded step routes through
    `handleRunFailure` (`:331`/`:339`, def `:803`).

## Acceptable extras (credit, never required)

- `AotDetector.useGeneratedArtifacts()` short-circuit in `prepareContext` (skips `load(sources)`
  under AOT) — bonus.
- The `SpringApplicationHook` run listener appended in `getRunListeners` (`SpringApplication.java:457-460`).
- `DefaultBootstrapContext` creation and `bootstrapRegistryInitializers` (`:344`), and
  `bootstrapContext.close(context)` inside `prepareContext`.
- The `Startup`/`StartupStep` instrumentation threaded through `doWithListeners`
  (`SpringApplicationRunListeners.java` `doWithListeners` ~`:98`).
- `bindToSpringApplication(environment)` and environment conversion in `prepareEnvironment`
  (`:361-365`).

## Common misses / wrong turns

- Jumping straight to `createApplicationContext`/`refreshContext` and skipping the
  `starting`/`environmentPrepared` front end (spine 2-3) — the prompt asks "from the moment `run` is
  invoked."
- Missing the cross-file hop from `environmentPrepared` to the `EnvironmentPostProcessor` dispatch
  (composite → `EventPublishingRunListener` event → `EnvironmentPostProcessorApplicationListener`)
  — that is the load-bearing multi-file link at this level.
- Treating the run listeners as a single black box instead of naming the composite
  (`SpringApplicationRunListeners`) AND the built-in event publisher
  (`EventPublishingRunListener`) — they are two distinct classes.
- Stopping at `refreshContext` and omitting the post-refresh tail (`started` → `callRunners` →
  `ready`), or confusing `started` (event) with `ready` (event).
- Naming `callRunners` but not connecting it to the `ApplicationRunner`/`CommandLineRunner`
  interfaces (`callRunner` dispatch `:786-792`).

## Completeness scale

- **Full:** all 11 spine stops, in order, run → listeners.starting → prepareEnvironment(env post
  processors via the event hop) → createApplicationContext → prepareContext(contextPrepared/load/
  contextLoaded) → refreshContext → afterRefresh → started → callRunners(ApplicationRunner/
  CommandLineRunner) → ready, with the composite↔EventPublishingRunListener↔listener hop named, with
  cites.
- **Partial:** ≥7 of 11 in roughly correct order, but missing either the environment-prepared
  cross-file hop OR the post-refresh runner tail, or treating listeners as one class.
- **Miss:** only the first few phases with no listener wiring, or an out-of-order/incorrect chain.

## Model answer

`SpringApplication.run` (`SpringApplication.java:304`) first gets the run listeners
(`getRunListeners` `:312`/def `:453`) and fires `listeners.starting` (`:313`; composite
`SpringApplicationRunListeners.java:55`; `EventPublishingRunListener.starting` `:75` publishes
`ApplicationStartingEvent`). It then `prepareEnvironment` (`:316`, def `:350`): creates and configures
the environment, attaches `ConfigurationPropertySources` (`:355`), and fires
`listeners.environmentPrepared` (`:356`; composite `SpringApplicationRunListeners.java:64`; the
built-in `EventPublishingRunListener.environmentPrepared` `:80` publishes
`ApplicationEnvironmentPreparedEvent` `:83`), which `EnvironmentPostProcessorApplicationListener`
consumes (`:118`→`:130`) to run each `EnvironmentPostProcessor.postProcessEnvironment` (`:136`). Back
in `run`, `printBanner` (`:317`), `createApplicationContext` (`:318`/def `:579`), then `prepareContext`
(`:320`/def `:380`) applies initializers, fires `listeners.contextPrepared` (composite `:69`;
`EventPublishingRunListener.contextPrepared` `:87` → `ApplicationContextInitializedEvent` `:88`),
loads sources, and fires `listeners.contextLoaded` (composite `:73`; `:92` →
`ApplicationPreparedEvent` `:99`). Then `refreshContext` (`:321`/def `:441`) registers the shutdown hook
(`:442`) and `refresh(context)` (`:443`), followed by the overridable `afterRefresh` (`:322`/def
`:764`). After refresh, `listeners.started` (`:327`; composite `:77`; `:103` →
`ApplicationStartedEvent` `:104`), `callRunners` (`:328`/def `:767`) orders `Runner` beans and via
`callRunner` (`:786`) invokes `ApplicationRunner.run` (`:788`, `ApplicationRunner.java:40`) and
`CommandLineRunner.run` (`:791`, `CommandLineRunner.java:43`). Finally, if running,
`listeners.ready` (`:335`; composite `:81`; `:109` → `ApplicationReadyEvent` `:110`) marks the
application ready; any failure routes through `handleRunFailure` (`:331`, def `:803`).
