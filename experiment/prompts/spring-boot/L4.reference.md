# L4 reference answer key — auto-configuration import & conditional-filter subsystem

Pinned source: `experiment/repos/spring-boot` (SHA e16c5d01…). All cites re-verified.

A bounded cluster spanning `AutoConfigurationImportSelector.java`,
`AutoConfigurationImportFilter.java`, `AutoConfigurationMetadataLoader.java`, the condition package
(`FilteringSpringBootCondition.java`, `OnClassCondition.java`, `ConditionEvaluationReport.java`),
and the `META-INF/spring/...AutoConfiguration.imports` resource. Several cooperating paths; order
within each matters less than covering all five pieces.

## Required spine (gradeable core — must hit all five pieces)

1. **Candidate loading (from the imports file)** — `AutoConfigurationImportSelector.selectImports`
   (`AutoConfigurationImportSelector.java:119`) → `getAutoConfigurationEntry` (`:142`) →
   `getCandidateConfigurations` (`:200`) uses `ImportCandidates.load(this.autoConfigurationAnnotation,
   getBeanClassLoader())` (`:201-202`). `ImportCandidates.load` (`ImportCandidates.java:81`) reads
   `META-INF/spring/<annotation-name>.imports` (location `ImportCandidates.java:47`/`:84`) — concretely
   `core/spring-boot-autoconfigure/src/main/resources/META-INF/spring/org.springframework.boot
   .autoconfigure.AutoConfiguration.imports`. Returns the raw candidate class-name list.
2. **De-dup + exclusions** — `removeDuplicates(configurations)` (`AutoConfigurationImportSelector
   .java:148`, def `:305`); `getExclusions(metadata, attributes)` (`:149`, def `:249`) gathers
   `exclude`/`excludeName` attributes and the `PROPERTY_NAME_AUTOCONFIGURE_EXCLUDE` property
   (`:270-271`); `checkExcludedClasses(configurations, exclusions)` (`:150`, def `:212`) rejects
   excludes that aren't candidates; `configurations.removeAll(exclusions)` (`:151`).
3. **Fast class-level filter (`AutoConfigurationImportFilter`) + metadata** —
   `configurations = getConfigurationClassFilter().filter(configurations)` (`:152`). The
   `ConfigurationClassFilter` (inner class `:392`) is built from
   `getAutoConfigurationImportFilters()` (`:278`, `SpringFactoriesLoader.loadFactories(
   AutoConfigurationImportFilter.class, ...)`) and `AutoConfigurationMetadataLoader.loadMetadata(
   classLoader)` (in the filter ctor `:394-398`; `AutoConfigurationMetadataLoader.loadMetadata` at
   `AutoConfigurationMetadataLoader.java:43`, reading `META-INF/spring-autoconfigure-metadata.properties`
   `:38`). `filter()` (`AutoConfigurationImportSelector.java:399`) calls each
   `filter.match(candidates, this.autoConfigurationMetadata)` (`:403`) and nulls out non-matches
   (`:404-407`) — pruning **before** the candidate bytecode is read by Spring's condition machinery.
   The filter SPI is `AutoConfigurationImportFilter` (`AutoConfigurationImportFilter.java:59`,
   `match(...)`).
4. **Filter→condition bridge + recording** — the registered filter is typically
   `OnClassCondition` (`OnClassCondition.java:46`, `extends FilteringSpringBootCondition`). Its
   `FilteringSpringBootCondition.match` (`FilteringSpringBootCondition.java:52`) computes
   `ConditionOutcome[] getOutcomes(...)` (abstract, overridden in `OnClassCondition.getOutcomes`
   `:49`), and for each non-match (`:60`) logs the outcome (`logOutcome` `:63`) and calls
   `report.recordConditionEvaluation(autoConfigurationClass, this, outcome)` (`:65`) into a
   `ConditionEvaluationReport` obtained via `ConditionEvaluationReport.find(this.beanFactory)`
   (`:54`); `ConditionEvaluationReport.recordConditionEvaluation` is at
   `ConditionEvaluationReport.java:82`. This records *why* a candidate was dropped.
5. **Event + deferred group / surfacing** — `fireAutoConfigurationImportEvents(configurations,
   exclusions)` (`AutoConfigurationImportSelector.java:153`, def `:314`) fires an
   `AutoConfigurationImportEvent` (`:317`). Because the selector is a `DeferredImportSelector`, results
   are grouped: `getImportGroup()` returns `AutoConfigurationGroup.class` (`:158`), whose `process`
   (`:467`) collects `AutoConfigurationEntry` instances and the surviving class→importer map, and
   `selectImports` emits them. The entry itself is `AutoConfigurationEntry` (`:541`, holding
   `configurations` + `exclusions`).

## Acceptable extras (credit, never required)

- `AutoConfigurationReplacements` (`getAutoConfigurationReplacements` `:288`) reconciling removed/
  renamed auto-configurations across versions — bonus.
- `isEnabled(metadata)` gate honouring `EnableAutoConfiguration.ENABLED_OVERRIDE_PROPERTY` (`:160`).
- `shouldExclude` / `getExclusionFilter()` (`:127-132`) used by Spring's import machinery for the
  `exclude` attribute.
- The Aware wiring of filters (`invokeAwareMethods` `:330`) — `Environment`/`BeanFactory`/`ClassLoader`
  /`ResourceLoader` are set on each filter before `match`.
- Noting that `OnClassCondition` evaluates `@ConditionalOnClass`/`@ConditionalOnMissingClass` and is
  the concrete bytecode-free fast path (a subclass also handles `@ConditionalOnBean` lazily, not here).

## Common misses / wrong turns

- Describing only `selectImports` and missing the candidate *source* — the
  `META-INF/spring/...AutoConfiguration.imports` file is the load-bearing entry; without it the rest is
  undefined. Spine 1.
- Confusing the two-stage filtering: the fast `AutoConfigurationImportFilter` (pre-bytecode,
  `OnClassCondition`) is NOT the same as Spring's later `@Conditional` `ConditionEvaluator` pass. The
  prompt asks specifically for the fast class-level filter; conflating the two misses the point.
- Missing the recording step (spine 4): `FilteringSpringBootCondition.match` writes the outcome into
  `ConditionEvaluationReport` (`:65`) — that is the "how that filter records its decisions" the prompt
  names.
- Not connecting `AutoConfigurationMetadata` (`spring-autoconfigure-metadata.properties`) to the
  filter — the filter matches against metadata so it can decide without loading classes (`:403`).
- Stopping at `filter()` and not surfacing the deferred-group/event tail (spine 5).

## Completeness scale

- **Full:** all five pieces — candidate loading from the imports file, de-dup+exclusions, the fast
  `AutoConfigurationImportFilter` (+ `AutoConfigurationMetadata`) pruning, the
  `FilteringSpringBootCondition`→`ConditionEvaluationReport` recording bridge, and the
  event/deferred-group surfacing — connected, with cites, and the "fast/pre-bytecode" rationale.
- **Partial:** 4 of 5 pieces (commonly missing the recording bridge OR the imports-file source), or
  conflating the fast filter with Spring's `ConditionEvaluator`.
- **Miss:** only `selectImports`/`getAutoConfigurationEntry` with no filter or imports-file, or only
  the `@ConditionalOnClass` evaluation without the import-selector framing.

## Model answer

When `@EnableAutoConfiguration` is processed, `AutoConfigurationImportSelector.selectImports`
(`AutoConfigurationImportSelector.java:119`) drives `getAutoConfigurationEntry` (`:142`). Candidates
are loaded by `getCandidateConfigurations` (`:200`) via `ImportCandidates.load` (`:201`) reading
`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (the file under
`core/spring-boot-autoconfigure/src/main/resources/META-INF/spring/`; location pattern at
`ImportCandidates.java:47`). The list is de-duplicated (`removeDuplicates` `:148`) and exclusions
applied — `getExclusions` (`:149`) gathers the `exclude`/`excludeName` attributes and the
`PROPERTY_NAME_AUTOCONFIGURE_EXCLUDE` property (`:270-271`), `checkExcludedClasses` (`:150`) rejects
invalid excludes, and the rest are removed (`:151`). The decisive step is
`getConfigurationClassFilter().filter(configurations)` (`:152`): `ConfigurationClassFilter` (`:392`)
combines filters loaded via `getAutoConfigurationImportFilters()` (`:278`,
`SpringFactoriesLoader.loadFactories(AutoConfigurationImportFilter.class)`) with
`AutoConfigurationMetadataLoader.loadMetadata` (`AutoConfigurationMetadataLoader.java:43`, reading
`META-INF/spring-autoconfigure-metadata.properties` `:38`), and its `filter()` (`:399`) calls each
`filter.match(candidates, autoConfigurationMetadata)` (`:403`) to null out non-matches **before** any
candidate bytecode is read. The default filter `OnClassCondition` (`OnClassCondition.java:46`)
extends `FilteringSpringBootCondition`, whose `match` (`FilteringSpringBootCondition.java:52`)
computes `getOutcomes` (`OnClassCondition.getOutcomes` `:49`) and for each dropped candidate records
the decision via `report.recordConditionEvaluation(autoConfigurationClass, this, outcome)` (`:65`)
into the `ConditionEvaluationReport` found from the bean factory (`:54`; report method
`ConditionEvaluationReport.java:82`). Finally `fireAutoConfigurationImportEvents` (`:153`, def `:314`)
publishes `AutoConfigurationImportEvent`, and because the selector implements `DeferredImportSelector`,
the `AutoConfigurationGroup` (`:431`, `process` `:467`) collects the surviving entries into an
`AutoConfigurationEntry` (`:541`) and emits them for import.
