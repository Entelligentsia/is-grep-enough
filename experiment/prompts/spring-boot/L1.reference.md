# L1 reference answer key — the bindable target descriptor (`Bindable<T>`)

Pinned source: `experiment/repos/spring-boot` (SHA e16c5d01…). All cites re-verified.

The entity is `Bindable<T>` in
`core/spring-boot/src/main/java/org/springframework/boot/context/properties/bind/Bindable.java`.
It is the immutable descriptor the `Binder` is given for each thing it binds.

## Required spine (gradeable core — must hit all)

1. **The descriptor is `Bindable<T>`** — `Bindable.java:46` (`public final class Bindable<T>`). A
   single immutable value-object describing one bind target; built via the static `of(...)` factory
   methods and the `withX(...)` copy methods.
2. **`type` = the declared `ResolvableType` to bind** — `Bindable.java:52` (`private final ResolvableType type;`).
   This is the logical type asked for (the bean type, element type, etc.).
3. **`boxedType` = the boxed type used for conversion, distinct from `type`** — `Bindable.java:54`
   (`private final ResolvableType boxedType;`). Primitives are boxed to their wrapper so the
   converter can handle them; the distinction between `type` and `boxedType` is the load-bearing
   "what vs how" analogue (declared type vs the type the converter actually sees). Boxed type is
   produced in the constructor (`box(type)`, see the `of(ResolvableType)` factory at `Bindable.java:316-317`).
4. **`value` = an existing or supplied value instance** — `Bindable.java:56`
   (`private final @Nullable Supplier<T> value;`). When non-null it is the live instance to mutate
   (JavaBean) or a supplier that provides one; absence means the binder must create.
5. **`annotations` carried with the target** — `Bindable.java:58` (`private final Annotation[] annotations;`).
   These travel with the bindable so conversion/validation handlers (e.g. `@Validated`,
   `@DefaultValue`) can see them.
6. **`bindRestrictions`** — `Bindable.java:60` (`private final EnumSet<BindRestriction> bindRestrictions;`),
   e.g. `BindRestriction.NO_DIRECT_PROPERTY`, restricting how the binder may read this target.
7. **`bindMethod` = JAVA_BEAN vs VALUE_OBJECT (the central distinction)** — `Bindable.java:62`
   (`private final @Nullable BindMethod bindMethod;`). `BindMethod.JAVA_BEAN` = getter/setter binding;
   `BindMethod.VALUE_OBJECT` = constructor binding (see `BindMethod.java:30`/`:34`).
8. **The invariant tying the pieces together** — value-object binding cannot take an existing/supplied
   value. `withExistingValue` asserts `this.bindMethod != BindMethod.VALUE_OBJECT` (`Bindable.java:197`)
   and then sets `BindMethod.JAVA_BEAN` (`Bindable.java:201`); symmetrically `withBindMethod` asserts
   `bindMethod != BindMethod.VALUE_OBJECT || this.value == null` (`Bindable.java:241`). So supplying an
   existing value forces JavaBean binding.

## Common misses / wrong turns

- Treating `Bindable` as the bound *result* — it is the *target* descriptor; `BindResult` is the result.
  Confusing input and output of the binder.
- Listing the fields but not stating the `type` vs `boxedType` distinction, or not stating the
  `bindMethod` (JAVA_BEAN vs VALUE_OBJECT) distinction — these are the central gradeable points.
- Missing the invariant (spine 8) — i.e. not explaining *why* an existing value rules out value-object
  binding; the prompt asks how the pieces constrain one another.
- Pointing at `Binder` or `BindHandler` instead of `Bindable` — those consume the descriptor, they
  are not it.

## Completeness scale

- **Full:** identifies `Bindable<T>` and explains all spine elements, crucially the `type` vs
  `boxedType` distinction and the `bindMethod` (JAVA_BEAN vs VALUE_OBJECT) distinction, AND the
  value-object/existing-value invariant, with cites to `Bindable.java`.
- **Partial:** names `Bindable` and ≥4 spine elements but blurs one of the two distinctions, or omits
  the invariant tying `value` and `bindMethod` together.
- **Miss:** points at `Binder`/`BindResult`/the bound bean, or only locates the class without
  explaining the fields' roles.

## Model answer

A bind target is described by `Bindable<T>` (`Bindable.java:46`), an immutable value-object the
`Binder` receives. It carries `type` (`:52`) — the declared `ResolvableType` being bound — alongside a
separate `boxedType` (`:54`) which is `type` with primitives boxed to their wrappers, the form the
conversion layer actually works against. `value` (`:56`) is an optional `Supplier<T>` giving an
existing instance to mutate or a supplier for one (null ⇒ the binder must create). `annotations`
(`:58`) ride along so conversion/validation handlers can see them, and `bindRestrictions`
(`:60`, an `EnumSet<BindRestriction>`) can disable behaviours such as direct-property lookup. The
load-bearing field is `bindMethod` (`:62`): `BindMethod.JAVA_BEAN` (setter binding) vs
`BindMethod.VALUE_OBJECT` (constructor binding). These constrain one another: `withExistingValue`
(`:194`) refuses a value-object target (`:197`, "an existing value cannot be provided when binding as
a value object") and instead pins `BindMethod.JAVA_BEAN` (`:201`); the mirror guard is in
`withBindMethod` (`:241`, "value object binding cannot be used with an existing or supplied value"). So
the descriptor tells the binder both *what* to bind (type/boxedType + the live value) and *how*
(bindMethod), and the two are not independent.
