# L5 reference answer key — relaxed property binding end-to-end (cross-cutting)

Pinned source: `experiment/repos/spring-boot` (SHA e16c5d01…). All cites re-verified.

Threads four subsystems: the environment/property-source layer, the source-adaptation + name-mapping
layer (`source/`), the `Binder` + `BindHandler` layer (`bind/`), and conversion. Each stage hands off to
the next; the env-var name-form mapping is the crux cross-cutting hop.

## Required spine (gradeable core — must hit all)

1. **Attach the property sources to the environment** — during `SpringApplication.prepareEnvironment`
   (`SpringApplication.java:350`), `ConfigurationPropertySources.attach(environment)`
   (`SpringApplication.java:355`; `ConfigurationPropertySources.attach` at `source/
   ConfigurationPropertySources.java:89`) inserts a `SpringConfigurationPropertySources`
   (`:95`) wrapper as a property source so the environment can later be iterated as
   `ConfigurationPropertySource`s (`from(...)` at `ConfigurationPropertySources.java:156` returns a
   `SpringConfigurationPropertySources` (`:157`); class `source/SpringConfigurationPropertySources.java:43`).
2. **Adapt one `PropertySource` → a `ConfigurationPropertySource` (with a `PropertyMapper`)** —
   `SpringConfigurationPropertySource` (`source/SpringConfigurationPropertySource.java:56`) adapts a
   raw `PropertySource<?>` using a `PropertyMapper[]` (`:67`). For a `SystemEnvironmentPropertySource`
   the mappers are `SYSTEM_ENVIRONMENT_MAPPERS = { SystemEnvironmentPropertyMapper.INSTANCE,
   DefaultPropertyMapper.INSTANCE }` (`:60-61`); otherwise `DEFAULT_MAPPERS = {
   DefaultPropertyMapper.INSTANCE }` (`:58`); chosen in the factory at `:174`.
3. **Resolve a configuration name → a property-name candidate → the value (the env-var name forms)** —
   `SpringConfigurationPropertySource.getConfigurationProperty(name)` (`:85`) loops its mappers,
   `mapper.map(name)` (`:91`) yields candidate property-name strings, then
   `getPropertySourceProperty(candidate)` (`:92`) reads the raw source; on a hit it builds
   `ConfigurationProperty.of(this, name, value, origin)` (`:95`, origin via `PropertySourceOrigin.get`,
   `ConfigurationProperty` factory at `source/ConfigurationProperty.java:79`). For env vars,
   `SystemEnvironmentPropertyMapper.map` (`source/SystemEnvironmentPropertyMapper.java:47`) returns up
   to four forms (`SYSTEM_ENVIRONMENT` and `LEGACY_SYSTEM_ENVIRONMENT`, each with/without element-index,
   `:49-52`); the reverse direction uses `ConfigurationPropertyName.adapt(propertySourceName, '_',
   processElementValue)` (`:74`, inside `convertName` `:72`), where `processElementValue` (`:81-83`)
   lower-cases elements and turning numbers into `[n]` indices. So `SERVER_PORT` is reached from
   `server.port`.
4. **`Binder.bind` entry + handler `onStart`** — `Binder.bind(name, target, handler)` (`bind/
   Binder.java:287`) calls the private `bind(...)` (`:365`): `handler.onStart(name, target, context)`
   (`:369`) may replace/short-circuit the target; `BindHandler` is `bind/BindHandler.java:31`
   (default `:36`, `onStart` default `:48`). If `onStart` returns null the bind yields no result.
5. **Resolve + bind the property (`bindObject`)** — `bindObject(name, target, handler, context, ...)`
   (`Binder.java:423`) calls `findProperty(name, target, context)` (`:424`, def `:476`), which iterates
   `context.getSources()` and calls `source.getConfigurationProperty(name)` (`:482`) — the hop back to
   spine 3. If found it calls `bindProperty` (`:432`/`bindProperty` `:490`): resolves placeholders
   (`:492`) and converts via `context.getConverter().convert(result, target)` (`:494`). Aggregates go
   through `getAggregateBinder` (`:451`) + `bindAggregate` (`:465`); nested beans via
   `bindDataObject` (`:498`) using the `DataObjectBinder`s keyed by `target.getBindMethod()`
   (`handleBindResult` `:384-406`).
6. **Result handling + conversion (`BindConverter`) + `BindResult`** — `handleBindResult` (`:384`) runs
   `handler.onSuccess(name, target, context, result)` (`:387`) then
   `context.getConverter().convert(result, target)` (`:388`); on success it finishes with
   `handler.onFinish(...)` (`:405`). The converter is `BindConverter` (`bind/BindConverter.java:58`,
   `BindConverter.get` `:126`, shared via `:136`), assembled in the `Binder` constructor at
   `Binder.java:212`. The public entry wraps the private result in `BindResult.of(bound)`
   (`Binder.java:289`).

## Common misses / wrong turns

- Claiming the env var is read directly by `Binder` — it is not; the value is surfaced through the
  `ConfigurationPropertySource` + `PropertyMapper` adaptation, and `findProperty` is the only place the
  binder touches a source (`Binder.java:482`). Skipping spine 2-3 misses the cross-cutting crux.
- Stopping at "the binder binds the property" without the conversion stage — `BindConverter`/the
  `context.getConverter().convert(...)` calls (`:388`/`:406`/`:494`) are the terminal transform; an
  account that omits conversion is incomplete.
- Missing the handler callbacks (`onStart`/`onSuccess`/`onFinish`, `Binder.java:369`/`:387`/`:405`) —
  the prompt explicitly asks about bind-handler callbacks.
- Confusing the two mapping directions: `PropertyMapper.map(ConfigurationPropertyName)`
  (name→candidate strings, used during resolution) vs `map(String)` (env string→`ConfigurationPropertyName`,
  used by `SystemEnvironmentPropertyMapper.map(String)` `:63`). Resolution uses the first.
- Forgetting the `attach` step (spine 1) — without `ConfigurationPropertySources.attach` the
  environment is not iterable as configuration-property sources and binding wouldn't see env vars at
  all.

## Completeness scale

- **Full:** all 6 spine elements connected — `ConfigurationPropertySources.attach` →
  `SpringConfigurationPropertySource` (+ `SystemEnvironmentPropertyMapper`) name-form resolution →
  `Binder.bind` (`onStart`) → `findProperty`/`bindProperty`/conversion → `handleBindResult`
  (`onSuccess`/`onFinish`, `BindConverter`) → `BindResult` — with cites, and explicit acknowledgement
  of the env-var name-mapping cross-cutting hop.
- **Partial:** ≥4 of 6, but missing either the source-adaptation/name-mapping hop OR the conversion +
  handler-callback tail (i.e. an account that doesn't bridge env → name → binder → converted result).
- **Miss:** a direct "env var → binder sets the field" story with no adaptation, name mapping, or
  conversion stages.

## Model answer

When the environment is prepared (`SpringApplication.prepareEnvironment`, `SpringApplication.java:350`),
`ConfigurationPropertySources.attach(environment)` (`:355`; `ConfigurationPropertySources.java:89`)
wraps the environment's raw `PropertySource<?>`s in a `SpringConfigurationPropertySources`
(`source/SpringConfigurationPropertySources.java:43`, `ConfigurationPropertySources.java:156-157`).
Later, `Binder.bind(name, target, handler)` (`bind/Binder.java:287`) → private `bind` (`:365`) first
asks `handler.onStart(name, target, context)` (`:367`; `BindHandler` at `bind/BindHandler.java:31`),
then `bindObject` (`:423`) resolves via `findProperty` (`:476`), which iterates the adapted
`ConfigurationPropertySource`s and calls `source.getConfigurationProperty(name)` (`:482`). That call
(`SpringConfigurationPropertySource.getConfigurationProperty`, `:85`) runs each `PropertyMapper`
(`:89`); for an env var the mappers are `SystemEnvironmentPropertyMapper` + `DefaultPropertyMapper`
(`:60-61`), and `SystemEnvironmentPropertyMapper.map` (`SystemEnvironmentPropertyMapper.java:47`)
yields the `SERVER_PORT`-style candidate forms (`:49-52`), reading the value with
`getPropertySourceProperty(candidate)` (`SpringConfigurationPropertySource.java:92`) and returning a
`ConfigurationProperty.of(this, name, value, origin)` (`:95`; `ConfigurationProperty.java:79`) — so
`server.port` resolves `SERVER_PORT`. `bindProperty` (`Binder.java:490`) then resolves placeholders
(`:492`) and converts via `context.getConverter().convert(result, target)` (`:494`, the
`BindConverter` at `bind/BindConverter.java:58`/`:126`). Finally `handleBindResult` (`:384`) runs
`handler.onSuccess` (`:387`) and `handler.onFinish` (`:405`) and re-converts, and `bind` wraps the result
in `BindResult.of(bound)` (`:289`). (`@Validated` targets additionally pass through
`ValidationBindHandler`, `bind/validation/ValidationBindHandler.java:50`, as a decorating handler.) So
one env var travels environment → source-adaptation/name-mapping → binder(lookup+handler) →
conversion → bound result.
