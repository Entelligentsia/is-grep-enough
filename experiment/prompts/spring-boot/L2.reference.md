## Common misses / wrong turns

- Stopping at `getExitCode()` and not naming the two entry points (`handleExitCode` for failure,
  `exit` for programmatic shutdown) — the prompt asks where a code comes from *and* how it's
  published/returned.
- Missing the cause-chain recursion of source A (`SpringApplication.java:919`) — a wrapped exception
  can still carry the code; omitting this under-describes "a thrown exception that carries a code."
- Confusing the two composition orders: mappers run *before* the generator-exception check
  (`getExitCodeFromException` `:895`/`:897`), not after.
- Forgetting the `@Order`/`Ordered` ordering that makes "first non-zero" well-defined
  (`add` sort `ExitCodeGenerators.java:79`).
- Treating `MappedExitCodeGenerator` as a standalone source rather than the adapter that lets an
  `ExitCodeExceptionMapper` bean participate via the same `getExitCode()` contract.

## Model answer

The final exit code is decided by `ExitCodeGenerators` (`ExitCodeGenerators.java:42`), an ordered list
of `ExitCodeGenerator`s whose `getExitCode()` (`:92`) returns the **first non-zero** value, iterating
in `@Order`/`Ordered` order (sorted on `add` at `:79`); a generator that throws collapses to code 1
(`:103`). Two entry points build it. On startup failure, `handleRunFailure` (`SpringApplication.java:803`)
calls `handleExitCode` (`:881`) → `getExitCodeFromException` (`:894`), which first tries mapped
exceptions via `getExitCodeFromMappedException` (`:902`): it collects `ExitCodeExceptionMapper` beans
(`:907`), `generators.addAll(exception, beans)` (`:908`) — the overload at `ExitCodeGenerators.java:50`
wraps each mapper in a `MappedExitCodeGenerator` (`:61`, adapter `:113` whose `getExitCode()`
(`:125`) calls `mapper.getExitCode(exception)`) — and returns `generators.getExitCode()` (`:909`); if
that is zero, it falls back to `getExitCodeFromExitCodeGeneratorException` (`:912`), which recurses the
cause chain for an exception that `instanceof ExitCodeGenerator` (`:916-919`). The chosen code is then
published as `ExitCodeEvent` (`:885`) and registered with the `SpringBootExceptionHandler` (`:889`).
Programmatic shutdown uses `SpringApplication.exit(context, ExitCodeGenerator...)` (`:1396`): it
aggregates the caller's generators (`:1403`) and bean `ExitCodeGenerator`s (`:1404`), calls
`generators.getExitCode()` (`:1405`), publishes the event (`:1407`), and closes the context
(`:1411`). So a code can come from a thrown exception's own `getExitCode()` (spine 3), a mapped
exception via a bean (spine 4), or any bean/caller-supplied `ExitCodeGenerator` (spine 7), combined
under one first-non-zero rule.
