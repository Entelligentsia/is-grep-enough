# L5 reference answer key — saving a model instance (cross-cutting write path)

Pinned source: `experiment/repos/django` (SHA 7903ee10…). All cites re-verified.
Paths are relative to the `django/` package dir. Threads three subsystems: the model layer
(`db/models/base.py`), the signal layer (`db/models/signals.py`), and the SQL/query +
backend layer (`db/models/query.py`, `db/models/sql/compiler.py`). The update-first-then-insert
decision and the manager→query→compiler→cursor hand-off are the crux.

## Required spine (gradeable core — must hit all)

1. **Public entry `Model.save`** — `db/models/base.py:841`; after bookkeeping it delegates to
   `self.save_base(...)` (`db/models/base.py:904`).
2. **Once-per-save work + lifecycle signals in `save_base`** — `db/models/base.py:950`. It fires
   `pre_save.send(sender=origin, instance=self, …)` (`db/models/base.py:976`) before writing and
   `post_save.send(..., created=(not updated), …)` (`db/models/base.py:1011`) after; the actual
   row write goes through `self._save_table(...)` (`db/models/base.py:996`). (`pre_save`/
   `post_save` are `ModelSignal`s, `db/models/signals.py:45-46`.)
3. **Update-or-insert decision in `_save_table`** — `db/models/base.py:1069`. When the pk is set
   and not forced to insert it attempts an UPDATE first via `self._do_update(...)`
   (`db/models/base.py:1140`); if nothing was updated (`if not updated:` `db/models/base.py:1157`)
   it performs an INSERT via `self._do_insert(...)` (`db/models/base.py:1199`). The
   `updated` boolean it returns is what `save_base` reads to set `created` on `post_save`.
4. **The insert hand-off into the query layer** — `_do_insert` (`db/models/base.py:1246`) calls
   `manager._insert([self], fields=…, returning_fields=…, using=…, raw=…)`
   (`db/models/base.py:1251`), i.e. `QuerySet._insert` (`db/models/query.py:2112`), which builds a
   `sql.InsertQuery` and runs
   `query.get_compiler(using=using).execute_sql(returning_fields)` (`db/models/query.py:2130-2137`).
   (Update mirror: `_do_update` (`db/models/base.py:1206`) calls `filtered._update(...)`
   (`db/models/base.py:1244`) → `QuerySet._update` `db/models/query.py:1428` → `SQLUpdateCompiler`.)
5. **Compile to SQL and run it on the connection** — `SQLInsertCompiler.execute_sql`
   (`db/models/sql/compiler.py:1925`) opens `with self.connection.cursor() as cursor:`
   (`compiler.py:1934`) and runs `cursor.execute(sql, params)` for each
   `self.as_sql()` statement (`compiler.py:1935-1936`), then reads back any returning fields
   (auto-pk etc.).
6. **The result flows back up** — returned values are assigned via `_assign_returned_values`
   (`db/models/base.py:1150` / `1203`, def `1259`); `_save_table` returns `updated`
   (`db/models/base.py:1204`) to `save_base`, which uses it for the `post_save` `created` flag
   (`db/models/base.py:1014`).

## Acceptable extras (credit, never required)

- `field.pre_save(self, add=…)` per-field value prep before insert/update
  (`db/models/base.py:1121`, `1188`) — distinct from the `pre_save` signal.
- Transaction wrapping: `transaction.atomic` / `mark_for_rollback_on_error` around the table
  writes (`db/models/base.py:984-988`).
- `_save_parents` for multi-table inheritance (`db/models/base.py:993`, def `1022`).
- `select_on_save` re-check in `_do_update` (`db/models/base.py:1231-1243`).
- The `ModelSignal.send` async-aware dispatch (the *mechanism* of sending) — bonus; this key
  centers the data path, not the signal machinery.
- The `db_for_write` router pick (`db/models/base.py:967`) choosing the connection alias.

## Common misses / wrong turns

- Saying `save` writes SQL directly — it does not; it goes `save` → `save_base` → `_save_table`
  → `_do_insert`/`_do_update` → manager/queryset → compiler → cursor. Skipping `save_base` or
  `_save_table` is a major miss.
- Missing the update-first-then-insert decision (`_save_table` 1115/1140 then 1157/1199) — the
  central "decides between updating and inserting" the prompt asks for.
- Omitting the `pre_save`/`post_save` signals (`db/models/base.py:976`/`1011`) — the prompt names
  "lifecycle notifications" explicitly.
- Stopping at `_do_insert`/`_insert` without reaching the compiler's `cursor.execute`
  (`compiler.py:1936`) — the account never actually hits the database.
- Confusing the per-field `pre_save()` method with the `pre_save` signal.

## Completeness scale

- **Full:** all 6 elements connected — `save`→`save_base` (with pre_save/post_save),
  `_save_table`'s update-first/insert-fallback decision, `_do_insert`/`_do_update` →
  `QuerySet._insert`/`_update` → compiler `execute_sql`, ending at `cursor.execute`, and the
  `updated`/`created` result feeding back — with cites and acknowledgement that the model layer
  never touches SQL directly.
- **Partial:** ≥4 of 6, but missing either the signals OR the update/insert decision OR the
  compiler→cursor execution bridge.
- **Miss:** a "save runs an SQL INSERT" story with no `save_base`/`_save_table` decision and no
  manager→compiler hand-off.

## Model answer

`Model.save` (`db/models/base.py:841`) does light setup and calls `save_base`
(`db/models/base.py:904`). `save_base` (950) owns the once-per-save work: unless the model is
auto-created it sends `pre_save` (976), wraps the writes in a transaction/rollback guard
(984-988), calls `self._save_table(...)` (996), then sends `post_save` with
`created=(not updated)` (1011-1018) — `pre_save`/`post_save` being `ModelSignal`s
(`db/models/signals.py:45-46`). `_save_table` (1069) makes the update-vs-insert decision: if the
pk is set and insert isn't forced it tries an UPDATE via `_do_update` (1140); if that updated no
rows (`if not updated:` 1157) it does an INSERT via `_do_insert` (1199), returning the `updated`
flag (1204). `_do_insert` (1246) calls `manager._insert([self], …)` (1251) — i.e.
`QuerySet._insert` (`db/models/query.py:2112`) — which builds a `sql.InsertQuery` and runs
`query.get_compiler(using=using).execute_sql(returning_fields)` (2130-2137). That lands in
`SQLInsertCompiler.execute_sql` (`db/models/sql/compiler.py:1925`), which opens a cursor on the
chosen connection (1934) and runs `cursor.execute(sql, params)` for each statement from
`as_sql()` (1935-1936), reading back auto-generated values. (An UPDATE follows the parallel path
`_do_update` 1206 → `QuerySet._update` `query.py:1428` → `SQLUpdateCompiler`.) Returned values are
applied via `_assign_returned_values` (1259) and the `updated` boolean flows back up to set the
`created` flag on `post_save`. The key point: the model layer never emits SQL itself — it
delegates through the queryset/manager to a compiler that talks to the DB connection.
