## Common misses / wrong turns

- Describing the resolver/pattern machinery (`URLResolver`, `RegexPattern`, `resolve()`)
  instead of the result object ‚Äî those produce a `ResolverMatch`, they are not it.
- Treating `args`/`kwargs` as the same thing as `captured_kwargs`/`extra_kwargs` without
  noting the split, or omitting the `__getitem__` triple (the load-bearing consumer contract).
- Conflating `url_name`, `view_name`, and `route` (each is distinct: pattern name vs
  namespaced identifier vs route string).

## Model answer

URL resolution returns a `ResolverMatch` (`urls/resolvers.py:34`). Its `__init__`
(`35-74`) records the selected view callable in `func` (48) and the arguments pulled from the
URL in `args` (49) and `kwargs` (50) ‚Äî `kwargs` being composed from the pattern-`captured_kwargs`
(54) plus any `extra_kwargs` (55). It carries the matched pattern's `url_name` (51) and `route`
string (52). Naming/namespacing is held as `app_names`/`app_name` (59-60) and
`namespaces`/`namespace` (61-62) ‚Äî lists joined with `:` ‚Äî and a fully-qualified `view_name`
(74) built from the namespaces plus the url_name (or the derived `_func_path`, 64-71, which
handles class- vs function-based views). For convenience, `__getitem__` (76-77) exposes the
object as the `(func, args, kwargs)` triple, which is how `resolve_request` unpacks it. It also
keeps `tried` (53) for 404 reporting and refuses pickling via `__reduce_ex__` (104-105).
