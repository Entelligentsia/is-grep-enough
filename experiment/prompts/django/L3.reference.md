# L3 reference answer key ‚Äî inbound request path (WSGI ‚Üí view invocation)

Pinned source: `experiment/repos/django` (SHA 7903ee10‚Ä¶). All cites re-verified.
Paths are relative to the `django/` package dir. A single ordered chain across
`core/handlers/wsgi.py`, `core/handlers/base.py`, and `urls/resolvers.py`. Order is gradeable.

## Required spine (gradeable core ‚Äî must hit all, in order)

1. **WSGI entry builds the request** ‚Äî `WSGIHandler.__call__` `core/handlers/wsgi.py:120`
   constructs the request via `self.request_class(environ)` (`core/handlers/wsgi.py:123`) and
   calls `self.get_response(request)` (`core/handlers/wsgi.py:124`). (The middleware stack was
   assembled once at init by `self.load_middleware()` `core/handlers/wsgi.py:118`.)
2. **Enter the middleware stack** ‚Äî `BaseHandler.get_response` `core/handlers/base.py:138` calls
   `self._middleware_chain(request)` (`core/handlers/base.py:142`).
3. **The chain bottoms out at the inner handler** ‚Äî `_middleware_chain` is built in
   `load_middleware` (`core/handlers/base.py:27`): the innermost handler is
   `convert_exception_to_response(self._get_response)` (`core/handlers/base.py:39`; wrapper def
   `core/handlers/exception.py:25`), with each configured middleware wrapped around it.
4. **Inner handler resolves and calls the view** ‚Äî `BaseHandler._get_response`
   `core/handlers/base.py:176` first does
   `callback, callback_args, callback_kwargs = self.resolve_request(request)`
   (`core/handlers/base.py:183`).
5. **Resolve the path to a view + args** ‚Äî `resolve_request` `core/handlers/base.py:302` picks
   the resolver (`get_resolver`) and calls `resolver.resolve(request.path_info)`
   (`core/handlers/base.py:315`, the `URLResolver.resolve` at `urls/resolvers.py:670`),
   returning a `ResolverMatch` it unpacks into `(callback, args, kwargs)`.
6. **Invoke the view** ‚Äî back in `_get_response`, after view middleware, it wraps the callback
   with `make_view_atomic` (`core/handlers/base.py:194`) and calls
   `response = wrapped_callback(request, *callback_args, **callback_kwargs)`
   (`core/handlers/base.py:199`).

## Common misses / wrong turns

- Jumping straight to `_get_response`/the view call without the WSGI entry and the middleware
  chain (skips spine 1-3) ‚Äî the prompt asks "from the moment the WSGI server hands it to Django."
- Treating `get_response` as calling `_get_response` directly ‚Äî it goes through
  `_middleware_chain`, whose innermost layer is the exception-wrapped `_get_response`; missing
  that indirection is a common error.
- Stopping at `resolve_request`/`ResolverMatch` and not naming the actual view invocation
  `wrapped_callback(request, ‚Ä¶)` (`core/handlers/base.py:199`).
- Describing it as a set of middleware rather than an ordered request‚Üíview chain.

## Completeness scale

- **Full:** all 6 stops in order ‚Äî WSGI `__call__`/request build ‚Üí `get_response` ‚Üí
  `_middleware_chain`/inner `_get_response` ‚Üí `resolve_request` ‚Üí `resolver.resolve` ‚Üí view
  call ‚Äî with cites, ending at the actual `wrapped_callback(request, ‚Ä¶)` invocation.
- **Partial:** ‚â•4 of 6 in roughly correct order but missing the WSGI front end OR the final

- **Miss:** only the resolver half, or an account that skips middleware/resolution entirely.

## Model answer

A request arrives at `WSGIHandler.__call__` (`core/handlers/wsgi.py:120`), which builds the
`HttpRequest` (`self.request_class(environ)`, 123) and calls `get_response(request)` (124); the
middleware stack itself was assembled once at startup by `load_middleware`
(`core/handlers/wsgi.py:118`). `BaseHandler.get_response` (`core/handlers/base.py:138`) pushes
the request into `self._middleware_chain(request)` (142). That chain is built in
`load_middleware` (`core/handlers/base.py:27`) by wrapping each configured middleware around an
innermost handler, `convert_exception_to_response(self._get_response)` (39; wrapper at
`core/handlers/exception.py:25`). So control reaches `_get_response`
(`core/handlers/base.py:176`), which calls `resolve_request` (183). `resolve_request`
(`core/handlers/base.py:302`) runs `resolver.resolve(request.path_info)` (315 ‚Üí
`URLResolver.resolve` `urls/resolvers.py:670`), yielding a `ResolverMatch` unpacked into
`(callback, callback_args, callback_kwargs)`. After view middleware, `_get_response` wraps the
callback with `make_view_atomic` (194) and finally invokes the view:
`wrapped_callback(request, *callback_args, **callback_kwargs)` (199). (An equivalent async spine
exists via `get_response_async`/`_get_response_async`.)
