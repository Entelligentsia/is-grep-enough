# L2 reference answer key — the escape-or-pass-through decision and its callers

Pinned source: `experiment/repos/django` (SHA 7903ee10…). All cites re-verified.
Paths are relative to the `django/` package dir.

Focal symbol: `conditional_escape` (`utils/html.py:120`) plus the `__html__` safe-string
convention it depends on and its direct callers (1 hop).

## Required spine (gradeable core — must hit all)

1. **The decision routine is `conditional_escape(text)`** — `utils/html.py:120`. It decides
   whether a value needs HTML-escaping or can be emitted unchanged.
2. **The branch / convention it relies on** — `utils/html.py:129-132`: if the value advertises
   the `__html__` method (`hasattr(text, "__html__")`) it returns `text.__html__()` unchanged
   (already-safe); otherwise it falls through to `escape(text)`. (It also coerces lazy
   `Promise` objects to `str` first, `utils/html.py:127-128`.) This `__html__` protocol is the
   "tell the two cases apart" mechanism.
3. **What "already safe" means** — `SafeData.__html__` returns `self` (`utils/safestring.py:16-22`),
   and `SafeString` is the `str` subclass marked safe (`utils/safestring.py:25`); `mark_safe`
   (`utils/safestring.py:65`) is how a value gets that marking. So a `SafeString`/anything with
   `__html__` (incl. third-party markupsafe) passes through untouched.
4. **The escaping fallback** — `escape(text)` (`utils/html.py:52`) HTML-escapes via
   `html.escape` and returns a `SafeString` (`utils/html.py:61`); note it *always* escapes (the
   docstring `54-60` contrasts it with `conditional_escape`).
5. **The direct callers, and how each depends on the result (≥2 of 3 required):**
   - `format_html` `utils/html.py:135` — passes every positional/keyword arg through
     `conditional_escape` (`utils/html.py:143-144`) before interpolating, then `mark_safe`s the
     result, so safe inputs aren't double-escaped while unsafe ones are.
   - `format_html_join` `utils/html.py:148` — runs the separator through `conditional_escape`
     (`utils/html.py:163`) and delegates each row to `format_html`.
   - `render_value_in_context` `template/base.py:1140` — the template auto-escape boundary:
     when `context.autoescape` is on it returns `conditional_escape(value)`
     (`template/base.py:1148-1151`), else just `str(value)`.

## Common misses / wrong turns

- Describing `escape` alone and missing the conditional `__html__` branch — the whole point is
  the *decision*, not unconditional escaping.
- Not naming the `__html__`/`SafeData`/`SafeString` convention as the thing that distinguishes
  already-safe values (the central gradeable mechanism).
- Listing callers but not stating how each uses the result (auto-escape boundary vs
  no-double-escape interpolation) — the synthesis the prompt asks for.
- Confusing `mark_safe` (mark as safe) with `conditional_escape` (decide + escape if needed).

## Model answer

The escape-or-pass-through decision is `conditional_escape(text)` (`utils/html.py:120`). After
coercing a lazy `Promise` to `str` (127-128), it checks for the `__html__` convention: if the
value has it, the value is already safe and is returned via `text.__html__()` unchanged
(129-130); otherwise it escapes through `escape(text)` (`utils/html.py:52`, which runs
`html.escape` and returns a `SafeString`, 61). The `__html__` protocol is what distinguishes the
two cases — `SafeData.__html__` returns `self` (`utils/safestring.py:16-22`), `SafeString` is the
safe `str` subclass (25), and `mark_safe` (65) is how content earns that marking (so markupsafe
values interoperate too). Callers route output through it: `format_html` (`utils/html.py:135`)
conditionally-escapes each argument (143-144) before interpolation so pre-marked-safe fragments
aren't double-escaped; `format_html_join` (148) escapes the separator (163) and reuses
`format_html` per row; and the template engine's `render_value_in_context` (`template/base.py:1140`)
calls `conditional_escape` only when `context.autoescape` is on (1148-1151) — that is the
auto-escaping boundary for variable output.
