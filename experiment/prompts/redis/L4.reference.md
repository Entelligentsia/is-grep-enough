# L4 reference answer key — non-blocking RDB snapshot (background save) subsystem

Pinned source: `experiment/repos/redis` (SHA d2d3390d…). All cites re-verified.

A bounded cluster across `rdb.c`, `server.c`, `childinfo.c`. Several cooperating paths
around a fork boundary; order within each path matters less than covering all four pieces.

## Required spine (gradeable core — must hit all four pieces)

1. **Launch + fork** — `rdbSaveBackground` (`rdb.c:2070`) forks via
   `redisFork(CHILD_TYPE_RDB)` (`rdb.c:2079`; def `server.c:7428`). The fork makes the save
   non-blocking: child does the writing while the parent keeps serving. `redisFork` sets up
   the info channel with `openChildInfoPipe()` (`server.c:7448`, def `childinfo.c:25`).
2. **Child writes the snapshot** — in the child branch (`rdb.c:2080-2089`) it runs
   `rdbSave(req, filename, rsi, rdbflags)` (`rdb.c:2085`, def `rdb.c:2027`), then
   `exitFromChild` with the result code (`rdb.c:2089`). The serialization work happens in
   the child address space (copy-on-write), not the parent.
3. **Child → parent communication (progress over the info pipe)** — during serialization
   the child reports via `sendChildInfo(CHILD_INFO_TYPE_CURRENT_INFO, …)` (`rdb.c:1858`; def
   `server.c:7529` → `sendChildInfoGeneric` `childinfo.c:49`); the parent drains it with
   `receiveChildInfo` (`childinfo.c:150`), called from the cron (`server.c:1694`).
4. **Parent reaps + finalizes** — the server cron calls `checkChildrenDone`
   (`server.c:1695`, def `server.c:1416`), which `waitpid`s (`server.c:1420`) and, for an
   RDB child (`server.c:1442`), calls `backgroundSaveDoneHandler(exitcode, bysignal)`
   (`server.c:1443`, def `rdb.c:4605`) → `backgroundSaveDoneHandlerDisk` (`rdb.c:4544`),
   which updates save state (dirty reset, lastsave, status).

## Acceptable extras (credit, never required)

- COW-size reporting: `sendChildCowInfo(CHILD_INFO_TYPE_RDB_COW_SIZE, "RDB")` (`rdb.c:2087`).
- The diskless/socket variant of bgsave (`rdbSaveToSlavesSockets`, the second `redisFork`
  at `rdb.c:4714`) and `RDB_CHILD_TYPE_DISK` vs `_SOCKET` (`rdb.c:2100`).
- Server state set in the parent branch (`server.rdb_save_time_start`, `rdb_child_pid`
  bookkeeping, `rdb.c:2098-2100`).
- `latency`/proc-title/CPU-affinity setup in the child (`rdb.c:2083-2084`).
- The signal-driven reaping note (SIGCHLD) vs the periodic `checkChildrenDone` poll.

## Common misses / wrong turns

- Describing only `rdbSave` (the synchronous, blocking save) and missing the fork that
  makes it background — `rdbSaveBackground`/`redisFork` is the load-bearing launch.
- Missing the childinfo pipe entirely (treating completion = the only communication);
  progress reporting is a distinct mechanism the prompt asks for ("how progress … is
  communicated back").
- Not connecting the parent's reaping (`checkChildrenDone` → `backgroundSaveDoneHandler`)
  to completion — leaving the snapshot "finished in the child" with no finalization.
- Confusing the two `redisFork` sites (disk-bgsave `rdb.c:2079` vs diskless `rdb.c:4714`).

## Completeness scale

- **Full:** all four pieces — launch/fork, child rdbSave, the childinfo progress pipe, and
  parent reap+finalize (`checkChildrenDone`→`backgroundSaveDoneHandler`) — connected, with
  cites, and the COW/fork rationale for non-blocking.
- **Partial:** 3 of 4 pieces (commonly missing the childinfo pipe OR the finalize handler).
- **Miss:** only `rdbSave`, or no account of how the parent learns of progress/completion.

## Model answer

A background save is launched by `rdbSaveBackground` (`rdb.c:2070`), which forks via
`redisFork(CHILD_TYPE_RDB)` (`rdb.c:2079`, def `server.c:7428`); the fork is what keeps the
server responsive — the parent returns to the event loop while the child snapshots a
copy-on-write view. In the child branch (`rdb.c:2080-2089`) the work is done by `rdbSave`
(`rdb.c:2085`, def 2027), which serializes the dataset and, periodically, reports progress
back through the child-info pipe via `sendChildInfo(CHILD_INFO_TYPE_CURRENT_INFO, …)`
(`rdb.c:1858`; `server.c:7529` → `childinfo.c:49`); that pipe is opened by
`openChildInfoPipe` (`childinfo.c:25`, from `redisFork` at `server.c:7448`) and drained by
the parent's `receiveChildInfo` (`childinfo.c:150`, called from cron `server.c:1694`). When
the child exits, the parent's cron runs `checkChildrenDone` (`server.c:1695`/1416), which
`waitpid`s (1420) and for an RDB child invokes `backgroundSaveDoneHandler` (`server.c:1443`,
def `rdb.c:4605`) → `backgroundSaveDoneHandlerDisk` (`rdb.c:4544`), finalizing save status,
clearing dirtiness, and updating lastsave. (A diskless variant forks similarly at
`rdb.c:4714` and streams to replica sockets instead of a file.)
