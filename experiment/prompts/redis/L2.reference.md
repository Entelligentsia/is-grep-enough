# L2 reference answer key — on-access (lazy) expiry trigger and its callers

Pinned source: `experiment/repos/redis` (SHA d2d3390d…). All cites re-verified.

Focal symbol: `expireIfNeeded` (`db.c:2935`) plus its direct callers (1 hop).

## Required spine (gradeable core — must hit all)

1. **The routine is `expireIfNeeded(db, key, kv, flags)`** — def `db.c:2935`. This is the
   on-access check that decides whether a logically-expired key is removed *now*.
2. **Return contract drives caller behavior** — enum `keyStatus` at `db.c:43-48`
   (`KEY_VALID`, `KEY_EXPIRED` = expired-but-not-deleted, `KEY_DELETED` = deleted now,
   `KEY_TRIMMED`). Callers branch on this.
3. **Expiry test** — it calls `keyIsExpired` (`db.c:2877`), which compares the key's
   expire time (`getExpire` `db.c:2762`) against the command-time snapshot.
4. **Master-vs-replica guard (why "ordinary handling" deletes only on a master)** —
   `db.c:2974-2977`: on a replica (`server.masterhost != NULL`) without
   `EXPIRE_FORCE_DELETE_EXPIRED` it returns `KEY_EXPIRED` instead of deleting; replicas
   wait for the master's DEL.
5. **The deletion branch** — on a master it performs `deleteExpiredKeyAndPropagate`
   (`db.c:2996` / 3000; helper def `db.c:2826`) and returns `KEY_DELETED` (`db.c:3003`).
6. **The direct callers (≥3 of 4 required), and what each does with the result:**
   - `lookupKey` `db.c:302` — the central key-access path; treats non-`KEY_VALID` as a miss.
   - `dbRandomKey` `db.c:834` — skips/retries when the random key turned out expired.
   - `delGenericCommand` `db.c:1419` — counts a `KEY_DELETED` toward the DEL reply.
   - `scanCallback` `db.c:1702` — kv-based variant, drops non-valid keys mid-scan.

## Model answer

The on-access check is `expireIfNeeded` (`db.c:2935`). It first asks `keyIsExpired`
(`db.c:2877`, comparing `getExpire` at `db.c:2762` to the current time); if not expired it
returns `KEY_VALID`. Crucially, on a replica it does **not** delete — `db.c:2974-2977`
returns `KEY_EXPIRED` so the key is removed only when the master's DEL arrives; on a master
it calls `deleteExpiredKeyAndPropagate` (`db.c:2996`, def 2826) and returns `KEY_DELETED`
(`db.c:3003`). Callers branch on that `keyStatus` enum (`db.c:43-48`): `lookupKey`
(`db.c:302`) — the path every read/write key access goes through — treats anything but
`KEY_VALID` as a cache miss; `dbRandomKey` (`db.c:834`) retries past an expired pick;
`delGenericCommand` (`db.c:1419`) counts a `KEY_DELETED` into its reply; and `scanCallback`
(`db.c:1702`) drops expired entries while scanning. So a past-TTL key is removed during
ordinary handling precisely when one of these access points runs on a master and
`expireIfNeeded` reaches its deletion branch.
