# L3 reference answer key — inbound command path (socket bytes → handler)

Pinned source: `experiment/repos/redis` (SHA d2d3390d…). All cites re-verified.

A single ordered chain across `networking.c` and `server.c`. Order is gradeable.

## Required spine (gradeable core — must hit all, in order)

1. **Read handler entry** — `readQueryFromClient` `networking.c:3830`, installed via
   `connSetReadHandler(conn, readQueryFromClient)` (`networking.c:133`). It reads bytes off
   the socket with `connRead(c->conn, c->querybuf+qblen, readlen)` (`networking.c:3915`)
   into `c->querybuf`.
2. **Drive input processing** — `readQueryFromClient` calls `processInputBuffer(c)`
   (`networking.c:3971`); def `processInputBuffer` `networking.c:3626`.
3. **Protocol parse into argv/argc** — inside the `processInputBuffer` loop:
   `processMultibulkBuffer` (`networking.c:3701`, def `3214`) for RESP arrays (`*`), else
   `processInlineBuffer` (`networking.c:3687`, def `3063`). These fill `c->argv` / `c->argc`.
4. **Hand a complete command to execution** — `processInputBuffer` calls
   `processCommandAndResetClient(c)` (`networking.c:3787`, def `3491`), which calls
   `processCommand(c)` (`networking.c:3495`).
5. **Resolve command name → implementation** — `processCommand` `server.c:4412` calls
   `lookupCommand(c->argv, c->argc)` (`server.c:4455`, def `3609`) and stores it in `c->cmd`.
6. **Execute** — `processCommand` calls `call(c, flags)` (`server.c:4788`); `call`
   (`server.c:3949`) invokes the handler `c->cmd->proc(c)` (`server.c:4015`).

## Acceptable extras (credit, never required)

- The iothread pre-parse layer (`preprocessCommand` / `pendingCommand`, around
  `networking.c:3720` and the `pcmd` parameters threaded through process*Buffer) — modern
  Redis offloads parsing; mentioning it is a plus, never required.
- `commandProcessed` (`networking.c:3445`) reset step after a command runs.
- Arity / unknown-command / ACL / MULTI-queue checks inside `processCommand` (4458 onward;
  the MULTI-queue branch at `server.c:4784`).
- The command-reuse fast path (`isCommandReusable` / `c->lastcmd`, `server.c:4452`).

## Common misses / wrong turns

- Jumping straight to `call`/`processCommand` without the socket-read and parse front end
  (skips spine 1-3) — the prompt asks "from the moment bytes arrive."
- Confusing the lookup site: `lookupCommand` is also called from `commandGetKeys`-style
  helpers (e.g. `server.c:5712`) — the live dispatch call is `server.c:4455`.
- Naming the wrong terminal step (e.g. stopping at `processCommand`) instead of the actual
  handler invocation `c->cmd->proc(c)` (`server.c:4015`).
- Treating it as a star of callers rather than an ordered chain.

## Completeness scale

- **Full:** all 6 spine stops, in order, read → parse → dispatch-resolve → call → proc,
  with cites; the terminal `c->cmd->proc(c)` named.
- **Partial:** ≥4 of 6 in roughly correct order, but missing either the read front end OR
  the final handler invocation, OR with the parse step left vague.
- **Miss:** only the server.c dispatch half, or an out-of-order/incorrect chain.

## Model answer

When the socket becomes readable, the registered handler `readQueryFromClient`
(`networking.c:3830`, installed at `networking.c:133`) reads bytes via `connRead`
(`networking.c:3915`) into `c->querybuf`, then calls `processInputBuffer`
(`networking.c:3971`/def 3626). That loop parses the buffered protocol — RESP arrays via
`processMultibulkBuffer` (`networking.c:3701`/3214) or the inline form via
`processInlineBuffer` (3687/3063) — populating `c->argv`/`c->argc`. For each fully parsed
command it calls `processCommandAndResetClient` (3787/3491), which calls `processCommand`
(`server.c:4412`). `processCommand` resolves the command name to its `redisCommand` with
`lookupCommand(c->argv, c->argc)` (`server.c:4455`/def 3609), sets `c->cmd`, and (unless the
command is queued for MULTI) calls `call(c, flags)` (`server.c:4788`). `call`
(`server.c:3949`) finally invokes the handler through `c->cmd->proc(c)` (`server.c:4015`).
(Recent Redis can pre-parse on an I/O thread, but this is the main-thread spine.)
