# L5 reference answer key — a write's effect propagating to replicas (cross-cutting)

Pinned source: `experiment/repos/redis` (SHA d2d3390d…). All cites re-verified.

Threads three subsystems: command execution (`server.c call`), the propagation machinery
(`server.c` also_propagate / postExec / propagateNow), and the replication transport
(`replication.c`). The deferred queue-then-flush indirection is the crux.

## Required spine (gradeable core — must hit all)

1. **Execute the handler inside `call`** — `call` (`server.c:3949`) runs the command via
   `c->cmd->proc(c)` (`server.c:4015`).
2. **Detect that data was modified (dirty accounting)** — `call` snapshots
   `dirty = server.dirty` before (`server.c:3978`) and recomputes
   `dirty = server.dirty - dirty` after (`server.c:4032`); the command counts as a write iff
   `dirty` changed.
3. **Queue the replicated op (deferred, not sent inline)** — when dirty, `call` sets
   propagate targets `if (dirty) propagate_flags |= (PROPAGATE_AOF|PROPAGATE_REPL)`
   (`server.c:4131`) and calls `alsoPropagate(c->db->id, c->argv, c->argc, propagate_flags)`
   (`server.c:4153`), which appends to the `server.also_propagate` op array via
   `redisOpArrayAppend` (`server.c:3744`); def `alsoPropagate` `server.c:3732`.
4. **Flush at the end of the execution unit** — `call` calls `afterCommand(c)`
   (`server.c:4189`, def `4271`) → `postExecutionUnitOperations` (`server.c:4274`, def `3867`)
   → `propagatePendingCommands` (`server.c:3875`, def `3811`), which (wrapping multiple ops
   in MULTI/EXEC, `server.c:3836`/`3847`) calls `propagateNow` (def `server.c:3704`).
5. **Fan out to the sinks** — `propagateNow` calls `replicationFeedSlaves(server.slaves,
   dbid, argv, argc)` (`server.c:3716`) for the replication target.
6. **Turn the op into replication-stream bytes** — `replicationFeedSlaves`
   (`replication.c:631`) serializes argv to RESP and appends via `feedReplicationBuffer`
   (`replication.c:773`, def `618`), injecting a `SELECT` when the dbid changed
   (`replication.c:682-688`), into the backlog / per-replica output buffers feeding
   connected replicas.

## Acceptable extras (credit, never required)

- **AOF as the parallel sink:** `propagateNow` also calls `feedAppendOnlyFile(dbid,argv,argc)`
  when AOF is on (`server.c:3713-3714`). Credit if mentioned; the replica path is required.
- Implicit deletions (expire/evict) entering the same machinery via `propagateDeletion`
  (`db.c:2854`) → `alsoPropagate` (`db.c:2866`).
- `shouldPropagate` gating and `server.replication_allowed` (`propagateNow` `server.c:3705`).
- `forceCommandPropagation` / `CLIENT_FORCE_REPL` / `preventCommandPropagation`
  (`server.c:3750`, `4135-4136`) as overrides of the dirty-based default.
- The replica-side application path (`readSyncBulkPayload` `replication.c:2253`, the master
  link applying the stream) — bonus; the prompt centers the primary's outbound journey.
- `master_repl_offset` advancing as bytes are fed.

## Common misses / wrong turns

- Claiming the command handler calls replication directly — it does not; the indirection
  through `server.dirty` → `also_propagate` → flush is the whole point and a likely miss.
- Stopping at `alsoPropagate` without the `postExecutionUnitOperations` /
  `propagatePendingCommands` flush — the op array would never be sent in that account.
- Missing the dirty-based write detection (spine 2) — i.e. not explaining *why* only
  mutating commands propagate.
- Citing `replicationFeedSlaves` but not connecting it back to `propagateNow`/the flush, or
  vice-versa — failing to bridge the propagation and transport subsystems.
- Conflating with full-sync/RDB transfer (the SYNC/PSYNC bootstrap) rather than the
  steady-state command-stream propagation the prompt describes.

## Completeness scale

- **Full:** all 6 spine elements connected — proc run, dirty detection, deferred
  `alsoPropagate` into `also_propagate`, flush via afterCommand→postExec→propagatePending→
  propagateNow, and `replicationFeedSlaves`→`feedReplicationBuffer` producing stream bytes —
  with cites, and explicit acknowledgement of the queue-then-flush indirection.
- **Partial:** ≥4 of 6, but missing either the dirty-detection rationale OR the
  postExec/propagatePending flush bridge (i.e. an account that doesn't explain how the
  queued op actually leaves the box).
- **Miss:** a direct "handler → replicationFeedSlaves" story, or only the SYNC/full-resync
  bootstrap.

## Model answer

Inside `call` (`server.c:3949`) the handler runs via `c->cmd->proc(c)` (4015). `call`
brackets it with `dirty = server.dirty` (3978) and `dirty = server.dirty - dirty` (4032),
so a command is treated as a write only when it bumped the dirty counter. On that condition
`call` sets `propagate_flags |= PROPAGATE_AOF|PROPAGATE_REPL` (4131) and calls
`alsoPropagate(c->db->id, c->argv, c->argc, …)` (4153) — which does **not** send anything,
it appends the op to `server.also_propagate` (`redisOpArrayAppend`, 3744; def 3732). The
actual send is deferred to the end of the execution unit: `call`→`afterCommand` (4189/def
4271)→`postExecutionUnitOperations` (4274/def 3867)→`propagatePendingCommands` (3875/def
3811), which (wrapping multiple ops in MULTI/EXEC, 3836/3847) calls `propagateNow` (def
3704). `propagateNow` feeds AOF via `feedAppendOnlyFile` (3713) and, for replicas, calls
`replicationFeedSlaves(server.slaves, dbid, argv, argc)` (3716). That function
(`replication.c:631`) serializes the argv into RESP, injects a `SELECT` if the db changed
(682-688), and appends the bytes with `feedReplicationBuffer` (773/def 618) into the
backlog and each replica's output buffer — which is what travels over the replication link
to connected replicas. (Implicit DELs from expiry/eviction join the same machinery via
`propagateDeletion`, `db.c:2854`.)
