The crux is the branch on `Poll::Ready` (proceed) vs `Poll::Pending` (forced yield).

## Required spine (gradeable core — must hit all)

1. **The focal routine is `poll_proceed(cx) -> Poll<RestoreOnPending>`** — `coop/mod.rs:343`. It reads
   the per-task budget cell (`context::budget(...)`, `:344`) and `budget.decrement()` (`:346`).
2. **The `Budget` + `BudgetDecrement` contract drives the branch** — `Budget(Option<u8>)`
   (`coop/mod.rs:97`); `decrement()` (`:413`) returns `BudgetDecrement { success, hit_zero }`
   (`:99-101`): `success = true` while budget remains (`:420`), `success = false` when exhausted
   (`:422`); `hit_zero` marks the transition to zero (`:418`).
3. **The branch — proceed vs forced yield** — in `poll_proceed`: if `decrement.success`
   (`coop/mod.rs:349`) it commits the decremented budget, builds a `RestoreOnPending` guard, and
   returns `Poll::Ready(restore)` (`:358`, the caller may proceed — budget was consumed); if it ran out
   (`else`, `:359`) it calls `register_waker(cx)` (`:360`) and returns `Poll::Pending` (`:361`) — the
   task yields back to the scheduler. If no budget cell exists (not in a runtime) it returns
   `Poll::Ready(RestoreOnPending::new(Budget::unconstrained()))` (`:361`).
4. **The `RestoreOnPending` guard + `made_progress`** — `RestoreOnPending` (`coop/mod.rs:260`) captures
   the pre-decrement budget; its `made_progress()` (`:273`) is what a caller invokes when it actually
   made forward progress (so an exhausted budget can be replenished), versus a no-op await that yields.
5. **The forced-yield metric** — when `decrement.hit_zero`, `inc_budget_forced_yield_count()` is called
   (`coop/mod.rs:354-355`) so the runtime records a budget-induced yield.
6. **The callers (≥4 required), and what each does with the outcome —** each uses the
   `ready!(coop::poll_proceed(cx))` pattern, which early-returns `Pending` (yielding) when
   `poll_proceed` returns `Pending`, and on `Ready` keeps the `RestoreOnPending` guard and marks
   progress once real work is done:

     (`let coop = ready!(crate::task::coop::poll_proceed(cx));`).
   - Join-handle poll — `runtime/task/join.rs:332`.
   - `tokio::io` copy — `io/util/copy.rs:97`; async-read helpers — `io/util/mem.rs:335`.
   - channel receive — `sync/mpsc/chan.rs:295`; semaphore acquire — `sync/batch_semaphore.rs:598`;
     oneshot receive — `sync/oneshot.rs:824`.
   - child wait — `process/mod.rs:1140`; sleep poll — `time/sleep.rs:402`.

  `poll_proceed` is `Pending` (the synthesis the prompt asks for).

## Model answer

The cooperative-scheduling gate is `coop::poll_proceed(cx) -> Poll<RestoreOnPending>`
(`coop/mod.rs:343`). It reads the per-task `Budget(Option<u8>)` (`:97`) and calls `decrement()`
(`:413`), which returns `BudgetDecrement { success, hit_zero }` (`:99-101`): `success` is true while
budget remains (`:420`) and false once it's exhausted (`:422`). `poll_proceed` branches on that: if
`decrement.success` (`:349`) it commits the new budget, builds a `RestoreOnPending` guard
(`coop/mod.rs:260`), and returns `Poll::Ready(restore)` (`:358`) — the caller is allowed to keep
polling and, having made real progress, calls `restore.made_progress()` (`:273`); if budget ran out
(`:359`) it instead calls `register_waker(cx)` (`:360`) and returns `Poll::Pending` (`:361`), forcing
the task to yield, and (when `hit_zero`) bumps the forced-yield counter (`:354-355`). Call sites
consume this with the `ready!(coop::poll_proceed(cx))` pattern: when `poll_proceed` is `Pending` the
`ready!` macro early-returns `Pending` from the caller's own poll, yielding; when `Ready` they hold the
`RestoreOnPending` guard and call `made_progress()` after doing real work. This appears at I/O
readiness (`runtime/io/registration.rs:151`), the join poll (`runtime/task/join.rs:332`), `io::copy`
(`io/util/copy.rs:97`), channel receive (`sync/mpsc/chan.rs:295`), semaphore acquire
(`sync/batch_semaphore.rs:598`), oneshot receive (`sync/oneshot.rs:824`), child wait
(`process/mod.rs:1140`), and `Sleep` (`time/sleep.rs:402`) — i.e. at the entry of every long-running
poll, so a task that loops without awaiting is forced to yield once its budget runs out.
