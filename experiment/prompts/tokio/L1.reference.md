# L1 reference answer key — the per-task in-memory container (`Cell<T, S>`)

Pinned source: `experiment/repos/tokio` (SHA 66e29121b3…). All cites re-verified.

The entity is `Cell<T, S>` in `tokio/src/runtime/task/core.rs` — the single heap allocation
that holds one task. It is referenced type-erased through `RawTask { ptr: NonNull<Header> }`
(`raw.rs:20`).

## Required spine (gradeable core — must hit all)

1. **The container is `Cell<T: Future, S>`** — `core.rs:126` (`pub(super) struct Cell<T: Future, S>`).
   A cache-line-aligned (`#[repr(C)]` + align attributes above it), single allocation laid out as
   three regions.
2. **Three-region layout: `header`, `core`, `trailer`** — `core.rs:128` (`pub(super) header: Header`),
   `core.rs:131` (`pub(super) core: Core<T, S>`), `core.rs:134` (`pub(super) trailer: Trailer`).
   `Header` must be first (the doc comment at `core.rs:~104` notes the structure is referenced by both
   `*mut Cell` and `*mut Header`).
3. **`Header` = hot, type-erased bookkeeping** — `core.rs:168` (`pub(crate) struct Header`), holding
   - `state: State` (`core.rs:170`) — the atomic ref-count + lifecycle bits (defined in `state.rs`),
   - `queue_next: UnsafeCell<Option<NonNull<Header>>>` (`core.rs:173`) — the injection-queue link,
   - `vtable: &'static Vtable` (`core.rs:176`) — the function-pointer table,
   - `owner_id: UnsafeCell<Option<NonZeroU64>>` (`core.rs:189`) — which `OwnedTasks` list owns it.
   `Header` carries no generic type parameter — it is the type-erased face of the task.
4. **`Core<T, S>` = the typed region** — `core.rs:148` (`pub(super) struct Core<T: Future, S>`),
   holding `scheduler: S` (`core.rs:150`, the scheduler driving the future), `task_id: Id`
   (`core.rs:153`), and `stage: CoreStage<T>` (`core.rs:163`) wrapping `UnsafeCell<Stage<T>>`.
5. **`Stage<T>` = the field that flips future↔output** — `core.rs:221`
   (`pub(super) enum Stage<T: Future>`): `Running(T)` (`:222`, holding the future),
   `Finished(super::Result<T::Output>)` (`:223`, holding the produced output),
   `Consumed` (`:224`, the output has been read out). `Cell::new` starts in `Stage::Running(future)`
   (`core.rs:267`); on completion the harness writes `Stage::Finished(output)` (`core.rs:406`) and the
   join path replaces with `Stage::Consumed` (`core.rs:394`/`:420`).
6. **`Trailer` = cold, consumer-facing data** — `core.rs:201` (`pub(super) struct Trailer`), holding
   `owned: linked_list::Pointers<Header>` (`core.rs:203`, links in the owner's `OwnedTasks` list),
   `waker: UnsafeCell<Option<Waker>>` (`core.rs:205`, the `JoinHandle` consumer's waker), and
   `hooks: TaskHarnessScheduleHooks` (`core.rs:208`).
7. **Type-erasure bridge: the `Vtable` offsets** — `Vtable` (`raw.rs:24`) stores the per-`T`,`S`
   function pointers (`poll`, `schedule`, `dealloc`, …) AND the byte offsets to reach the typed
   regions from a bare `*mut Header`: `trailer_offset` (`raw.rs:47`), `scheduler_offset`
   (`raw.rs:50`, the first field of `Core`), `id_offset` (`raw.rs:53`). So the runtime holds only a
   `NonNull<Header>` (`RawTask`, `raw.rs:20`) and reaches the typed `Core`/`Trailer` through these
   offsets — the type-erased `Header` vs typed `Core`/`Trailer` split is the load-bearing distinction.

## Acceptable extras (credit, never required)

- The cache-line alignment rationale (the `repr(align(128))`/`align(64)` attributes above
  `Cell`, `core.rs:~50-124`) — avoiding false sharing on the hot header.
- `OffsetHelper<T,S>` const-computing the offsets (`raw.rs:~75-110`), and the comment that the
  vtable stores offsets because the pointer is type-erased.
- `generate_addr_of_methods!` macro generating the offset accessors (`core.rs:~213`).
- The `Cell::new` constructor (`core.rs:~235`) building header/core/trailer together.

## Common misses / wrong turns

- Describing only the `RawTask` `NonNull<Header>` pointer and not the three-region layout behind it —
  that is the handle, not the container.
- Listing `Header` fields but missing the **vtable-offset bridge** (spine 7): the crux is *how* the
  type-erased `Header` reaches the typed `Core`/`Trailer`. Without it the type/encoding-style
  distinction is lost.
- Treating the future and the output as separate fields — they share the single `Stage<T>` slot
  (`core.rs:221-224`), which flips Running→Finished→Consumed; this is the "single field that flips"
  the prompt names.
- Pointing at `Task`/`Notified`/`JoinHandle` (the harness wrappers in `raw.rs`/`join.rs`) instead of
  the underlying `Cell` allocation — those are views over the cell, not the cell.

## Model answer

A spawned task lives in one heap allocation, `Cell<T, S>` (`core.rs:126`), laid out `#[repr(C)]` into
three contiguous regions: `header: Header` (`:128`), `core: Core<T, S>` (`:131`), and
`trailer: Trailer` (`:134`). The first region, `Header` (`:168`), is **type-erased** — it holds the hot
bookkeeping every task shares regardless of `T`/`S`: the atomic `state: State` (`:170`, ref counts +
lifecycle), the injection-queue link `queue_next` (`:173`), the `vtable: &'static Vtable` (`:176`),
and `owner_id` (`:189`). The runtime only ever stores a `NonNull<Header>` (`RawTask`, `raw.rs:20`), so
the typed regions are reached **by offset**: the `Vtable` (`raw.rs:24`) carries `trailer_offset`
(`:47`), `scheduler_offset` (`:50`), and `id_offset` (`:53`) (computed by `OffsetHelper<T,S>`), letting
a bare header pointer find the `Core`/`Trailer`. `Core<T, S>` (`:148`) holds `scheduler: S` (`:150`),
`task_id: Id` (`:153`), and `stage: CoreStage<T>` (`:163`) wrapping the `Stage<T>` enum (`:221`) — the
single field that flips between `Running(T)` (`:222`, the live future), `Finished(Result<T::Output>)`
(`:223`, the produced output), and `Consumed` (`:224`, output drained); `Cell::new` starts it at
`Stage::Running` (`:267`). `Trailer` (`:201`) is the cold, consumer-facing tail: the `OwnedTasks`
linked-list pointers `owned` (`:203`), the `JoinHandle` consumer's `waker` (`:205`), and `hooks`
(`:208`). So the task is a type-erased header (for the runtime) plus a typed core/trailer (for the
future and the joiner), bridged by vtable offsets.
