# L3 reference answer key ‚Äî I/O readiness wake path (mio event ‚Üí task rescheduled)

Pinned source: `experiment/repos/tokio` (SHA 66e29121b3‚Ä¶). All cites re-verified.

A single ordered chain through `runtime/io/driver.rs` and `runtime/io/scheduled_io.rs`. Order is
gradeable.

## Required spine (gradeable core ‚Äî must hit all, in order)

1. **Park entry ‚Üí `turn`** ‚Äî `Driver::park` (`runtime/io/driver.rs:159`) / `park_timeout` (`:164`)
   call `turn(handle, max_wait)` (`:179`). `Driver` holds the `mio::Poll` (`driver.rs:33`).
2. **System poll for ready events** ‚Äî `turn` calls `self.poll.poll(events, max_wait)`
   (`driver.rs:188`), blocking in the OS (epoll/kqueue) for readiness up to `max_wait`; the returned
   events fill `self.events`. (`TOKEN_WAKEUP`/`TOKEN_SIGNAL` are reserved tokens, `:103-104`.)
3. **Iterate events and decode each** ‚Äî `for event in events.iter()` (`driver.rs:201`), taking
   `event.token()` (`:202`); for a real resource event (not wakeup/signal) it computes
   `let ready = Ready::from_mio(event)` (`:209`).
4. **Token ‚Üí `ScheduledIo` (the resource)** ‚Äî `let ptr = super::EXPOSE_IO.from_exposed_addr(token.0)`
   (`driver.rs:210`) recovers the `*const ScheduledIo` that was stored as the mio token, cast to
   `&ScheduledIo` (`driver.rs:216`) ‚Äî mio tokens are pointers to the per-resource state.
5. **Set readiness + wake waiters on that resource** ‚Äî `io.set_readiness(Tick::Set, |curr| curr | ready)`
   (`driver.rs:218`; `ScheduledIo::set_readiness` at `scheduled_io.rs:207`) ORs the new readiness in,
   then `io.wake(ready)` (`driver.rs:219`; `ScheduledIo::wake` at `scheduled_io.rs:236`).
6. **`wake` dispatches to the registered waker(s)** ‚Äî `ScheduledIo::wake` (`scheduled_io.rs:236`)
   locks `self.waiters` (`:239`); if readable it takes `waiters.reader` (`:243`), if writable
   `waiters.writer` (`:250`), and drains `waiters.list` filtered by `ready.satisfies(w.interest)`
   (`:256`), pushing their wakers into a `WakeList`; it drops the lock (`:274`) and calls
   `wakers.wake_all()` (`:285`, final; `:276` mid-loop batch).
7. **Waker ‚Üí scheduler hands the task back to a run queue** ‚Äî each waker is the task's `Waker`; waking
   it calls the `Schedule::schedule` implementation (e.g. the multi-thread
   `Handle::schedule` at `scheduler/multi_thread/handle.rs:110` ‚Üí `schedule_task` at
   `scheduler/multi_thread/worker.rs:1327`), which pushes the `Notified` task onto the local run
   queue or the injection queue and notifies a parked worker ‚Äî the task is now runnable again.

## Acceptable extras (credit, never required)

- `Driver::unpark` (`driver.rs:~262`) ‚Äî the waker stored in the driver used to interrupt a blocked
  `turn`.
- `Handle::add_source` (`driver.rs:~280`) ‚Äî the registration side that allocates the `ScheduledIo` and
  the token (the inverse of spine 4), mentioned as contrast.
- The uring dispatch branch in `turn` (`driver.rs:~219`, `ctx.dispatch_completions()`) under
  `io-uring` ‚Äî bonus, never required.
- `metrics.incr_ready_count_by(ready_count)` (`driver.rs:~228`).

## Common misses / wrong turns

- Jumping straight to "the task wakes up" without the mio `poll` ‚Üí event iteration front end
  (spine 1-3) ‚Äî the prompt asks "from the moment the system poll reports ready."
- Missing the token‚Üí`ScheduledIo` decode (spine 4): mio tokens are not opaque ints here, they are
  the `from_exposed_addr` of the resource's pointer; skipping this loses the link between event and
  resource.
- Confusing `set_readiness` (the cached atomic readiness, spine 5) with `wake` (notifying waiters);
  both happen per event and the prompt names both ("readiness update and a wake").
- Stopping at `wake` and not connecting it to the scheduler rescheduling the task (spine 7) ‚Äî the
  prompt asks "until the task ‚Ä¶ is scheduled to run again."
- Treating it as a star of callers rather than an ordered chain.

## Model answer

When a worker has no work it parks on the I/O driver: `Driver::park` (`runtime/io/driver.rs:159`)
calls `turn` (`:179`), which blocks on `self.poll.poll(events, max_wait)` (`:188`) ‚Äî the OS
(epoll/kqueue) reports which registered resources are ready. For each ready `event`
(`:201`) it takes `event.token()` (`:202`) and, for a real resource (not the reserved wakeup/signal
tokens), computes `Ready::from_mio(event)` (`:209`). The token is a stored pointer:
`EXPOSE_IO.from_exposed_addr(token.0)` (`:210`) recovers the `&ScheduledIo` (`:216`) for that
resource. `turn` then `io.set_readiness(Tick::Set, |curr| curr | ready)` (`:218`,
`ScheduledIo::set_readiness` `scheduled_io.rs:207`) caches the new readiness, and
`io.wake(ready)` (`driver.rs:219`, `ScheduledIo::wake` `scheduled_io.rs:236`) notifies the waiters:
under the `waiters` lock (`:239`) it takes the read-slot waker (`:243`), the write-slot waker
(`:250`), and any list waiters whose `interest` is satisfied (`:256`), drops the lock (`:274`) and
calls `wakers.wake_all()` (`:285`). Each waker is the parked task's `Waker`; waking it calls the
scheduler's `Schedule::schedule` (e.g. `Handle::schedule` `scheduler/multi_thread/handle.rs:110` ‚Üí
`schedule_task` `scheduler/multi_thread/worker.rs:1327`), which pushes the `Notified` task onto the
local run queue (or injection queue if remote) and notifies a parked worker ‚Äî so the task that was
awaiting the I/O resource is put back onto a run queue to be polled again.
