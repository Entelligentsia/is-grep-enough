# L4 reference answer key ‚Äî multi-thread work-stealing scheduler subsystem

Pinned source: `experiment/repos/tokio` (SHA 66e29121b3‚Ä¶). All cites re-verified.

A bounded cluster across `scheduler/multi_thread/{handle,worker,queue,inject,idle,overflow}.rs`.
Several cooperating paths around a per-worker local queue, a shared injection queue, and an
idle/searching coordinator; order within each path matters less than covering all five pieces.

## Required spine (gradeable core ‚Äî must hit all five pieces)

1. **Task placement on schedule (local vs shared injection queue)** ‚Äî a notified task enters via the
   `Schedule` impl: `Handle::schedule` (`scheduler/multi_thread/handle.rs:110`) and `yield_now`
   (`:120`) both call `schedule_task(task, is_yield)` (`worker.rs:1327`). `schedule_task` checks the
   runtime context: if this is the current worker and it still holds a `core`, it calls
   `schedule_local` (`worker.rs:1353`); otherwise it `push_remote_task` (`worker.rs:1397`, pushing to
   the shared `inject` queue) and `notify_parked_remote` (`worker.rs:1453`). `schedule_local` either
   puts the task in the LIFO `lifo_slot` (for non-yield, LIFO-on) or `run_queue.push_back_or_overflow`
   (`worker.rs:1362`/`:1371`), then `notify_parked_local` (`worker.rs:1441`) if a worker is parked.
2. **Worker run loop + task selection** ‚Äî `run(worker)` (`worker.rs:508`) ‚Üí `Worker::run` (`:561`) loops
   `tick`/`maintenance`, then `core.next_task(&worker)` (`:572`): `next_task` (`worker.rs:1062`)
   periodically (every `global_queue_interval` ticks) checks the injection queue first via
   `next_remote_task` (`worker.rs:1387`, `inject.pop`) then falls back to local; otherwise it tries
   `next_local_task` (`:1132`, `lifo_slot.take().or_else(|| run_queue.pop())`) and, when the local queue
   is empty but inject is non-empty, drains a batch from inject (`inject.pop_n`) back into the local
   queue (`:~1080-1110`). Found work runs via `run_task` (`worker.rs:630`).
3. **Work stealing between workers** ‚Äî when a worker has no local or injected work, `steal_work`
   (`worker.rs:1141`) calls `transition_to_searching` (`worker.rs:1171` ‚Üí
   `idle.transition_worker_to_searching`) and then `run_queue.steal` (`queue.rs`) from a randomly
   chosen peer worker (`:~1155`), returning a stolen task to run.
4. **Idle coordination (who searches, who is parked, who gets woken)** ‚Äî `Idle` (`idle.rs:9`) tracks a
   `State` (`idle.rs:31`) of searching/parked counts. `transition_worker_to_searching`
   (`idle.rs:104`) caps searchers at ~half the workers (`2 * num_searching() >= num_workers`, `:106`);
   `transition_worker_from_searching` (`idle.rs:121`) returns whether this was the last searcher (the
   caller must then notify a new worker); `worker_to_notify` (`idle.rs:51`) picks a parked worker to
   wake; `notify_parked_local`/`notify_parked_remote` (`worker.rs:1441`/`:1453`) call it and
   `remotes[index].unpark.unpark(&driver)` (`:1446`) to wake that worker; `transition_to_parked`
   (`worker.rs:1204`) / `transition_from_parked` (`:1231`) move a worker in/out of the parked set
   (`idle.sleepers`, via `unpark_worker_by_id` `idle.rs:129` / `unpark_one` `:172`).
5. **Overflow of the local queue into the injection queue** ‚Äî `run_queue.push_back_or_overflow`
   (`queue.rs:188`) pushes onto the local queue, and when it is full calls `push_overflow`
   (`queue.rs:253`) through the `Overflow<T>` trait (`overflow.rs:9`), whose multi-thread impl routes
   the surplus tasks to the shared `inject` queue (`push_remote_task` path); a later `next_task`
   batch-drain (spine 2) pulls them back. The `inject` queue is also where remote spawns go (spine 1),
   so overflow and remote-submitted tasks share one global queue.

## Acceptable extras (credit, never required)

- `Handle::bind_new_task` (`handle.rs:81`) and `spawn` (`handle.rs:55`) as the spawn entry that first
  schedules a fresh task ‚Äî the spawn-side precursor to spine 1.
- The `lifo_slot` LIFO optimization and `core.lifo_enabled` gating (`worker.rs:~1359-1373`) ‚Äî bonus.
- `next_task`'s `global_queue_interval` / `tune_global_queue_interval` (`worker.rs:~1064-1067`) ‚Äî how
  often the injection queue is checked before the local one.
- `maintenance` (`worker.rs:780`/`:1262`) draining deferred tasks/stats between polls.
- `Idle::State` bit-packing (`idle.rs:31`/`:160-208`) of num_searching/num_unparked/sleepers.

## Common misses / wrong turns

- Describing only the per-worker run loop and missing the **injection queue** as the shared channel
  for remote spawns AND overflow (spine 1/5) ‚Äî without it, "no central dispatcher" is unexplained.
- Missing the work-stealing path (spine 3) ‚Äî the whole reason idle workers stay busy is stealing from
  peers, not just draining inject.
- Conflating the local queue (`queue.rs`, FIFO-ish + LIFO slot) with the injection queue (`inject/`)
  ‚Äî they are distinct; the local queue is per-worker, the injection queue is shared.
- Omitting the idle/searching coordination (spine 4): the `2 * num_searching >= num_workers` cap and
  the "last searcher must notify" rule are what keep stealing from thrashing and ensure parked
  workers are woken.
- Treating overflow as a separate subsystem rather than the same injection queue remote spawns use.

## Completeness scale

- **Full:** all five pieces ‚Äî schedule placement (local vs inject), worker run loop + task selection
  (local/inject/steal), work stealing, idle coordination (searching cap + notify parked), and
  overflow ‚Üí inject ‚Äî connected, with cites, and the "no central dispatcher" rationale (workers pull
  work; parked workers are woken on demand).
- **Partial:** 4 of 5 pieces (commonly missing the idle coordination OR the overflow/inject unification).
- **Miss:** only "workers run tasks from a queue" with no inject/steal/idle story.

## Model answer

A notified task enters via the `Schedule` impl: `Handle::schedule` (`handle.rs:110`) calls
`schedule_task` (`worker.rs:1327`), which, if the current thread is the owning worker with a live
`core`, calls `schedule_local` (`worker.rs:1353`) ‚Äî pushing to the LIFO `lifo_slot` or
`run_queue.push_back_or_overflow` (`:1362`) and `notify_parked_local` (`:1441`); otherwise it
`push_remote_task` (`worker.rs:1397`) onto the shared `inject` queue and `notify_parked_remote`
(`:1453`). Each worker thread runs `Worker::run` (`worker.rs:561`): per tick it calls `next_task`
(`:1062`), which (every `global_queue_interval` ticks) checks the injection queue first
(`next_remote_task` `:1387`) then the local queue, and otherwise tries `next_local_task`
(`lifo_slot` then `run_queue.pop`, `:1132`), batch-draining `inject` back into the local queue when
local is empty; found work runs in `run_task` (`:630`). When local and inject are both empty, the
worker `steal_work` (`:1141`) `transition_to_searching` (`:1171`) and steals from a random peer's
`run_queue`. `Idle` (`idle.rs:9`) coordinates: `transition_worker_to_searching` (`:104`) caps
searchers at ~half (`2*num_searching >= num_workers`), `transition_worker_from_searching` (`:121`)
makes the last searcher notify a replacement, and `worker_to_notify` (`:51`) + `notify_parked_local`
(`worker.rs:1441`/`:1446`) wake a parked worker (`unpark_worker_by_id` `idle.rs:129`); a worker with
no work `transition_to_parked` (`worker.rs:1204`) sleeps until notified. If `push_back_or_overflow`
(`queue.rs:188`) finds the local queue full it `push_overflow` (`queue.rs:253`) via the `Overflow`
trait (`overflow.rs:9`) into the same shared `inject` queue, later batch-drained by `next_task`. So
work is placed locally when possible, shared via inject otherwise, pulled by each worker from its own
queue / inject / stolen peers, and parked workers are woken on demand ‚Äî no central dispatcher.
