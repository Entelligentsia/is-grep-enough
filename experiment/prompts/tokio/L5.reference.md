# L5 reference answer key — task awaiting I/O + timer, woken and resumed (cross-cutting)

Pinned source: `experiment/repos/tokio` (SHA 66e29121b3…). All cites re-verified.

Threads four subsystems: the scheduler (`scheduler/multi_thread/{handle,worker}.rs`), the I/O
driver (`runtime/io/{driver,scheduled_io}.rs`), the time driver (`runtime/time/{mod,entry}.rs`), and
the task cell (`runtime/task/core.rs`). The combined-driver park is the integration crux.

## Required spine (gradeable core — must hit all)

1. **Spawn + first poll** — `Handle::spawn` (`scheduler/multi_thread/handle.rs:55`) →
   `bind_new_task` (`:81`) → `schedule_option_task_without_yield` (`worker.rs:1347`) →
   `schedule_task` (`worker.rs:1327`) places the `Notified` task on a run queue (local or inject,
   `schedule_local` `:1353` / `push_remote_task` `:1397`) and notifies a parked worker. A worker's
   `run_task` (`worker.rs:630`) polls the future under `coop::budget` (spine-1 of L2).
2. **Park on the I/O resource** — the awaited TCP read future polls
   `ScheduledIo::readiness(interest)` (`runtime/io/scheduled_io.rs:382`) → `Readiness::poll`
   (`:427`): if not already ready, it locks `waiters`, stores the task waker
   `waker.replace(cx.waker().clone())` (`:495`) and inserts the waiter into `waiters.list`
   (`:~502`), returning `Poll::Pending` — the task registers itself with the I/O driver and parks.
3. **Park on the timer** — the `Sleep` future's `poll_elapsed` (`time/sleep.rs:391`) on first poll
   builds `Timer::new(handle, deadline)` (`time/sleep.rs:424`) and `timer.init(deadline)` (`:427`) →
   `TimerEntry::init` (`runtime/time/entry.rs:481`) registers the entry into the wheel via
   `self.driver().reregister(...)` (`:486`); `TimerEntry::poll_elapsed` (`:535`) calls
   `self.inner.state.poll(cx.waker())` (`:545`) → `TimerShared::poll` (`entry.rs:142`)
   `waker.register_by_ref(waker)` (`:146`) and returns `Poll::Pending` (`:160`) — the task registers
   its waker with the time driver and parks.
4. **Worker parks on the combined driver stack** — with no more work, `Worker::park`
   (`worker.rs:810`) → `park_internal` (`:844`) takes the runtime `Driver` from `core.park` and calls
   `park.park(&handle.driver)` (`:873`). That is `runtime::Driver::park` (`runtime/driver.rs:66`) →
   `TimeDriver::park` (`:328`) → `time::Driver::park` (`runtime/time/mod.rs:181`) → `park_internal`
   (`:213`): it computes `next_wake = wheel.next_expiration_time()` (`:219`) and parks for
   `min(limit, time_until_next_timer)` (`:~219-237`) by calling `self.park.park_timeout` — which is
   the **I/O driver** (`io::Driver::park_timeout` `runtime/io/driver.rs:164` → `turn` `:179` →
   `self.poll.poll(events, max_wait)` `:188`, the OS mio poll). So one park call drives both the I/O
   and time subsystems; whichever fires first unblocks it.
5. **Whichever resource is ready wakes the task** —
   - **I/O ready:** `io::Driver::turn` (`driver.rs:179`) decodes the mio event → `io.wake(ready)`
     (`:219`) → `ScheduledIo::wake` (`scheduled_io.rs:236`) drains the registered waiter(s)
     (`:256`) and `wakers.wake_all()` (`:285`) calls the task's stored waker.
   - **Timer expired:** after the park returns, `handle.process(clock)` (`runtime/time/mod.rs:255`)
     → `process_at_time` (`:296`) loops `wheel.poll(now)` (`:311`) and `entry.fire(Ok(()))`
     (`:315`) returns the stored waker, collected into a `WakeList` and `waker_list.wake_all()`
     (`:336`, final; `:322` mid-loop batch).
6. **Wake → reschedule → re-poll → complete** — the woken waker is the task's `Waker`, whose `wake`
   calls `Schedule::schedule` (`handle.rs:110`) → `schedule_task` (`worker.rs:1327`) pushing the
   task back onto a run queue and notifying a parked worker (`notify_parked_local/remote`
   `worker.rs:1441`/`:1453`). The unparked worker `next_task` (`worker.rs:1062`) → `run_task`
   (`:630`) re-polls the future: the awaited branch is now satisfied (`Readiness::poll` sees the
   cached readiness `Ready` and returns `ReadyEvent`, `scheduled_io.rs:~457`; or
   `TimerShared::poll` sees `STATE_DEREGISTERED` and returns the fired result, `entry.rs:~152`), so
   the future makes progress; on completion the harness writes `Stage::Finished(output)`
   (`runtime/task/core.rs:406`), `Schedule::release` (`handle.rs:106`) removes the task from
   `owned`, and the `JoinHandle` consumer's waker (in `Trailer.waker`, `core.rs:205`) is woken so the
   output is observable.

## Acceptable extras (credit, never required)

- The `coop::poll_proceed` budget gate that wraps the re-poll (`worker.rs` `coop::budget` in
  `run_task`) — bonus context, the L2 key owns it.
- `runtime::Driver::Handle::unpark` (`driver.rs:~88`) waking both the time and I/O driver on
  external unpark.
- `time::Handle::process_at_time`'s `WakeList` batching and lock-drop-to-wake deadlock avoidance
  (`runtime/time/mod.rs:~311-327`).
- Noting that the same task may park on *both* resources simultaneously (select/`tokio::select!`)
  and the *first* to fire wins; the other registration is cleaned up (timer `cancel`
  `entry.rs:496`, I/O waiter removal) — credit if mentioned, not required.
- The `time_alt` alternative wheel (`TimerFlavor::Alternative`, `driver.rs:~318`) — bonus; the
  traditional wheel above is the required spine.

## Common misses / wrong turns

- Treating the I/O driver and the time driver as two independent park calls — they are not; the time
  driver **wraps** the I/O driver and one `park_internal` (`runtime/time/mod.rs:213`) computes
  `min(io, timer)` and parks the I/O driver, then fires timers (`handle.process`). Missing this
  layered-park (spine 4) loses the whole cross-cutting crux.
- Stopping at "the task is woken" without connecting the waker back to `Schedule::schedule`
  (spine 6) — the prompt asks "rescheduled and polled again until it completes."
- Describing only the I/O wake (L3) without the timer registration/firing half (spine 3/5b) — the
  prompt is specifically about a task awaiting **both** a socket read and a timeout.
- Claiming the task itself polls the OS — it does not; the worker thread parks on the combined
  driver, and the driver's wake calls the task's stored waker (`ScheduledIo::waiters` /
  `TimerShared::waker`).
- Forgetting the task-cell lifecycle bookend (`Stage::Running` → `Stage::Finished`/`Consumed`,
  `core.rs:267`/`:406`) — the "until it completes" half of the prompt.

## Completeness scale

- **Full:** all 6 spine elements connected — spawn/first-poll → park on I/O (register waker in
  `waiters`) → park on timer (`TimerEntry::init` + `state.poll` register waker) → worker parks on
  the **combined** driver (time wraps I/O; `min(io,timer)` + `handle.process`) → wake from I/O
  (`ScheduledIo::wake`) or timer (`entry.fire` + `wake_all`) → reschedule (`Schedule::schedule`) →
  re-poll → `Stage::Finished`/release — with cites, and explicit acknowledgement of the layered
  combined driver.
- **Partial:** ≥4 of 6, but missing either the combined-driver park OR the timer registration/firing
  half (i.e. an account that covers only the I/O side, or only the scheduler side).
- **Miss:** a direct "task waits, then runs" story with no driver integration, no waker
  registration, or no reschedule link.

## Model answer

A task spawned with `Handle::spawn` (`handle.rs:55` → `bind_new_task` `:81` → `schedule_task`
`worker.rs:1327`) is polled by a worker's `run_task` (`worker.rs:630`). Its future awaits a TCP read
and a `Sleep`: the read future polls `ScheduledIo::readiness` (`scheduled_io.rs:382`) →
`Readiness::poll` (`:427`), and when not ready stores `cx.waker()` in `waiters` (`:495`) and inserts
into `waiters.list` (`:~502`), returning `Pending`; the `Sleep` future's `poll_elapsed`
(`time/sleep.rs:391`) builds and `init`s a `TimerEntry` (`time/sleep.rs:424`/`:427` → `entry.rs:481`
→ `reregister` into the wheel `:486`) and `TimerEntry::poll_elapsed` (`entry.rs:535` →
`state.poll(cx.waker())` `:545` → `waker.register_by_ref` `:146`) returns `Pending` — so the task has
registered its waker with **both** drivers. The owning worker, now idle, `park`s (`worker.rs:810` →
`park_internal` `:844` → `park.park(&handle.driver)` `:873`), which is `runtime::Driver::park`
(`driver.rs:66`) → `TimeDriver::park` (`:328`) → `time::Driver::park_internal` (`runtime/time/mod.rs:213`):
it computes `min(limit, time_until_next_timer)` (`:219-237`) and parks the **I/O driver** for that
duration (`io::Driver::park_timeout` `driver.rs:164` → `turn` `:179` → `mio poll` `:188`), then
`handle.process` (`:255`) fires any expired timers — one combined park drives both subsystems. If the
socket becomes ready, `io::Driver::turn` decodes the event and `io.wake(ready)` (`driver.rs:219` →
`ScheduledIo::wake` `scheduled_io.rs:236`) drains the registered waiter (`:256`) and
`wakers.wake_all()` (`:285`); if the timer expires first, `process_at_time` (`runtime/time/mod.rs:296`)
`wheel.poll(now)` (`:311`) → `entry.fire(Ok(()))` (`:315`) returns the stored waker and
`waker_list.wake_all()` (`:336`). Either way the task's `Waker` is woken, whose `wake` calls
`Schedule::schedule` (`handle.rs:110` → `schedule_task` `worker.rs:1327`), pushing the task back onto
a run queue and notifying a parked worker (`notify_parked_local/remote` `:1441`/`:1453`). The
unparked worker `next_task` (`:1062`) → `run_task` (`:630`) re-polls the future; the awaited branch is
now satisfied (I/O `Ready` or timer fired), the future progresses, and on completion the harness
writes `Stage::Finished(output)` (`runtime/task/core.rs:406`), `release` removes it from `owned`, and
the `JoinHandle` waker (`Trailer.waker`, `core.rs:205`) is notified. So one task threads the
scheduler, I/O driver, and time driver together through a single combined park and one
waker-driven reschedule.
