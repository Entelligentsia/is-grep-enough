# L5 reference answer key — how a dispatched event reaches sync listeners, queued listeners, and broadcasters

Pinned source: `experiment/repos/laravel` (SHA 2107d3d7…, PHP). All cites re-verified.

A cross-cutting flow threading **Events** (`src/Illuminate/Events/Dispatcher.php`,
`CallQueuedListener.php`), **Broadcasting** (`BroadcastFactory::queue`), the **Queue**
transport, and the **Bus**/queue-job execution path (`src/Illuminate/Queue/CallQueuedHandler.php`,
`src/Illuminate/Bus/Dispatcher.php`, `src/Illuminate/Queue/Jobs/Job.php`). Order of the
dispatch-side spine is gradeable; the worker-side terminal is a separate stage reached
asynchronously.

## Required spine (gradeable core — must hit all)

### A. Dispatch entry and the two sinks

1. **Fire the event — `Dispatcher::dispatch($event, $payload, $halt)`** —
   `src/Illuminate/Events/Dispatcher.php:280`. It parses the event/payload and calls
   `invokeListeners($parsedEvent, $parsedPayload, $halt)` (`:313`) (with deferred/after-commit
   short-circuits at `:293-309`).
2. **Broadcasting sink — `invokeListeners` broadcasts first** — `Dispatcher.php:320`. Before
   iterating listeners it checks `shouldBroadcast($payload)` (`:328`) and, if true, calls
   `broadcastEvent($payload[0])` (`:330`). `shouldBroadcast` (`:372`) tests
   `$payload[0] instanceof ShouldBroadcast` plus `broadcastWhen` (`:380`); `broadcastEvent`
   (`:398`) calls `$this->container->make(BroadcastFactory::class)->queue($event)` (`:403`),
   handing the broadcastable event to the broadcasting subsystem — a sibling sink to the
   listener path.

### B. Listener resolution and the sync-vs-queued decision

3. **Resolve listeners — `getListeners($eventName)`** — `Dispatcher.php:409`, iterated by
   `invokeListeners` (`:333`). Each class-based listener is wrapped by `makeListener`
   (`:484`) → `createClassListener` (`:510`) → `createClassCallable` (`:529`).
4. **The queued-listener detection — `handlerShouldBeQueued($class)`** — `Dispatcher.php:570`:
   `(new ReflectionClass($class))->implementsInterface(ShouldQueue::class)` (`:574-576`). This
   is the branch that decides a listener runs on a queue rather than inline.
5. **Build the serializable job — `createQueuedHandlerCallable` → `queueHandler` →
   `createListenerAndJob`** — `createQueuedHandlerCallable` `Dispatcher.php:588` returns a
   closure that, when `handlerWantsToBeQueued` is true, calls `queueHandler($class, $method,
   $arguments)` (`:660`). `queueHandler` resolves the queue connection/queue/delay from the
   listener (`viaConnection`/`viaQueue`/`withDelay`) and calls `$connection->pushOn($queue,
   $job)` / `$connection->laterOn($queue, $delay, $job)` (`:690-691`). The job is built by
   `createListenerAndJob` (`:706`) as `new CallQueuedListener($class, $method, $arguments)`
   (`:711`) — the serializable job handed to the **Queue** sink. (Sync listeners, the other
   branch of step 4, are instantiated via `$this->container->make($class)` in
   `createClassCallable` and returned as `[$listener, $method]`, run inline by
   `invokeListeners`.)

### C. Worker-side terminal — job → listener method

6. **Worker pops and fires the job — `Job::fire()`** —
   `src/Illuminate/Queue/Jobs/Job.php:96`. It parses `JobName::parse($payload['job'])`
   (`src/Illuminate/Queue/Jobs/JobName.php:15`) and calls
   `$this->resolve($class)->{$method}($this, $payload['data'])` (`:101`). For a queued
   listener the payload's `job` is `Illuminate\Queue\CallQueuedHandler@call` (set by
   `Queue::createObjectPayload`, `src/Illuminate/Queue/Queue.php:175`), so the resolved class
   is `CallQueuedHandler` and the method is `call`.
7. **Re-hydrate and dispatch through the Bus — `CallQueuedHandler::call`** —
   `src/Illuminate/Queue/CallQueuedHandler.php:67`. `getCommand($data)` (`:111`) unserializes
   the `CallQueuedListener` from `data.command`; `dispatchThroughMiddleware` (`:131`) runs the
   job's middleware pipeline and then calls
   `$this->dispatcher->dispatchNow($command, $this->resolveHandler($job, $command))`
   (`:153-155`) — re-entering the **Bus**.
8. **Run the listener method — `Bus\Dispatcher::dispatchNow` → `CallQueuedListener::handle`** —
   `src/Illuminate/Bus/Dispatcher.php:118` resolves a handler (`getCommandHandler`, `:232`)
   and calls its `handle`/`__invoke` (`:124-126`), else `container->call([$command, $method])`
   (`:135-137`). For `CallQueuedListener`, `CallQueuedListener::handle(Container $container)`
   (`src/Illuminate/Events/CallQueuedListener.php:133`) runs: it
   `$container->make($this->class)` to build the real listener (`:138-139`) and calls
   `$handler->{$this->method}(...array_values($this->data))` (`:140-141`) — the actual listener
   method, reached asynchronously on the worker.

## Common misses / wrong turns

- Stopping at `dispatch`/`invokeListeners` and never reaching the queued-listener handoff
  (`handlerShouldBeQueued` → `CallQueuedListener` → `pushOn`) — that is the cross-cutting
  crux the prompt names.
- Describing the queued listener as "the listener runs on a queue" without identifying
  `CallQueuedListener` as the serializable job and `CallQueuedHandler@call` as the
  worker-side entry — the indirection is the load-bearing part.
- Omitting broadcasting as a sibling sink (`shouldBroadcast`/`broadcastEvent` →
  `BroadcastFactory::queue`) — the prompt explicitly asks where broadcasting fits.
- Following the sync-listener branch only and treating it as the whole answer — that is just
  half the flow; the queued path is required.
- Confusing the Bus `Dispatcher` (`Illuminate\Bus\Dispatcher`, the command bus) with the
  Event `Dispatcher` (`Illuminate\Events\Dispatcher`) — they are distinct; the queued
  listener crosses from the latter into the former on the worker.
- Reversing direction (worker → dispatch) or omitting that the worker stage is asynchronous
  and reached only after the queue stores the job.

## Completeness scale

- **Full:** describes the dispatch-side chain `dispatch` → `invokeListeners` → broadcasting
  sink (`shouldBroadcast`/`broadcastEvent`) + listener resolution (`getListeners` →
  `makeListener` → `createClassListener` → `createClassCallable` → `handlerShouldBeQueued`
  → `createQueuedHandlerCallable` → `queueHandler` → `CallQueuedListener` → `pushOn`), AND
  the worker-side terminal `Job::fire` → `CallQueuedHandler::call` → Bus `dispatchNow` →
  `CallQueuedListener::handle` → listener method, with cites; notes broadcasting as a
  sibling sink.
- **Partial:** covers the dispatch-side handoff to the queue but not the worker-side
  re-hydration (or vice versa), OR misses the broadcasting sink, OR blurs the
  `CallQueuedListener`/`CallQueuedHandler` indirection.
- **Miss:** only describes sync listener invocation, or only "the event is dispatched to
  listeners," without the queue handoff and worker terminal.

## Model answer

Firing an event calls `Dispatcher::dispatch` (`Dispatcher.php:280`) → `invokeListeners`
(`:320`), which first checks `shouldBroadcast` (`:372`: `ShouldBroadcast` + `broadcastWhen`)
and, if true, calls `broadcastEvent` (`:398`) → `BroadcastFactory::class`'s `queue($event)`
(`:403`) — broadcasting is a sibling sink alongside the listener path. It then iterates
`getListeners($eventName)` (`:409`); each class-based listener is wrapped by `makeListener`
(`:484`) → `createClassListener` (`:510`) → `createClassCallable` (`:529`), which calls
`handlerShouldBeQueued($class)` (`:570`) — a `ReflectionClass` `ShouldQueue` interface check
(`:574`). For a queued listener it returns `createQueuedHandlerCallable` (`:588`), whose
closure calls `queueHandler($class, $method, $arguments)` (`:660`); `queueHandler` resolves
the connection/queue/delay and pushes via `$connection->pushOn(...)`/`laterOn(...)`
(`:690-691`) a job built by `createListenerAndJob` (`:706`) as `new CallQueuedListener($class,
$method, $arguments)` (`:711`) — the serializable job handed to the queue. (Sync listeners are
instead resolved inline via `container->make($class)` and run immediately by `invokeListeners`.)
Asynchronously, a worker pops that job and `Job::fire()` (`Job.php:96`) parses
`Illuminate\Queue\CallQueuedHandler@call` (set by `Queue::createObjectPayload`,
`Queue.php:175`) and calls `CallQueuedHandler::call($job, $data)` (`CallQueuedHandler.php:67`).
`call` unserializes the `CallQueuedListener` (`getCommand`, `:111`), runs it through its
middleware (`dispatchThroughMiddleware`, `:131`), and calls
`Bus\Dispatcher::dispatchNow($command, $handler)` (`:153-155`). `dispatchNow`
(`Bus\Dispatcher.php:118`) resolves and calls the handler — here `CallQueuedListener::handle`
(`CallQueuedListener.php:133`), which `$container->make($this->class)` builds the real
listener (`:138-139`) and invokes `$handler->{$this->method}(...$this->data)` (`:140-141`).
So an event threads Events → (Broadcasting and) Queue → Bus → the listener, with
`CallQueuedListener`/`CallQueuedHandler` as the serialization bridge across the queue
boundary.
