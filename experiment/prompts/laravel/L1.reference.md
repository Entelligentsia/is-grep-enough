# L1 reference answer key — the inbound HTTP request object (`Illuminate\Http\Request`)

Pinned source: `experiment/repos/laravel` (SHA 2107d3d7…, PHP). All cites re-verified.

The entity is `Illuminate\Http\Request` in `src/Illuminate/Http/Request.php`. It extends
Symfony's `Symfony\Component\HttpFoundation\Request`, so it inherits the raw input bags
(`query`, `request`, `cookies`, `files`, `server`, `headers`, `content`) and adds a small
set of Laravel-specific fields and a body-representation switch.

## Required spine (gradeable core — must hit all)

1. **The class is `Illuminate\Http\Request extends SymfonyRequest`** —
   `src/Illuminate/Http/Request.php:29` (`class Request extends SymfonyRequest implements
   Arrayable, ArrayAccess`). It composes the behavior traits `CanBePrecognitive`,
   `InteractsWithContentTypes`, `InteractsWithFlashData`, `InteractsWithInput`,
   `Conditionable`, `Macroable` — `Request.php:31-36`.
2. **Inherited raw input bags** come from the Symfony parent (`extends SymfonyRequest`,
   `:29`): `query`, `request` (form body), `cookies`, `files`, `server`, `headers`, plus
   `content`. A correct answer must note these are inherited, not redeclared here.
3. **`$json` — a decoded-JSON `InputBag`, distinct from the form/query bags** — declared
   `Request.php:43` (`protected $json;`, typed `InputBag|null`, docblock `:38-42`). It is
   built lazily by `json()` (`:462`) from `$this->getContent()`, so the *same* request body
   has two concrete representations selected by content type.
4. **`getInputSource()` is the selector that distinguishes JSON vs form vs query** —
   `Request.php:480`. For a JSON request it returns `$this->json()` (the decoded-JSON bag,
   `:483`); otherwise `$this->query` for `GET`/`HEAD` or `$this->request` (the form bag) for
   other methods (`:486`). This is the load-bearing "logical body → concrete representation"
   distinction the prompt names.
5. **`$userResolver` — a lazily-set `Closure` resolving the authenticated user** —
   `Request.php:57` (`protected $userResolver;`, docblock `:52-56`). Decouples the request
   from the auth subsystem until needed.
6. **`$routeResolver` — a lazily-set `Closure` resolving the matched route** —
   `Request.php:64` (`protected $routeResolver;`, docblock `:59-63`). Decouples the request
   from the routing subsystem until needed.
7. **`$convertedFiles` and `$cachedAcceptHeader`** — `$convertedFiles` `Request.php:50`
   (normalized `UploadedFile` tree, docblock `:45-49`); `$cachedAcceptHeader` `:71`
   (memoized `Accept` header for content negotiation, docblock `:66-70`).

## Common misses / wrong turns

- Treating the Symfony bags as if they were declared on `Illuminate\Http\Request` — they are
  *inherited*; the prompt explicitly asks what the object holds *beyond* them.
- Pointing at `request()` / `Request::capture()` / the HTTP kernel (`Foundation/Http/Kernel`)
  instead of the request object's own shape — those consume the object, they are not it.
- Listing methods (`input`, `json`, `all`, `old`) without naming the *fields* (`$json`,
  `$userResolver`, `$routeResolver`, `$convertedFiles`, `$cachedAcceptHeader`) the prompt
  asks about.
- Conflating `$json` with `$request` (the form bag) — the prompt is keyed on the distinction
  between the decoded-JSON representation and the form/query representation, mediated by
  `getInputSource()` (`:480`).

## Model answer

A single incoming request is `Illuminate\Http\Request` (`src/Illuminate/Http/Request.php:29`),
which extends Symfony's `SymfonyRequest` and so inherits the raw input bags — `query`,
`request` (form body), `cookies`, `files`, `server`, `headers`, and `content`. On top of
those it declares a handful of Laravel-specific fields. `$json` (`:43`) is a lazily-built
`InputBag` holding the decoded JSON body, produced by `json()` (`:462`) from `getContent()`;
`getInputSource()` (`:480`) is the selector that returns the JSON bag for JSON requests and
otherwise `$this->query` (GET/HEAD) or `$this->request` (other methods) — so the request body
has two concrete representations chosen by content type, the central distinction the object
encodes. `$userResolver` (`:57`) and `$routeResolver` (`:64`) are `Closure`s that defer
resolution of the authenticated user and the matched route to the auth and routing
subsystems, respectively, so the request object stays decoupled from them until asked.
`$convertedFiles` (`:50`) holds the normalized `UploadedFile` tree, and `$cachedAcceptHeader`
(`:71`) memoizes the `Accept` header for content negotiation. The class also composes the
`InteractsWithInput`, `InteractsWithFlashData`, `InteractsWithContentTypes`, `Conditionable`,
`Macroable`, and `CanBePrecognitive` traits (`:31-36`), which is where most of the input/flash
API actually lives.
