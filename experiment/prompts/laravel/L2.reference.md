# L2 reference answer key — the Gate ability check and its callers

Pinned source: `experiment/repos/laravel` (SHA 2107d3d7…, PHP). All cites re-verified.

The focal routine is `Gate::inspect($ability, $arguments)` in
`src/Illuminate/Auth/Access/Gate.php`, which evaluates an ability and returns a
`Illuminate\Auth\Access\Response`. Its callers each consume that `Response` differently.

## Required spine (gradeable core — must hit all)

1. **Focal routine `Gate::inspect($ability, $arguments)` returns a `Response`** —
   `src/Illuminate/Auth/Access/Gate.php:408`. Inside it calls `raw($ability, $arguments)`
   (`:413`) and normalizes the result: a `Response` is returned as-is, otherwise `true` →
   `Response::allow()` and falsy → `Response::deny()` (the default denial) (`:415-422`);
   an `AuthorizationException` is caught and converted via `toResponse()` (`:423-425`).
2. **`raw($ability, $arguments)` is the underlying callback evaluation** —
   `src/Illuminate/Auth/Access/Gate.php:434`. It runs the `before` callbacks
   (`callBeforeCallbacks`, `:454`), and only if they return null does it call the real
   ability callback (`callAuthCallback`, `:460`), then the `after` callbacks
   (`callAfterCallbacks`, `:464`). This is the verdict producer; `inspect` wraps it.
3. **`Response` is the verdict object** — `src/Illuminate/Auth/Access/Response.php:8`
   (`class Response`), with `Response::allow()` (`:59`), `Response::deny()` (`:71`),
   `allowed()` (`:106`), `denied()` (`:116`), and `authorize()` (`:148`).
4. **`check($abilities, $arguments)` → boolean via `inspect(...)->allowed()`** —
   `Gate.php:356`. It runs `inspect()` for every ability and reduces with `->every(...)` /
   `allowed()` (`:360-361`). This is the plain yes/no caller.
5. **`authorize($ability, $arguments)` → throws on denial** — `Gate.php:396`. It calls
   `inspect(...)->authorize()` (`:400`), where `Response::authorize()` (`Response.php:148`)
   throws an `AuthorizationException` (`src/Illuminate/Auth/Access/AuthorizationException.php:8`)
   when the verdict is denied. This is the "abort with an authorization error" caller.
6. **The boolean wrappers around `check`** — `allows($ability, $arguments)` `Gate.php:332`
   (returns `$this->check(...)`, `:336`); `denies($ability, $arguments)` `:344`
   (`! $this->allows(...)`, `:348`); `any($abilities, $arguments)` `:370`
   (`contains(check)`, `:374`); `none($abilities, $arguments)` `:380` (`! $this->any(...)`,
   `:384`). These are the branch-on-the-answer callers that surface the verdict as a
   predicate to calling code.

## Common misses / wrong turns

- Stopping at `allows`/`denies` and never reaching the focal `inspect`/`raw` — the prompt asks
  for "the routine that actually evaluates an ability," which is `inspect`/`raw`, not the
  boolean wrappers.
- Treating `allows`/`denies`/`any`/`none`/`check`/`authorize` as six independent checks rather
  than callers of one `inspect` routine that branch on its `Response`.
- Naming `Response::allow()`/`deny()` as the *check* — they are verdict *constructors* used by
  `inspect`; the check is `inspect`/`raw`.
- Confusing the policy/callback registration (`define`/`policy`) with the evaluation path —
  registration is upstream of `raw`, not part of the on-request check the prompt describes.
- Pointing at `Illuminate\Foundation\Auth\Access\AuthorizesRequests` (the trait that adds the
  `authorize()` helper on controllers) instead of the `Gate` itself — that is a *caller* of
  `Gate::authorize`, not the central routine.

## Model answer

The central evaluation is `Gate::inspect($ability, $arguments)` (`Gate.php:408`), which calls
`raw($ability, $arguments)` (`:413`/def `:434`) and normalizes whatever the ability callback
returns into a `Illuminate\Auth\Access\Response` (`Response.php:8`) — `Response::allow()`
(`:59`) for a truthy result, `Response::deny()` (`:71`) for a falsy one, a `Response` returned
as-is, and a thrown `AuthorizationException` (`AuthorizationException.php:8`) converted via
`toResponse()` (`:423`). `raw` itself runs the registered `before` callbacks, then the matching
ability callback, then the `after` callbacks (`:454`/`:460`/`:464`). Around that one routine
sit several callers that branch on the verdict: `check($abilities, $arguments)` (`:356`)
returns a plain bool by running `inspect(...)->allowed()` for every ability; `allows` (`:332`)
wraps `check`, `denies` (`:344`) negates `allows`, and `any` (`:370`) / `none` (`:380`) give
any/all variants; `authorize($ability, $arguments)` (`:396`) instead calls
`inspect(...)->authorize()` (`:400`), and `Response::authorize()` (`Response.php:148`) throws
an `AuthorizationException` when the verdict is denied — that is the "abort with an
authorization error" path. During a request, controller code (often via the
`AuthorizesRequests` trait) or form-request authorization reaches `Gate::authorize`/`allows`,
which in turn funnel through `inspect`/`raw`.
