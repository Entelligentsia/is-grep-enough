# L5 reference answer key — the full program pipeline (source → bind → check → emit)

Pinned source: `experiment/repos/typescript` (SHA 8ef3e2f3…). All cites re-verified.

A cross-cutting concern threading four subsystems around producing output + diagnostics for input
source: parse (parser + scanner), bind (binder), check (checker), emit (emitter + transformer). Files:
`src/compiler/program.ts` (the `Program` hub), `src/compiler/parser.ts` (+ `scanner.ts`),
`src/compiler/binder.ts`, `src/compiler/checker.ts`, `src/compiler/emitter.ts` (+ `transformer.ts`).
The `Program` holds the source files; the `TypeChecker` is the hub for bind+check. Entry ambiguity:
binding is triggered by the checker's `initializeTypeChecker`, not by `createProgram` directly, and
checking is lazy (runs only when diagnostics/emit are requested).

## Required spine (gradeable core — must hit all, in order of the pipeline)

1. **Parse — text → syntax tree** — `createProgram` `program.ts:1515` processes source files via
   `processSourceFile` (`:3486`) → `findSourceFile` (`:3522`) → `createSourceFile`
   (`parser.ts:1344`, invoked at `program.ts:410`), which runs the parse pipeline (L3): scanner
   `setText`/`scan` + parser → a `SourceFile` AST.
2. **Bind — attach symbols** — the checker's `createTypeChecker` `checker.ts:1486` calls
   `initializeTypeChecker()` (`:2410`), which binds every file:
   `for (const file of host.getSourceFiles()) { bindSourceFile(file, compilerOptions); }`
   (`checker.ts:51557-51558` → `binder.ts:502`). Binding walks each tree, sets `parent`, creates
   `Symbol`s and inserts them into `locals`/`members`/`exports` tables, and builds the control-flow
   graph (L4). `bindDiagnostics` are attached to each file.
3. **Check — lazily type-check and produce diagnostics** — `getSemanticDiagnostics`
   `program.ts:2798` → `getSemanticDiagnosticsForFile` (`:2858`) → `getBindAndCheckDiagnosticsForFile`
   (`:2869`) → `getBindAndCheckDiagnosticsForFileNoCache` (`:2887`) → `typeChecker.getDiagnostics(...)`
   (`:2906`) → `checkSourceFile` (`checker.ts:49547`) → `checkSourceFileWorker`. Checking is deferred
   until a diagnostics/emit request and is cached per file (`cachedBindAndCheckDiagnosticsForFile`,
   `:2878`); it merges `bindDiagnostics` with the freshly computed check diagnostics
   (`getMergedBindAndCheckDiagnostics`, `:2907+`).
4. **Emit — transform + write JS/declaration files** — `emit` `program.ts:2688` → `emitWorker`
   (`:2699`) → `emitFiles(...)` (`emitter.ts:752`, called at `program.ts:2753`) with
   `getTransformers(options, customTransformers, emitOnly)` (`:2757`). `emitFiles` runs the
   transforms via `transformNodes` (`transformer.ts:248`) — the syntax transform (ES-next → target),
   the module/declaration transforms — and the emitter writes out JavaScript and/or `.d.ts`
   (`emitJavaScript`/`emitDeclaration` paths in `emitter.ts`) plus build info and sourcemaps.

## Acceptable extras (credit, never required)

- `emitResolverSkipsTypeChecking` / `emitOnly` (`program.ts:2745`) — emit can skip full type checking
  for emit-only modes; the `forceDtsEmit` parameter controls declaration emit.
- `getTypeChecker()` laziness (`program.ts:2899`) — the checker itself is created on first request and
  `initializeTypeChecker` runs once, which is why binding appears to be driven by the checker.
- `getProgramDiagnostics` (`program.ts` within `getSemanticDiagnosticsForFile` `:2864`) — the
  program-level diagnostics concatenated with the per-file bind+check diagnostics.
- `getTransformers` selecting the transformer chain (typescript-transformers, module transformers,
  declaration transformer) — the `transformers/` directory; `transformNodes` (`transformer.ts:248`)
  is the engine.
- The `SourceFile`/`Program` as the hand-off object between stages: parse produces `SourceFile`s the
  `Program` holds; bind mutates them in place (symbols/parent/flow); check reads them via the
  `Checker`; emit reads them via `getSourceFilesToEmit`.
- `getSyntacticDiagnostics` (separate from semantic) — the parser diagnostics (`parseDiagnostics`)
  vs the binder's `bindDiagnostics` vs the checker's diagnostics; the prompt's "reported errors" span
  all three but the semantic-check step is the load-bearing one.

## Common misses / wrong turns

- Stopping at "parse then check" and missing the bind stage, or missing emit — the prompt asks for the
  whole flow to emitted JS + diagnostics across all four stages.
- Placing binding inside `createProgram` — it is actually triggered by `createTypeChecker`'s
  `initializeTypeChecker` (`checker.ts:2410`/`:51557`); the indirection is a real, gradeable point.
- Treating checking as eager — it is lazy, fired by `getSemanticDiagnostics`/`emit` and cached per
  file; an answer that says "createProgram checks everything" is wrong.
- Omitting the transform step in emit (claiming the emitter just prints the AST) — `emitFiles` runs
  `transformNodes` first (`transformer.ts:248`); the emitter prints the *transformed* tree.
- Confusing the order: e.g. emit-before-check, or check-before-bind. Bind must precede check; check
  precedes (or gates) emit.

## Model answer

`createProgram` (`program.ts:1515`) starts the pipeline by reading and parsing files:
`processSourceFile` (`:3486`) → `findSourceFile` (`:3522`) → `createSourceFile` (`parser.ts:1344`,
called at `program.ts:410`) turns each file's text into a `SourceFile` AST (scanner `setText`/`scan`
+ parser, as in L3). Binding is driven by the checker, not `createProgram` directly: when the
`TypeChecker` is created, `createTypeChecker` (`checker.ts:1486`) runs `initializeTypeChecker()`
(`:2410`), which binds every file — `for (const file of host.getSourceFiles()) bindSourceFile(file,
compilerOptions);` (`checker.ts:51557-51558` → `binder.ts:502`) — setting `parent`, creating `Symbol`s
in the `locals`/`members`/`exports` tables, and building the flow graph (L4), attaching
`bindDiagnostics`. Checking is lazy: `getSemanticDiagnostics` (`program.ts:2798`) →
`getSemanticDiagnosticsForFile` (`:2858`) → `getBindAndCheckDiagnosticsForFile` (`:2869`) →
`getBindAndCheckDiagnosticsForFileNoCache` (`:2887`) → `typeChecker.getDiagnostics(...)` (`:2906`) →
`checkSourceFile` (`checker.ts:49547`); the result is cached per file (`:2878`) and merges bind +
check diagnostics (`getMergedBindAndCheckDiagnostics`). Finally `emit` (`program.ts:2688`) →
`emitWorker` (`:2699`) → `emitFiles(...)` (`emitter.ts:752`, at `:2753`) with `getTransformers(...)`
(`:2757`); `emitFiles` first transforms the tree via `transformNodes` (`transformer.ts:248`) —
ES-next→target, module, and declaration transforms — and then the emitter writes out the JavaScript
and/or `.d.ts` files plus build info/sourcemaps. So source text flows parse → bind (checker-driven) →
check (lazy, cached) → emit (transform-then-write), with the `Program` holding the files and the
`Checker` being the hub for the middle two stages.
