# L2 reference answer key — on-access re-tokenization (scan + re-scan call sites)

Pinned source: `experiment/repos/typescript` (SHA 8ef3e2f3…). All cites re-verified.

The focal routine is the scanner's `scan()`, which produces one `SyntaxKind` token at a time. The
gradeable point is that several character sequences are *lexically ambiguous* (their meaning depends
on grammar context), so the parser keeps a family of re-scan hooks and, at specific grammar
positions, asks the scanner to re-tokenize the current character(s) and then branches on the
returned `SyntaxKind`. Files: `src/compiler/scanner.ts` (the focal + re-scan methods) and
`src/compiler/parser.ts` (the wrappers + grammar call sites). One hop (parser → scanner).

## Required spine (gradeable core — must hit all)

1. **The focal `scan()`** — `scanner.ts:1891` (`function scan(): SyntaxKind`). It advances `pos`,
   classifies the next character(s), and stores/returns the `SyntaxKind` in `token`. The parser
   drives it through `nextTokenWithoutCheck` (`parser.ts:2198`), which does
   `currentToken = scanner.scan()` (`:2200`); `nextToken` (`parser.ts:2207`) wraps it with a
   keyword-Unicode-escape check. So ordinary advancement is `nextToken` → `scan`.
2. **The re-scan family on the scanner — each re-tokenizes one ambiguous character sequence:** all
   return a `SyntaxKind` and update the scanner's `token`.
   - `reScanGreaterToken` `scanner.ts:2438` — the `>` family (`>`, `>>`, `>>>`, `>=`, `>>=`, `>>>=`).
   - `reScanAsteriskEqualsToken` `scanner.ts:2461` — `*` vs `**` vs `*=`.
   - `reScanSlashToken` `scanner.ts:2467` — `/` as divide vs the start of a `RegularExpressionLiteral`.
   - `reScanTemplateToken` `scanner.ts:3658` — template-literal continuation (the closing backtick /
     `${` substitution).
   - `reScanLessThanToken` `scanner.ts:3673` — `<` as less-than vs the start of type arguments / JSX.
   - `reScanQuestionToken` `scanner.ts:3689` — `?` vs `?.` (optional chaining) vs `??` (nullish).
3. **The parser wrappers** that expose them and reassign `currentToken` — `reScanGreaterToken`
   `parser.ts:2224`, `reScanSlashToken` `:2228`, `reScanTemplateToken` `:2232`, `reScanLessThanToken`
   `:2236` (each `return currentToken = scanner.reScan…()`).
4. **Grammar call sites that branch on the re-scanned kind (the gradeable "what they do with it"):**
   - **Arrow `=>`**: `parser.ts:5128` —
     `if (isLeftHandSideExpression(expr) && isAssignmentOperator(reScanGreaterToken()))` — after
     parsing a parenthesized/type-annotated argument list, re-scan `>` to see whether it is actually
     `=>` and the expression is an arrow function.
   - **Regex vs divide**: `parser.ts:6650` —
     `if (reScanSlashToken() === SyntaxKind.RegularExpressionLiteral)` — in expression context, re-scan
     `/` to decide regex literal versus division operator.
   - **JSX `<`**: `parser.ts:3792` —
     `if (!scanner.hasPrecedingLineBreak() && reScanLessThanToken() === SyntaxKind.LessThanToken)` —
     re-scan `<` to decide a JSX element vs a type-argument list / less-than.
   - **Type-arguments `>`**: `parser.ts:6574` —
     `if (reScanGreaterToken() !== SyntaxKind.GreaterThanToken)` (and the matching `<` at `:6568`,
     `reScanLessThanToken() !== SyntaxKind.LessThanToken`) — re-scan to recognize `>>` as two closing
     angle brackets of nested type arguments rather than a shift token.
5. **Synthesis** — the answer must state that the *same characters* are scanned differently
   depending on grammar position: `>` is `GreaterThanToken` by default but becomes `EqualsGreaterThanToken`
   (`=>`) after an arg list, `>>` after type args; `/` is `SlashToken` by default but a regex literal
   in expression-primary position; `<` is `LessThanToken` by default but opens type args/JSX in
   expression position; `?` becomes `?.`/`??`. The parser, not the scanner, decides *when* to re-scan;
   the scanner just re-tokenizes on demand.

## Common misses / wrong turns

- Treating `scan()` as a one-shot tokenizer that produces all tokens up front — it is incremental and
  re-entrant; re-scan re-tokenizes the *current* position on demand. The prompt is about the
  re-scan-on-access pattern.
- Listing the re-scan methods without the parser branch sites (or vice versa) — the gradeable insight
  is the *pairing*: scanner re-tokenizes, parser decides when and branches on the result.
- Naming the wrong trigger: e.g. claiming the scanner itself decides arrow-vs-greater. The *parser*
  decides at a grammar position; the scanner only re-tokenizes when asked.
- Missing the "same characters, different token by context" synthesis — that is the central point.
- Confusing this with the *binder's* disambiguation or the *checker's* overload resolution — those are
  later phases; this is purely lexical/parser-level.

## Model answer

The scanner produces tokens one at a time via `scan()` (`scanner.ts:1891`), which classifies the next
character(s) into a `SyntaxKind` and stores it in `token`; the parser advances with `nextToken`
(`parser.ts:2207`) → `nextTokenWithoutCheck` (`:2198`) → `scanner.scan()` (`:2200`). Several character
sequences are lexically ambiguous, so the scanner exposes a re-scan family that re-tokenizes the
current position on demand: `reScanGreaterToken` (`scanner.ts:2438`, the `>` family),
`reScanSlashToken` (`:2467`, `/` vs regex), `reScanLessThanToken` (`:3673`, `<` vs type-args/JSX),
`reScanQuestionToken` (`:3689`, `?` vs `?.`/`??`), `reScanAsteriskEqualsToken` (`:2461`, `*` vs `**`),
`reScanTemplateToken` (`:3658`, template continuation), wrapped by `reScanGreaterToken`/`reScanSlashToken`/
`reScanLessThanToken`/`reScanTemplateToken` in the parser (`parser.ts:2224`/`:2228`/`:2236`/`:2232`). The
parser decides *when* to re-scan and branches on the result: after an argument list it re-scans `>` and
checks `isAssignmentOperator(...)` to detect `=>` arrow functions (`parser.ts:5128`); in expression
position it re-scans `/` and checks for `RegularExpressionLiteral` (`:6650`); it re-scans `<` and checks
for `LessThanToken` to detect JSX (`:3792`); and it re-scans `>`/`<` checking against `GreaterThanToken`/
`LessThanToken` to peel apart nested type-argument brackets (`:6574`/`:6568`). So the same characters
yield different tokens depending on grammar position — the scanner re-tokenizes on access, the parser
branches on what comes back.
