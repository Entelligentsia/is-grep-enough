# L1 reference answer key — the AST node container (Node)

Pinned source: `experiment/repos/typescript` (SHA 8ef3e2f3…). All cites re-verified.

The entity is `Node` in `src/compiler/types.ts`. It is the base interface every AST node extends — the
in-memory container for one piece of parsed syntax, independent of which specific construct it is.
Analogous to how a single Redis value lives in a `robj` independent of the key.

## Required spine (gradeable core — must hit all)

1. **The container is `Node`** — `types.ts:942` (`export interface Node extends ReadonlyTextRange {`).
   It extends `ReadonlyTextRange` (`types.ts:33`), so every node inherits `readonly pos: number` and
   `readonly end: number` (`:34-35`) — its byte span in the source text.
2. **`kind: SyntaxKind` — what syntactic construct it is** — `types.ts:943` (`readonly kind: SyntaxKind;`).
   `SyntaxKind` is the big const enum at `types.ts:40` (NumericLiteral, StringLiteral, Identifier,
   FunctionDeclaration, ClassDeclaration, …). Set by the parser when the node is created; it is the
   node's logical identity.
3. **`flags: NodeFlags` — a separate bit-flag set categorizing the node, distinct from `kind`** —
   `types.ts:944` (`readonly flags: NodeFlags;`). `NodeFlags` is the const enum at `types.ts:782`. A
   correct answer must state that `kind` is the fixed syntactic type while `flags` is a bitset: some
   bits are set at parse time (e.g. `Let`/`Const`/`Using`, `YieldContext`, `DisallowInContext`,
   `DecoratorContext` — `:783-795`), others are *initialized by the binder* (e.g. `ExportContext`,
   `HasImplicitReturn`, `HasExplicitReturn`, `NestedNamespace`, `HasAsyncFunctions` — comments at
   `:790`, `:799-803`). So `kind` is the "what" and `flags` is contextual/categorical metadata.
4. **`parent: Node` — tree linkage, set by the binder** — `types.ts:948` (`readonly parent: Node; //
   Parent node (initialized by binding)`). The parser does not always set it; `fixupParentReferences`
   or binding populates it.
5. **The attached/lazy slots the binder and transformer write into** — `id?: NodeId` (`:947`, unique id
   used to look up per-node `NodeLinks` in the checker), `modifierFlagsCache: ModifierFlags` (`:945`),
   `readonly transformFlags: TransformFlags` (`:946`, flags for transforms), `original?: Node`
   (`:949`, the original node if this is an updated/synthesized one), `emitNode?: EmitNode` (`:950`,
   associated emit data initialized by transforms). These are the "what the binder/transform attach
   to it as they process it" the prompt names.

## Acceptable extras (credit, never required)

- The brand interfaces that extend `Node` and carry the binder-attached data instead of `Node` itself:
  `JSDocContainer` (`:953`, `jsDoc?`), `LocalsContainer` (`:971`, `locals?` SymbolTable +
  `nextContainer?`), `FlowContainer` (`:981`, `flowNode?` FlowNode). The comment at `:951-957` notes
  `symbol`/`locals`/`flowNode` were moved off `Node` into these brands. Mentioning this is a plus;
  the spine is the `Node` base itself.
- `modifierFlagsCache` being a mutable cache (the only non-readonly field besides the attached slots)
  vs the `readonly` core fields — the parse-time fields are immutable, the binder/transform fields are
  mutable.
- The `HasFlowNode` type alias (`:985`) listing which node kinds carry flow nodes — ties `FlowContainer`
  to control-flow analysis.
- The observation that the header is a few words (kind + flags + pos/end + parent + a few optional
  attached slots) tying to the per-node footprint framing.

## Common misses / wrong turns

- Conflating `kind` and `flags` (treating them as the same thing, or calling `flags` "the type") — the
  prompt explicitly asks for the distinction; this is the central gradeable point. `kind` is the
  SyntaxKind identity (parse-time, fixed); `flags` is a bitset, partly parse-time and partly
  binder-initialized.
- Describing a specific node subtype (`FunctionDeclaration`, `Identifier`) instead of the universal
  `Node` base — those extend `Node`; the container asked for is `Node`.
- Omitting `pos`/`end` (the source span, from `ReadonlyTextRange`) or `parent` (the tree linkage the
  prompt names).
- Pointing at the `Symbol` or `Type` interfaces instead of `Node` — those are the binder/checker
  products, not the AST node container.
- Omitting the attached slots (`id`/`transformFlags`/`original`/`emitNode`) the binder/transformer
  attach — the prompt names "extra slots the binder and transformer attach."

- **Full:** identifies `Node` (`types.ts:942`) extending `ReadonlyTextRange`, and explains all 5 spine

## Model answer

Every AST node is a `Node` (`src/compiler/types.ts:942`), which extends `ReadonlyTextRange`
(`:33`) so it carries `readonly pos`/`readonly end` (`:34-35`) — its byte span in the source. Its
first field `readonly kind: SyntaxKind` (`:943`) records what syntactic construct it is
(`SyntaxKind` enum at `:40` — FunctionDeclaration, Identifier, …), set by the parser and fixed.
Separately, `readonly flags: NodeFlags` (`:944`, enum at `:782`) is a bitset: some bits are set at
parse time (`Let`/`Const`/`Using`, `YieldContext`, `DisallowInContext` — `:783-795`) while others are
*initialized by the binder* (`ExportContext`, `HasImplicitReturn`, `NestedNamespace` — `:790`,
`:799-803`). So `kind` is the logical identity and `flags` is contextual/categorical metadata — the
"what" versus a bitset the binder augments. `readonly parent: Node` (`:948`) links it into the tree
(initialized by binding). On top of the immutable parse fields sit the mutable attached slots:
`id?: NodeId` (`:947`, the key for the checker's per-node `NodeLinks`), `modifierFlagsCache`
(`:945`), `readonly transformFlags: TransformFlags` (`:946`), `original?: Node` (`:949`, the
original if this node is an updated one), and `emitNode?: EmitNode` (`:950`, emit data the transforms
attach). (Binder-attached data like `locals`/`flowNode` actually live on the `LocalsContainer`/
`FlowContainer` brands at `:971`/`:981` rather than on `Node` directly.)
