# L3 reference answer key — the parse pipeline (source text → parented SourceFile AST)

Pinned source: `experiment/repos/typescript` (SHA 8ef3e2f3…). All cites re-verified.

A single ordered chain across `src/compiler/parser.ts` and `src/compiler/scanner.ts`. Order is
gradeable. (The TS/JS path is the spine; the JSON branch is an acceptable extra.)

## Required spine (gradeable core — must hit all, in order)

1. **Entry** — `createSourceFile` `parser.ts:1344` (`export function createSourceFile(...)`). It
   resolves `languageVersion`/`scriptKind` and delegates to `Parser.parseSourceFile`
   (`parser.ts:1362`).
2. **Initialize the scanner with the source text** — `parseSourceFile` `parser.ts:1603` calls
   `initializeState(...)` (`:1633`), which calls `scanner.setText(sourceText)` (`parser.ts:1772` →
   `scanner.ts:3993`) and `scanner.setScriptTarget`/`setLanguageVariant`/`setJSDocParsingMode`
   (`parser.ts:1774-1777`). After parsing, `clearState()` (`:1636`) calls `scanner.setText("")`
   (`:1783`).
3. **Prime the scanner and drive the parse** — `parseSourceFileWorker` `parser.ts:1803` primes the
   first token with `nextToken()` (`:1812`) → `scanner.scan()` (`scanner.ts:1891`), then parses the
   top level with `parseList(ParsingContext.SourceElements, parseStatement)` (`:1814`); it asserts the
   terminal `EndOfFileToken` (`:1815`) and builds the `EndOfFileToken` node (`:1817`).
4. **Consume tokens and build AST nodes** — `parseList` `parser.ts:3094` repeatedly calls the element
   parser (`parseStatement` `parser.ts:7380`) until the closing token; each statement/expression
   parser advances via `nextToken` (`parser.ts:2207`) → `scanner.scan()` (`scanner.ts:1891`) and
   constructs nodes with the factory (`factory.createX` / `createNode`), threading the `pos`/`end` of
   each consumed token into the new `Node`.
5. **Link the tree** — when `setParentNodes` is set, `parseSourceFileWorker` calls
   `fixupParentReferences(sourceFile)` (`parser.ts:1836` → def `:1970`), which walks the tree and sets
   each node's `parent` (the `Node.parent` field from L1) so the AST is parented.
6. **Return the root** — `parseSourceFileWorker` constructs and returns the `SourceFile` node
   (`parser.ts:1819`, `:1839`) carrying `statements`, `endOfFileToken`, `parseDiagnostics`,
   `nodeCount`, `identifiers`, etc.

## Acceptable extras (credit, never required)

- The JSON branch in `parseSourceFile` (`parser.ts:1614`) and `parseJsonText` (`:1647`) — a parallel
  parse path; the TS/JS path above is the required spine.
- `parseIsolatedEntityName` (`parser.ts:1657`) — the "prime the scanner, parse a fragment" cousin that
  shows the same `nextToken()`-then-parse pattern.
- `initializeState`/`clearState` themselves (`parser.ts:1735`/`:1780`) — the per-file parser state
  reset (identifiers map, parseDiagnostics, context flags).
- The `processCommentPragmas`/`processPragmasIntoFields` steps (`parser.ts:1828-1829`) that attach
  `referencedFiles`/`typeReferenceDirectives`/pragmas to the `SourceFile`.
- `tracing`/`performance` marks around parse (`parser.ts:1345`/`:1378`) — instrumentation, never
  required.

## Common misses / wrong turns

- Stopping at `createSourceFile` and never reaching the scanner prime / `parseList`/`parseStatement` /
  `fixupParentReferences` — skips the middle and back of the chain; the prompt asks end-to-end.
- Omitting the scanner hand-off (`scanner.setText` + `nextToken`→`scan`) — the prompt explicitly asks
  how the text is given to the scanner and how the first token is primed.
- Missing `fixupParentReferences` — the prompt asks how finished nodes are linked into a parented
  tree; that is the dedicated step (`parser.ts:1970`).
- Confusing this with binding (which also sets `parent`) — binding runs later (`bindSourceFile`); the
  *parser* sets parents via `fixupParentReferences` when `setParentNodes` is requested.
- Treating it as an unordered set of functions rather than a directed sequence from text to tree.

## Completeness scale

- **Full:** all 6 spine stops in order — `createSourceFile` → `parseSourceFile`/`initializeState`+
  `scanner.setText` → `parseSourceFileWorker`/`nextToken`+`parseList` → `parseStatement`/`scan` build
  nodes → `fixupParentReferences` → return `SourceFile` — with cites.
- **Partial:** ≥4 of 6 in roughly correct order, but missing either the scanner hand-off OR the
  `fixupParentReferences` step, OR with the token-consumption step left vague.
- **Miss:** only names `createSourceFile`, or asserts "the parser builds an AST" without the scanner
  prime / parse-loop / parent-linking chain.

## Model answer

`createSourceFile` (`parser.ts:1344`) is the entry: it resolves script target/kind and calls
`Parser.parseSourceFile` (`:1362`). `parseSourceFile` (`:1603`) calls `initializeState` (`:1633`),
which hands the text to the scanner via `scanner.setText(sourceText)` (`parser.ts:1772` →
`scanner.ts:3993`) and configures it (`setScriptTarget`/`setLanguageVariant`, `:1774-1777`), then calls
`parseSourceFileWorker` (`:1634`) and finally `clearState` (`:1636`, `scanner.setText("")` `:1783`).
`parseSourceFileWorker` (`:1803`) primes the scanner with `nextToken()` (`:1812`) — i.e.
`scanner.scan()` (`scanner.ts:1891`) — then drives the top level with
`parseList(ParsingContext.SourceElements, parseStatement)` (`:1814`); `parseList` (`:3094`) repeatedly
calls `parseStatement` (`:7380`), each of which advances via `nextToken` (`:2207`) → `scanner.scan()`
and constructs AST nodes with the factory, threading each token's `pos`/`end` into the new `Node`.
When the token stream reaches `EndOfFileToken` (`:1815`) the closing node is built (`:1817`). If
`setParentNodes` was requested, `fixupParentReferences(sourceFile)` (`:1836` → `:1970`) walks the tree
setting each node's `parent`. The worker then assembles and returns the `SourceFile` root node
(`:1819`/`:1839`) carrying `statements`, `endOfFileToken`, `parseDiagnostics`, and identifier/count
metadata — a complete, parented syntax tree. (For JSON input `parseSourceFile` instead takes the
`parseJsonText` branch at `:1614`.)
