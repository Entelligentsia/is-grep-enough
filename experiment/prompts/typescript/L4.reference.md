# L4 reference answer key — the binder subsystem

Pinned source: `experiment/repos/typescript` (SHA 8ef3e2f3…). All cites re-verified.

A bounded cooperating cluster in `src/compiler/binder.ts`. The binder walks a parsed `SourceFile` and
attaches a `Symbol` to each declaration, inserting it into the right symbol table (`locals`/`members`/
`exports`), and sets each node's `parent` and flow node — producing the tables the checker later
resolves names against. Cooperating components: the per-file entry/state setup, the focal `bind` walk,
the `bindWorker` kind-dispatch, the `declareSymbol` symbol-table insert (merge/conflict), and the
container-scope stack.

## Required spine (gradeable core — must hit all)

1. **Per-file entry + state** — `bindSourceFile` `binder.ts:502` (public) calls `binder(file, options)`
   (`:504`), the closure created by `createBinder()` (`binder.ts:509`). The inner `bindSourceFile`
   (`:571`) sets up binder state (`inStrictMode`, `classifiableNames`, `Symbol` constructor, etc.) and,
   if `file.locals` is not yet set, calls `bind(file)` (`:586`), then clears state.
2. **The focal `bind` walk** — `bind(node)` `binder.ts:2751`: it `setParent(node, parent)` (`:2755`)
   first, then `bindWorker(node)` (`:2779`) to declare the node's symbol, then — for non-token nodes —
   saves `parent`, sets `parent = node`, computes `getContainerFlags(node)` (`:2788`) and recurses via
   `bindChildren(node)` (`:2790`) (or specialized recursion for containers). So each node gets
   parent-set, symbol-declared, then its children walked.
3. **`bindWorker` kind-dispatch** — `bindWorker(node)` `binder.ts:2846` switches on `node.kind`
   (`:2847`) and routes to per-kind handlers, e.g. `bindBlockScopedDeclaration` (`:2437`),
   `bindFunctionDeclaration` (`:3709`), `bindVariableDeclarationOrBindingElement` (`:3648`). Each
   handler computes the right `SymbolFlags`/excludes and calls `declareSymbolAndAddToSymbolTable`.
4. **Symbol-table insert: `declareSymbolAndAddToSymbolTable` + `declareSymbol`** —
   `declareSymbolAndAddToSymbolTable` `binder.ts:2264` picks the destination table
   (`container.symbol.exports` / `.members` / `container.locals` / `file.locals`) and calls
   `declareSymbol(table, parent, node, symbolFlags, symbolExcludes)` (`:2281`/`:2293`/`:2320`/…).
   `declareSymbol` `binder.ts:749` is the merge/conflict routine: it looks up the name in the table
   and branches — **no existing symbol → create a new one and add it** (`:786-788`); **existing +
   `isReplaceableByMethod` mismatch → return the existing one unchanged** (`:790-792`); **existing +
   conflicts (`symbol.flags & excludes`) → report a duplicate-identifier error and create a fresh
   symbol** (`:793+`, messages chosen by flag at `:815-824`); **existing + compatible → merge: add
   this node into the symbol's declarations list** (the "Otherwise" branch). The returned `Symbol` is
   attached to the node.
5. **Container-scope tracking** — `getContainerFlags(node)` `binder.ts:3815` (called at `:2788`)
   classifies whether a node opens a new container (function/class/module/block scope); `bindChildren`
   (`:1096`) / `bindEach` (`:1084`) / `bindEachChild` (`:1092`) drive the walk while the binder's
   `container`/`blockScopeContainer`/`lastContainer` variables track the current scope. That current
   container is what `declareSymbolAndAddToSymbolTable` consults to decide whether a local goes into
   `container.locals`, `container.symbol.members`, or `container.symbol.exports`.

## Acceptable extras (credit, never required)

- The default-export naming inside `declareSymbol` (`binder.ts:754-756`, `InternalSymbolName.Default`)
  and the `Computed` name for computed-property members.
- `createSymbol` (`binder.ts:630`) — the underlying symbol constructor `declareSymbol` uses.
- The flow-node / reachability side of binding: `bindVariableDeclarationFlow` (`:2070`), the
  `currentFlow`/`currentReturnTarget`/`currentBreakTarget` control-flow targets, and
  `(node as Identifier).flowNode = currentFlow` in `bindWorker` (`:2873`). Binding builds the
  control-flow graph the checker's CFA later uses.
- `delayedBindJSDocTypedefTag` / `bindJSDocImports` (`:590-591`) — JSDoc typedef/import binding that
  runs after the main walk.
- The `bindEachFunctionsFirst` hoisting step (`:1079`) — function declarations are bound before other
  siblings to model hoisting.
- `classifiableNames` (`:578`) populated when a symbol has `SymbolFlags.Classifiable`, used by the
  language service.

## Common misses / wrong turns

- Describing only `declareSymbol` (the merge/conflict) and forgetting the walk (`bind`/`bindWorker`) —
  that's half the subsystem; the prompt names both the walk and the symbol-table insert.
- Omitting the container-scope tracking (`getContainerFlags` + `container`/`blockScopeContainer`) —
  the prompt explicitly asks how the walk tracks which scope it's in so locals go to the right table.
- Treating `bindWorker`'s switch as the whole story without `bind`'s setParent + recurse structure.
- Confusing the binder with the checker — the binder attaches symbols/parent/flow; it does *not*
  compute types or resolve imports to their final targets (that is the checker's `resolveSymbol`).
- Citing `getContainerFlags`/`declareSymbol` without linking them through `bind` → `bindWorker` →
  `declareSymbolAndAddToSymbolTable` → `declareSymbol` (the cooperation is the point).

## Model answer

Binding a file starts at `bindSourceFile` (`binder.ts:502`), which calls the `binder` closure made by
`createBinder()` (`:509`); the inner `bindSourceFile` (`:571`) sets up state and, if `file.locals`
isn't set, calls `bind(file)` (`:586`). `bind(node)` (`:2751`) is the focal walk: it first
`setParent(node, parent)` (`:2755`), then `bindWorker(node)` (`:2779`) to declare the node's symbol,
then for non-token nodes sets `parent = node`, computes `getContainerFlags(node)` (`:2788`) and
recurses via `bindChildren` (`:2790`). `bindWorker` (`:2846`) switches on `node.kind` (`:2847`) and
routes to per-kind handlers — `bindBlockScopedDeclaration` (`:2437`), `bindFunctionDeclaration`
(`:3709`), `bindVariableDeclarationOrBindingElement` (`:3648`) — each of which calls
`declareSymbolAndAddToSymbolTable` (`:2264`), which picks the destination table
(`container.symbol.exports`/`.members`/`container.locals`/`file.locals`) and calls `declareSymbol`
(`:749`). `declareSymbol` looks the name up and branches: no existing symbol → create + add
(`:786-788`); existing + `isReplaceableByMethod` mismatch → return the existing (`:790-792`); existing
+ conflicting flags → report a duplicate-identifier error and start a fresh symbol (`:793+`); existing
+ compatible → merge by appending this node to the symbol's declarations. Throughout, the binder
tracks the current scope via `getContainerFlags` (`:3815`) and the `container`/`blockScopeContainer`
variables, advanced by `bindChildren`/`bindEach` (`:1096`/`:1084`), so each declaration is inserted
into the correct table. The result is a file whose nodes carry `parent`, a `Symbol` (in the right
table), and a flow node — ready for the checker.
