# L4 reference answer key — the connection pool subsystem

Pinned source: `experiment/repos/rails` (SHA abbcb8d7…). All cites re-verified.

A bounded cooperating cluster in
`activerecord/lib/active_record/connection_adapters/abstract/connection_pool.rb` plus its
`connection_pool/queue.rb` and `connection_pool/reaper.rb` siblings. Four cooperating components —
the pool, the available-connections `Queue`, the background `Reaper`, and the per-thread lease —
around the lifecycle of a leased connection: checkout, return, reclaim, idle-evict.

## Required spine (gradeable core — must hit all)

1. **`ConnectionPool` — the central manager** — `connection_pool.rb:129` (`class ConnectionPool`).
   It owns `@connections` (all live connections), `@available` (the `Queue` of idle ones), and
   `connection_lease` (the per-thread checkout state).
2. **Checkout / acquire — get a connection, waiting if the pool is exhausted** — `checkout`
   (`connection_pool.rb:638`) calls `acquire_connection(checkout_timeout)` (`:640`).
   `acquire_connection` (`:1166`) tries, in order: `@available.poll` (grab an idle one) →
   `try_to_queue_for_background_connection` (wait) → `try_to_checkout_new_connection` (make a new
   one up to `@max_connections`); if all fail it calls `reap` and retries, finally blocking on
   `@available.poll(checkout_timeout)` (`:1167-1183`). The waiting itself happens inside the
   `Queue`'s condition variable.
3. **The `Queue` — the idle/waiting FIFO** — `connection_pool/queue.rb:12` (`class Queue`).
   `add(element)` (`:36`) returns a connection to the available set and signals waiters;
   `poll(timeout)` (`:86`) pops an idle connection or blocks on the condition variable until one
   appears or the timeout fires; `any_waiting?` (`:21`) / `num_waiting` (`:29`) report how many
   threads are blocked. This is the synchronization boundary between checkout and checkin.
4. **Checkin / return — release a connection for reuse** — `checkin` (`connection_pool.rb:666`)
   clears the lease (`connection_lease.clear(conn)`), expires the connection, and
   `@available.add conn` (`:671-673`), which wakes any thread waiting in `Queue#poll`. The user-facing
   wrapper is `with_connection` (`:451`): it `yield lease.connection = checkout` and
   `ensure release_connection(lease)` (`:466-469`), so the return is automatic.
5. **Reaper — background reclaim of dead-thread connections** — `Reaper`
   (`connection_pool/reaper.rb:33`) is a per-frequency background thread registered via
   `Reaper.register_pool` (`:44`); it periodically calls `pool.reap`.
   `ConnectionPool#reap` (`connection_pool.rb:712`) finds connections whose `conn.owner` thread is
   not alive (`:715-717`), steals them (`conn.steal!` `:719`), then for each resets+checks it in if
   still active, else removes it (`:724-730`). This recovers connections leaked by dying threads.
6. **Flush — idle eviction over time** — `flush(minimum_idle = @idle_timeout)`
   (`connection_pool.rb:735`) removes connections idle longer than the threshold via `remove`
   (`:680`), which deletes from `@connections`/`@available` and, if waiters exist, proactively makes
   a new connection (`bulk_make_new_connections` `:703`) so blocked threads are not stranded.

## Acceptable extras (credit, never required)

- `release_connection` (`:432`) and the per-thread `connection` accessor (just above `:432`) — the
  implicit checkout tied to the current thread's lease.
- `clear_reloadable_connections` (`:596`) / `clear_reloadable_connections!` (`:620`) / `flush!`
  (`:774`) — the reload/flush-on-fork variants of eviction, used when the app code reloads.
- `new_connection` (`:893`) / `try_to_checkout_new_connection` — the actual adapter construction
  when a brand-new connection is needed.
- `@pinned_connection` handling in `checkout` (`:639-656`) — connections pinned to a thread (e.g.
  for a fiber/isolation) bypass the queue.
- `ConnectionTimeoutError` raised from `acquire_connection`'s rescue (`:1185`) when the checkout
  timeout fires — the user-visible "pool exhausted" signal.
- The `Lease` / `connection_lease` stickiness (`with_connection`'s `prevent_permanent_checkout:`,
  `:451-469`) — leases can be sticky across requests.

## Common misses / wrong turns

- Describing only checkout and forgetting the return (`checkin`/`with_connection` ensure) — that's
  half the lifecycle; the prompt names both.
- Omitting the `Reaper`/`reap` path — the "background process reclaims dead-thread connections" arm
  is explicitly asked for and is one of the four cooperating pieces.
- Omitting `flush`/idle eviction — the "idle connections evicted over time" arm.
- Treating the `Queue` as just a list rather than the synchronization boundary (condition variable)
  where exhausted-pool threads actually wait.
- Citing the adapter (`AbstractAdapter`) or `with_connection` only, without the pool/queue/reaper
  cluster — the adapter is the *thing pooled*, not the pool machinery.

## Model answer

Rails shares a fixed set of connections across threads via `ConnectionPool`
(`connection_pool.rb:129`), which owns `@connections` (all live ones), `@available` (a `Queue` of
idle ones, `connection_pool/queue.rb:12`), and a per-thread lease. To **check out** a connection,
`checkout` (`:638`) calls `acquire_connection` (`:1166`), which tries `@available.poll`, then
queuing/background checkout, then making a new connection up to the max; if all fail it `reap`s and
retries, finally blocking on `@available.poll(checkout_timeout)`. The blocking happens in the
`Queue`'s condition variable — that is where an exhausted-pool thread waits (`poll` `:86`), and
`add` (`:36`) signals it. To **return** a connection, `checkin` (`:666`) clears the lease and does
`@available.add conn` (`:673`); user code normally goes through `with_connection` (`:451`) which
`yield lease.connection = checkout` and `ensure release_connection(lease)` (`:466-469`). A
background **`Reaper`** (`reaper.rb:33`, registered via `register_pool` `:44`) periodically calls
`pool.reap` (`:712`), which steals connections whose owner thread is dead (`:715-719`) and either
resets+checks them in or removes them. And **`flush`** (`:735`) evicts connections idle longer than
`@idle_timeout` via `remove` (`:680`), proactively making new connections if threads are waiting.
The pool, queue, reaper, and lease cooperate around one lifecycle: acquire, return, reclaim, evict.
