# L5 reference answer key — the journey of a `save` across subsystems

Pinned source: `experiment/repos/rails` (SHA abbcb8d7…). All cites re-verified.

A cross-cutting concern threading five subsystems around one operation: Validations (ActiveModel),
Callbacks (ActiveSupport), attribute/change selection (AttributeMethods), Persistence/SQL
(Persistence + connection adapters), Associations (AutosaveAssociation), and Transactions. Files:
`activerecord/lib/active_record/validations.rb`, `activemodel/lib/active_model/validations.rb`,
`activesupport/lib/active_support/callbacks.rb`, `activerecord/lib/active_record/callbacks.rb`,
`activerecord/lib/active_record/persistence.rb`, `activerecord/lib/active_record/attribute_methods.rb`,
`activerecord/lib/active_record/autosave_association.rb`, `activerecord/lib/active_record/transactions.rb`.

## Required spine (gradeable core — must hit all, in order of the save flow)

1. **Validation gate (Validations subsystem)** — the validations module overrides `save`
   (`activerecord/lib/active_record/validations.rb:47`) to do `perform_validations(options) ? super
   : false` (`:48`). `perform_validations` (`:90`) is `options[:validate] == false ||
   valid?(options[:context])`. `valid?` (`validations.rb:69`) calls `super` (ActiveModel's
   `valid?` `activemodel/lib/active_model/validations.rb:377`, which clears `errors` and runs
   `run_validations!`) and checks `errors.empty?`. **If invalid, `save` returns false and the write
   is skipped** — validation gates everything below.
2. **Callback gate (Callbacks subsystem)** — `super` reaches `Persistence#save`
   (`persistence.rb:415`), which calls `create_or_update`. AR::Callbacks overrides
   `create_or_update` (`activerecord/lib/active_record/callbacks.rb:439`) as `_run_save_callbacks {
   super }`, and overrides `_create_record` (`:443`) / `_update_record` (`:447`) as
   `_run_create_callbacks { super }` / `_run_update_callbacks { record_update_timestamps { super } }`.
   The `_run_*_callbacks` shims are generated by `define_model_callbacks :save, :create, :update,
   :destroy` (`callbacks.rb:415`) and ultimately call `run_callbacks` (`activesupport/lib/active_support/callbacks.rb:97`),
   whose return is `env.value = !env.halted && (block result)` (`activesupport/lib/active_support/callbacks.rb:110`) — a halted
   before-callback skips the block (the real write) and yields false.
3. **Change / attribute selection (AttributeMethods subsystem)** — `create_or_update`
   (`persistence.rb:990`) branches on `new_record?` to `_create_record` (`:1019`) or `_update_record`
   (`:999`). `_update_record` filters with `attributes_for_update(attribute_names)`
   (`attribute_methods.rb:508`) — drops readonly/counter-cache/virtual columns; `_create_record`
   filters with `attributes_for_create` (`attribute_methods.rb:519`) and `attributes_with_values`.
   This is the dirty/attribute layer deciding *which columns* go into the SQL.
4. **Persistence / SQL (Persistence + adapters)** — `_create_record` (`persistence.rb:1019`) calls
   `self.class._insert_record(connection, attributes_with_values(attribute_names), returning_columns)`
   (`:1027`); `_insert_record` (`persistence.rb:238`) builds an `Arel::InsertManager` and runs
   `connection.insert(…)` (`:257`). `_update_record` (`:999`) calls `_update_row` (`:957`), which
   calls `_update_record_with_result` / `connection.update`. This is where the record becomes an
   actual INSERT/UPDATE against the DB.
5. **Association cascade (AutosaveAssociation)** — saving a record cascades to its associations via
   autosave callbacks registered in `autosave_association.rb`: `around_save
   :around_save_collection_association` (`:193`) and `before_save save_method` (`:213`) (for
   has_one/belongs_to: `save_has_one_association` `:473` / `save_belongs_to_association` `:536`;
   collections: `save_collection_association` `:419` via `around_save_collection_association` `:402`).
   These run as part of the save-callback chain (step 2), so associated records are saved in lockstep
   with the parent.
6. **Transaction tie-in (Transactions subsystem)** — when the save runs inside a `transaction`,
   commit/rollback dispatches back to the record: `committed!` (`transactions.rb:402`) runs
   `_run_commit_callbacks` (`:406`) and `rolledback!` (`:414`) runs `_run_rollback_callbacks`
   (`:416`) and restores state (`:419`), using the `:commit`/`:rollback` callbacks declared via
   `define_callbacks :commit, :rollback,` (`transactions.rb:10`). So the save's outcome is hooked
   to the surrounding transaction's fate — commit fires after-commit callbacks, rollback restores
   the record and fires after-rollback.

## Acceptable extras (credit, never required)

- `save!` (`validations.rb:53`) raising `RecordInvalid` when `perform_validations` is false — the
  raising variant of the validation gate.
- `touch` / `_touch_row` (`persistence.rb:947`) and timestamp updating
  (`record_update_timestamps` in `_update_record`'s callback wrapper, `callbacks.rb:447`) — the
  updated_at/on touch path that piggybacks on save.
- Partial-writes / `changed_attribute_names` — the deeper dirty-tracking that, when
  `partial_writes` is on, narrows `_update_record`'s `attribute_names` to only changed columns (the
  `AttributeMethods::Dirty` / mutation-tracker layer beneath `attributes_for_update`).
- `before_save`/`after_save` user callbacks and `:if`/`:unless` halting — concrete examples of the
  callback gate halting the chain.
- The query-cache invalidation around the write (`skip_query_cache_if_necessary` /
  `connection.clear_query_cache` on writes) — a sibling cross-cutting effect of a save.
- Nested-attributes / `accepts_nested_attributes_for` building on top of the autosave cascade.

## Common misses / wrong turns

- Stopping at "save runs validations then writes the row" and missing the callback wrapping
  (`_run_save_callbacks`/`create_or_update` override) — that is the central architectural point
  (the write is the *block inside* the callback chain, not a sibling of it).
- Treating association autosave as separate from save rather than implemented *as* save callbacks
  (`around_save`/`before_save` in `autosave_association.rb`) — it threads through the same callback
  gate.
- Omitting the transaction tie-in (`committed!`/`rolledback!` + `:commit`/`:rollback` callbacks) —
  the prompt explicitly asks how the outcome ties into the surrounding transaction.
- Citing `valid?`/`save` but not explaining that invalid → `save` returns false *before* any write
  (the gating semantics).
- Confusing the order: e.g. placing association autosave after the SQL write rather than as a
  callback woven around it.

## Completeness scale

- **Full:** traces all six stages in order — validation gate (`validations.rb:47`/`:90`/`:69`) →
  callback gate (`callbacks.rb:439`/`:443`/`:447` via `run_callbacks` AS:97) → attribute/change
  selection (`attribute_methods.rb:508`/`:519`) → SQL write (`persistence.rb:238`/`:957`) →
  association autosave (`autosave_association.rb:193`/`:213`) → transaction commit/rollback
  (`transactions.rb:402`/`:414`) — and explains how each stage gates or threads the next, with
  cites across the modules.
- **Partial:** covers validation + write but misses the callback-wrapping, or misses autosave, or
  misses the transaction tie-in (i.e. 4 of 6 stages with one whole subsystem dropped).
- **Miss:** only "save validates then inserts/updates," or lists the stages without showing how they
  connect (no callback-chain, no transaction hook).

## Model answer

A `save` threads several subsystems. First the Validations module overrides `save`
(`validations.rb:47`) so it runs `perform_validations` (`:90` = `options[:validate] == false ||
   valid?(…)`) where `valid?` (`:69`) delegates to ActiveModel's `valid?`
(`activemodel/…/validations.rb:377`, clearing errors and running validators); if invalid, `save`
returns false and nothing is written — the gate. On success `super` reaches `Persistence#save`
(`persistence.rb:415`) → `create_or_update`. AR::Callbacks wraps that: `create_or_update`
(`callbacks.rb:439`) is `_run_save_callbacks { super }`, and `_create_record`/`_update_record`
(`:443`/`:447`) are `_run_create_callbacks`/`_run_update_callbacks { … super }`; these shims come
from `define_model_callbacks :save, :create, :update, :destroy` (`:415`) and call `run_callbacks`
(`activesupport/…/callbacks.rb:97`), whose result is `!env.halted && (block result)` (`activesupport/…/callbacks.rb:110`) — a
halted before-callback skips the block, i.e. skips the write. Inside that block,
`create_or_update` (`persistence.rb:990`) picks `_create_record` (`:1019`) or `_update_record`
(`:999`); `_update_record` narrows columns via `attributes_for_update`
(`attribute_methods.rb:508`) and `_create_record` via `attributes_for_create` (`:519`) — the
change/attribute layer. Then `_insert_record` (`persistence.rb:238`) issues `connection.insert`
(`:257`) or `_update_row` (`:957`) issues the UPDATE — the actual SQL. Saving also cascades to
associations: `AutosaveAssociation` registers `around_save :around_save_collection_association`
(`autosave_association.rb:193`) and `before_save save_method` (`:213`) (has_one/belongs_to via
`save_has_one_association`/`save_belongs_to_association` `:473`/`:536`), so associated records save
as part of the same callback chain. Finally, if the save is inside a `transaction`, the
transaction's outcome dispatches back to the record — `committed!` (`transactions.rb:402`) runs
`_run_commit_callbacks` (`:406`) and `rolledback!` (`:414`) runs `_run_rollback_callbacks` (`:416`)
and restores state, via the `:commit`/`:rollback` callbacks declared at `transactions.rb:10`. So
one `save` crosses validation, callbacks, change-selection, SQL, associations, and the transaction
boundary, each stage gating or threading the next.
