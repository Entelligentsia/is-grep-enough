# L1 reference answer key — the per-attribute value container (ActiveModel::Attribute)

Pinned source: `experiment/repos/rails` (SHA abbcb8d7…). All cites re-verified.

The entity is `ActiveModel::Attribute` in `activemodel/lib/active_model/attribute.rb`. It is the
in-memory container that holds a single named attribute's value for one model instance, independent
of the record object — analogous to how a single Redis value lives in a `robj` independent of the
key. A model's attributes are an `AttributeSet` of these.

## Required spine (gradeable core — must hit all)

1. **The container is `ActiveModel::Attribute`** — `attribute.rb:6` (`class Attribute # :nodoc:`).
   Instances are created through factory class methods, not `new`: `from_database` (`:8`),
   `from_user` (`:12`), `with_cast_value` (`:16`), `null` (`:20`), `uninitialized` (`:24`).
2. **`name` — the attribute's name** — exposed via `attr_reader :name, :value_before_type_cast, :type`
   (`attribute.rb:29`); set in `initialize` (`:33`).
3. **`value_before_type_cast` — the raw value exactly as it arrived, before any conversion** —
   attr_reader `:29`; stored untouched in `initialize` (`:35`). This is the "as-received" value.
4. **`value` — the type-cast value, lazily computed and memoized, distinct from
   `value_before_type_cast`** — `def value(&)` `attribute.rb:41`; body `@value = type_cast(value_before_type_cast)
   unless defined?(@value)` (`:43`). The cast only happens on first read and is cached.
5. **`type` — the ActiveModel::Type object that performs the conversion** — attr_reader `:29`; the
   same `type` instance supplies `cast`/`deserialize`/`serialize`. `type_cast` itself is abstract on
   the base class (`def type_cast(*); raise NotImplementedError; end` `:107`) — the subclass chosen
   by the factory decides *which* type method runs (see spine 6).
6. **`original_attribute` — where the value came from, for change detection** — private
   `attr_reader :original_attribute` (`:160`), aliased to `assigned?` (`:161`). When a value is
   assigned via `with_value_from_user`, the previous attribute is stashed as `original_attribute`
   so `changed?`/`original_value` can compare the new value against the prior one.

## Acceptable extras (credit, never required)

- **The variant subclasses differ in `type_cast`**, which is the real point of the factory split:
  `FromDatabase` (`:181`) casts with `type.deserialize(value)` (`:183`); `FromUser` (`:205`) casts
  with `type.cast(value)` (`:207`); `WithCastValue` (`:220`) returns the value unchanged (`:221`);
  `Null` (`:230`) returns `nil` (`:235`); `Uninitialized` (`:250`) yields the name and has no value.
  This mirrors how one logical "attribute" can be rehydrated differently depending on its source.
- **`value_for_database`** — the serialized-for-SQL form — `attribute.rb:55`, computed via
  `_value_for_database` → `type.serialize(value)` (`:173`); `FromUser` overrides it to use
  `Type::SerializeCastValue.serialize` (`:215`, `type/serialize_cast_value.rb:25`). So three value
  forms exist: `value_before_type_cast` (raw in), `value` (cast out), `value_for_database` (serialized out).
- `has_been_read?` / `defined?(@value)` (`:119`) — whether `value` has been materialized yet (used
  by in-place change detection `changed_in_place?` `:70`).
- `came_from_user?` (`:115`, `FromUser#came_from_user?` `:210`) — provenance flag for mass-assignment
  detection.
- The immutability/sharing note: `with_value_from_user`/`with_value_from_database`/`with_type`
  (`attribute.rb:81`/`:86`/`:90`) return *new* attribute objects rather than mutating — attributes
  are effectively immutable values.

## Common misses / wrong turns

- Pointing at `ActiveRecord` column objects or `AttributeSet` (`attribute_set.rb`) instead of the
  single-attribute container — those are the collection/DB-metadata layer, not the per-value holder.
- Treating `value` and `value_before_type_cast` as the same thing — the prompt explicitly asks for
  the distinction; this is the central gradeable point (raw-in vs cast-out, lazy memoized).
- Naming `type` as a Ruby class or a symbol rather than the ActiveModel::Type instance that owns
  `cast`/`deserialize`/`serialize`.
- Omitting `original_attribute` (the change-detection/provenance field the prompt names) or not
  explaining *why* it exists (so `changed?` can compare new vs prior).

## Model answer

A single attribute on a model is held in an `ActiveModel::Attribute` object
(`activemodel/lib/active_model/attribute.rb:6`), built through a factory rather than `new`:
`from_database`/`from_user`/`with_cast_value`/`null`/`uninitialized` (`:8`/`:12`/`:16`/`:20`/`:24`).
It carries the attribute's `name` and, critically, two value forms: `value_before_type_cast`
(`:29`, set in `initialize` `:35`) — the raw value exactly as it arrived, untouched — and `value`
(`:41`), the type-cast value, computed lazily as `@value = type_cast(value_before_type_cast) unless
defined?(@value)` (`:43`) and then memoized. The conversion is done by `type` (`:29`), the
ActiveModel::Type instance attached to the attribute; `type_cast` is abstract on the base class
(`:107`) and the factory-chosen subclass supplies the behavior — `FromDatabase` uses
`type.deserialize` (`:183`), `FromUser` uses `type.cast` (`:207`) — so the same logical attribute is
rehydrated differently depending on whether it came from the database or the user. Finally
`original_attribute` (private attr_reader `:160`, aliased `assigned?` `:161`) remembers the prior
attribute when a new value is assigned, which is what lets `changed?`/`original_value` compare the
current value against where it came from. (A third form, `value_for_database` `:55` via
`type.serialize` `:173`, is the SQL-bound serialization; attributes are immutable values, so
`with_value_from_user` etc. return new objects rather than mutating.)
