# L2 reference answer key — on-access lazy materialization of a Relation

Pinned source: `experiment/repos/rails` (SHA abbcb8d7…). All cites re-verified.

The focal routine is `ActiveRecord::Relation#load` plus the `loaded?` state flag. Everything lives
in `activerecord/lib/active_record/relation.rb`. The relation is lazy: a query is not run until an
accessor forces it. The gradeable point is *which accessors force materialization vs which
deliberately avoid it*, and how each branches on `loaded?`.

## Required spine (gradeable core — must hit all)

1. **The focal materializer `load`** — `relation.rb:1205` (`def load(&block)`). Body: `if !loaded? ||
   scheduled?; @records = exec_queries(&block); @loaded = true; end; self` (`:1206-1210`). It runs
   `exec_queries` only when not already loaded, caches the rows in `@records`, flips `@loaded`, and
   returns `self` (the relation, not the records).
2. **`loaded?` — the state guard** — `attr_reader … :loaded` (`:72`) with `alias :loaded? :loaded`
   (`:75`). It is the boolean every accessor branches on.
3. **Materializing accessors — force `load` and return records:** `records` (`:353`) calls `load`
   then returns `@records` (`:354`); `to_ary`/`to_a` (`:348`) returns `records.dup` (`:349`), so
   touching the array form triggers load via `records`. These are the "I need the actual rows" path.
4. **Short-circuiting accessors — branch on `loaded?` to AVOID loading when possible:**
   - `size` (`:364`): `if loaded?; records.length; else; count(:all); end` — uses a COUNT query
     rather than materializing when not loaded (`:365-370`).
   - `empty?` (`:373`): `if loaded?; records.empty?; else; !exists?; end` — uses an existence query
     instead of loading (`:376-380`).
   - `inspect` (`:1316`) and `pretty_print` (`:1291`): `subject = loaded? ? records : annotate("loading for
     inspect"/"loading for pp")` (`:1317`/`:1292`) — when not loaded they print a placeholder
     instead of forcing the query.
5. **`reload` resets then loads** — `relation.rb:1215` (`def reload`): `reset; load` (`:1217`),
   where `reset` (`:1220`) clears `@loaded` (sets `@loaded = nil` `:1226`) among other state, so the
   next access re-materializes. This is the explicit invalidate-and-refetch counterpart to the lazy
   trigger.

## Acceptable extras (credit, never required)

- `none?`/`any?`/`one?`/`many?` (`:383`/`:401`/`:417`/`:433`) follow the same loaded?-branching
  pattern (use cached records if loaded, else a limited count/existence query) — additional
  examples of the short-circuiting family.
- `empty?`'s `@none` early-out (`:374`, `return true if @none`) — a `none`-relation never loads.
- `load_async` (`:1164`) / `scheduled?` — the async pre-fetch path; `load` also re-runs when
  `scheduled?` (`:1206`) to collect an already-launched future result.
- `each` (defined via `Enumerable` / delegated through `records`) ultimately materializes via
  `records` — belongs to the materializing family.
- The `@future_result` / `FutureResult` machinery behind `scheduled?` — async loading; a plus,
  never required.

## Common misses / wrong turns

- Listing user-facing query builders (`where`, `order`) as the materialization trigger — those only
  mutate the relation; they do not run SQL. The trigger is `load`/`records`/`to_ary`.
- Treating it as a star of *callers of `load`* without the loaded?-branching distinction — the
  gradeable insight is the split between accessors that *force* load and accessors that *avoid* it.
- Missing the short-circuiting family entirely (only mentioning `records`/`to_ary`), or vice versa.
- Claiming `size`/`empty?` always load — they explicitly avoid loading via COUNT/EXISTS when not
  loaded; that's the whole point.
- Confusing `reload` (explicit reset+load) with `load` (lazy materialize-if-needed).

## Completeness scale

- **Full:** names `load` (`:1205`) + `loaded?` (`:75`) as the focal pair, and explains both families
  — materializers (`records` `:353`, `to_ary` `:348`) that force load, and short-circuiters
  (`size` `:364`, `empty?` `:373`, `inspect` `:1316`) that branch on `loaded?` to avoid it — with
  cites; notes `reload` as the invalidate path.
- **Partial:** names `load`/`loaded?` and one family but omits the other, or lists accessors
  without explaining the loaded?-branch.
- **Miss:** only describes query building (`where`), or says "records load when you use them"
  without the focal routine or the branching distinction.

## Model answer

A relation doesn't run its query until something forces it. The focal routine is `Relation#load`
(`relation.rb:1205`): when called it checks `if !loaded? || scheduled?`, and only then runs
`exec_queries` to fetch rows into `@records` and sets `@loaded = true`, returning `self`
(`:1206-1210`). `loaded?` (alias of `@loaded`, `:75`) is the state flag every accessor branches on.
Two families of accessors consult it. The **materializing** ones force `load`: `records` (`:353`)
calls `load` then returns `@records`, and `to_ary`/`to_a` (`:348`) returns `records.dup`, so
converting to an array triggers the query. The **short-circuiting** ones deliberately avoid
loading when not loaded: `size` (`:364`) uses `records.length` if loaded else a `count(:all)`
query; `empty?` (`:373`) uses `records.empty?` if loaded else `!exists?`; and `inspect`
(`:1316`)/`pretty_print` (`:1291`) print `annotate("loading for …")` instead of the records when not
loaded. `reload` (`:1215`) is the explicit counterpart — `reset` clears `@loaded` (`:1226`) and
then `load` re-materializes on next access. So the same `loaded?` flag splits ordinary access into
"fetch the rows now" versus "answer without fetching."
