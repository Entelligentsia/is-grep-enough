# L4 reference answer key — block acceptance & activation subsystem

Pinned source: `experiment/repos/bitcoin` (SHA 6d8e15dff015d3df8e87de63e650a5aee32ff12f, C++). All cites re-verified.

A bounded cluster across `validation.cpp` (the `ChainstateManager`/`Chainstate` classes). Four
cooperating stages around the check → record → activate → finalize boundary. Order within the
activation stage matters; the four stages must all be present and connected.

## Required spine (gradeable core — must hit all four stages)

1. **Entry: context-free check + accept** — `ChainstateManager::ProcessNewBlock`
   (`validation.cpp:4405`) is the entry point (from P2P, RPC, or the miner). It takes `cs_main`,
   runs `CheckBlock(*block, state, GetConsensus())` (`validation.cpp:4426`) — the context-free
   block checks (PoW, merkle root, per-tx `CheckTransaction`) — and only if that passes calls
   `AcceptBlock(block, state, &pindex, force_processing, nullptr, new_block, min_pow_checked)`
   (`validation.cpp:4429`). On failure it fires `BlockChecked` and returns
   (`validation.cpp:4433-4437`).
2. **Accept: contextual check + disk/index** — `ChainstateManager::AcceptBlock`
   (`validation.cpp:4305`) re-runs `CheckBlock(block, state, params.GetConsensus()) || !
   ContextualCheckBlock(block, state, *this, pindex->pprev)` (`validation.cpp:4357-4358`)
   (contextual = time, height, BIP34, etc. against the prior block), short-circuits already-known
   or low-work blocks (`validation.cpp:4338-4352`), then writes the block to disk
   (`m_blockman.WriteBlock(...)`, `validation.cpp:4376-4380`) and adds it to the block index so
   it is a candidate tip. `AcceptBlock` is what makes the block *available* to be activated.
3. **Activate: choose most-work tip + apply to UTXO set** — `ProcessNewBlock` then calls
   `ActiveChainstate().ActivateBestChain(state, block)` (`validation.cpp:4440`).
   `Chainstate::ActivateBestChain` (`validation.cpp:3330`) repeatedly calls
   `ActivateBestChainStep(state, *pindexMostWork, ...)` (`validation.cpp:3393`, def
   `validation.cpp:3198`) to advance toward the most-work valid tip. Each step calls
   `ConnectTip(state, pindexConnect, ..., disconnectpool)` (`validation.cpp:3243`, def
   `validation.cpp:3019`), which runs `ConnectBlock(*block_to_connect, state, pindexNew, view)`
   (`validation.cpp:3051`; `ConnectBlock` def `validation.cpp:2301`) — this is the stage that
   **applies the block's transactions to the UTXO set** (spend inputs, add outputs, enforce
   consensus on the connected block).
4. **Finalize: tip update + signals** — on a successful `ConnectTip`, `ActivateBestChainStep`
   calls `UpdateTip(pindexNew)` (`validation.cpp:3094`; `UpdateTip` def `validation.cpp:2896`),
   which advances `m_chain` and emits the tip-change notification (e.g.
   `m_chainman.m_options.signals->ActiveTipChange(...)`, fired from the activation path at
   `validation.cpp:3449`/`:3700`, gated on IBD at `validation.cpp:3370`/`:3419`). This is the
   completion/finalize stage that makes the new tip visible to the rest of the node (wallet, ZMQ,
   RPC).

## Acceptable extras (credit, never required)

- The `InvalidBlockFound` / `InvalidChainFound` path that `AcceptBlock` feeds on a bad block
  (`validation.cpp:4361`) — error handling, never required.
- The background/validation chainstate: `ProcessNewBlock` also calls
  `HistoricalChainstate()->ActivateBestChain(bg_state, block)` (`validation.cpp:4447`) when a
  background chainstate exists — a plus to mention; the active-chainstate spine above is
  required.
- `NotifyHeaderTip()` between accept and activate (`validation.cpp:4439`) — header-tip UI
  signaling; never required.
- The `NewPoWValidBlock` signal fired from `AcceptBlock` when the block builds on the current tip
  (`validation.cpp:4367-4369`) — distinct from the post-connection `ActiveTipChange`; credit if
  distinguished.
- The `disconnectpool`/rollback side of `ActivateBestChainStep` (disconnecting the old tip when
  reorging) — correct context, never required.

## Common misses / wrong turns

- Describing only `CheckBlock` and never reaching activation (`ActivateBestChain`/`ConnectBlock`)
  — the prompt asks how the block is *integrated into the active chain*, which requires the
  activation + UTXO-connect + tip-finalize stages.
- Stopping at `AcceptBlock` (disk/index) and missing `ConnectBlock` applying txs to the UTXO set
  (spine 3) — the load-bearing "applies the block's transactions to the UTXO set" stage.
- Missing `UpdateTip`/the tip-change signal (spine 4) — leaving the block "connected" with no
  account of how the tip change is finalized/signaled.
- Conflating `CheckBlock` (context-free, `validation.cpp:3925`/called at `:4426` and `:4357`)
  with `ContextualCheckBlock` (`validation.cpp:4358`) — the prompt distinguishes context-free
  *then* contextual; both must appear in the right order.
- Treating the whole thing as a single linear call chain rather than four cooperating stages
  (check / accept-to-disk / activate-UTXO / finalize-tip) — that misses the "how the stages fit
  together" synthesis.

## Completeness scale

- **Full:** all four stages — (1) `ProcessNewBlock`→`CheckBlock`+`AcceptBlock`, (2)
  `AcceptBlock`→`CheckBlock`+`ContextualCheckBlock`+disk/index, (3)
  `ActivateBestChain`→`ActivateBestChainStep`→`ConnectTip`→`ConnectBlock` (UTXO apply), (4)
  `UpdateTip` finalize + tip-change signal — connected, with cites, and the context-free vs
  contextual distinction preserved.
- **Partial:** 3 of 4 stages (commonly missing the `ConnectBlock` UTXO-apply stage OR the
  `UpdateTip`/signal finalize stage).
- **Miss:** only `CheckBlock`, or only the accept-to-disk half, or no account of how the block
  is connected to the active chain and the tip updated.

## Model answer

A newly received block enters via `ChainstateManager::ProcessNewBlock` (`validation.cpp:4405`).
It first runs the context-free `CheckBlock(*block, state, GetConsensus())` (`validation.cpp:4426`)
— proof-of-work, merkle roots, per-transaction `CheckTransaction` — and, on success, calls
`AcceptBlock` (`validation.cpp:4429`). `ChainstateManager::AcceptBlock` (`validation.cpp:4305`)
then applies the **contextual** checks: `CheckBlock` again together with
`ContextualCheckBlock(block, state, *this, pindex->pprev)` (`validation.cpp:4357-4358`, checking
time/height/BIP34 against the prior block), drops already-known or low-work blocks, writes the
block to disk (`m_blockman.WriteBlock`, `validation.cpp:4376-4380`) and inserts it into the block
index as a candidate tip. With the block recorded, `ProcessNewBlock` drives activation:
`ActiveChainstate().ActivateBestChain(state, block)` (`validation.cpp:4440`).
`Chainstate::ActivateBestChain` (`validation.cpp:3330`) loops over
`ActivateBestChainStep` (`validation.cpp:3393`, def `:3198`) toward the most-work valid tip; each
step calls `ConnectTip` (`validation.cpp:3243`, def `:3019`), which runs
`ConnectBlock(*block_to_connect, state, pindexNew, view)` (`validation.cpp:3051`, def `:2301`) —
the stage that **spends the inputs and adds the outputs**, applying the block's transactions to
the UTXO set under consensus rules. On success, `ActivateBestChainStep` finalizes via
`UpdateTip(pindexNew)` (`validation.cpp:3094`, def `:2896`), advancing `m_chain` and firing the
tip-change signal (`m_chainman.m_options.signals->ActiveTipChange(...)`, `validation.cpp:3449`/
`:3700`), so wallets, ZMQ, and RPC see the new tip. So the four cooperating stages — context-free
check, contextual check + record-to-disk/index, activate-and-connect-to-UTXO, finalize-tip-and-
signal — fit together as check → store → connect → announce.
