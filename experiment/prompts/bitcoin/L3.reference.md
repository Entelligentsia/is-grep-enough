# L3 reference answer key — inbound peer message path (socket bytes → per-type handler)

Pinned source: `experiment/repos/bitcoin` (SHA 6d8e15dff015d3df8e87de63e650a5aee32ff12f, C++). All cites re-verified.

A single ordered chain threaded through `net.cpp` and `net_processing.cpp`. Order is gradeable.
Two threads are involved (the socket thread reads/parse-prefills; the message-handler thread
dispatches); the agent must follow the handoff between them.

## Required spine (gradeable core — must hit all, in order)

1. **Socket thread entry + read** — `CConnman::ThreadSocketHandler` (`net.cpp:2264`) loops
   calling `SocketHandler()` (`net.cpp:2271`). `CConnman::SocketHandler` (`net.cpp:2118`) calls
   `SocketHandlerConnected(...)` (`net.cpp:2140`, def `net.cpp:2147`), which `Recv`s bytes from
   the socket (`pnode->m_sock->Recv(pchBuf, ...)`, `net.cpp:2206`) and feeds them to
   `pnode->ReceiveMsgBytes({pchBuf, (size_t)nBytes}, notify)` (`net.cpp:2210`).
2. **Transport absorbs bytes** — `CNode::ReceiveMsgBytes` (`net.cpp:668`) calls
   `m_transport->ReceivedBytes(msg_bytes)` (`net.cpp:677`) in a loop, accumulating into the
   transport (V1 at `net.cpp`/`V1Transport::ReceivedBytes`, or V2 at `net.cpp:1330`).
3. **A complete message is parsed out** — when `m_transport->ReceivedMessageComplete()` is true
   (`net.cpp:681`), `ReceiveMsgBytes` calls `CNetMessage msg = m_transport->GetReceivedMessage(
   time, reject_message)` (`net.cpp:685`; V1 impl `net.cpp:809`, V2 impl `net.cpp:1460`), turning
   the buffered bytes into a transport-agnostic `CNetMessage` (`net.h:236`) held on the node.
4. **Message-handler thread pulls a parsed message** — `CConnman::ThreadMessageHandler`
   (`net.cpp:3193`) loops over nodes and calls `m_msgproc->ProcessMessages(*pnode,
   flagInterruptMsgProc)` (`net.cpp:3214`). Inside `PeerManagerImpl::ProcessMessages`
   (`net_processing.cpp:5141`) it calls `node.PollMessage()` (`net_processing.cpp:5179`) to
   dequeue the next completed `CNetMessage` (`CNode::PollMessage` def `net.cpp:4110`).
5. **Hand to the per-message dispatcher** — `ProcessMessages` calls
   `ProcessMessage(peer, node, msg.m_type, msg.m_recv, msg.m_time, interruptMsgProc)`
   (`net_processing.cpp:5201`), passing the deserialized payload `msg.m_recv` and type
   `msg.m_type`.
6. **Resolve type → handler** — `PeerManagerImpl::ProcessMessage`
   (`net_processing.cpp:3603`) branches on `msg_type` against `NetMsgType` constants (e.g.
   `if (msg_type == NetMsgType::VERSION)` `net_processing.cpp:3621`, with the constants defined
   in `protocol.h`: `VERSION` `protocol.h:65`, `INV` `protocol.h:92`, `TX` `protocol.h:117`,
   `BLOCK` `protocol.h:127`).
7. **Execute the handler logic** — for the transaction case the terminal step is
   `if (msg_type == NetMsgType::TX)` (`net_processing.cpp:4460`) which deserializes
   `vRecv >> TX_WITH_WITNESS(ptx)` (`net_processing.cpp:4473`) and proceeds into tx handling; the
   `VERSION` branch analogously reads its fields (`net_processing.cpp:3637+`). This is where
   control finally reaches the per-type logic.

## Acceptable extras (credit, never required)

- The two-thread split (socket-thread prefill vs message-handler dispatch) and the
  `PollMessage`/`fMoreWork` re-loop (`net_processing.cpp:5201-5230`) — correctly noting the
  cross-thread handoff is a plus; the spine above already accounts for it.
- `ProcessGetData` / orphan-tx reprocessing that runs inside `ProcessMessages` before
  `PollMessage` (`net_processing.cpp:5155-5175`) — order-of-responses bookkeeping; never
  required.
- The `TRACEPOINT(net, inbound_message, ...)` / `CaptureMessage` instrumentation between the
  pull and the dispatch (`net_processing.cpp:5186-5198`) — observability, never required.
- V1-vs-V2 transport detail (V2 falls back to V1, `net.cpp:1337`) — the spine is
  transport-agnostic; mentioning either is fine.
- `SendMessages` (`net_processing.cpp:5792`) as the *outbound* counterpart — contrast only; the
  prompt is the inbound path.

## Common misses / wrong turns

- Stopping at `ReceiveMsgBytes` / transport parse without the cross-thread handoff to
  `ProcessMessages` → `ProcessMessage` (skips spine 4-5) — the prompt asks "until the
  per-message-type handler actually runs."
- Jumping straight to `ProcessMessage` and ignoring the socket-read/parse front end (skips
  spine 1-3) — the prompt asks "from the moment its bytes are read off the socket."
- Confusing `ProcessMessages` (the per-node loop, `net_processing.cpp:5141`) with
  `ProcessMessage` (the per-message dispatcher, `net_processing.cpp:3603`) — both must appear,
  in that order.
- Naming the terminal step as `ProcessMessage` itself rather than the actual per-type branch
  (`if (msg_type == NetMsgType::TX)` `net_processing.cpp:4460`) that runs the handler.
- Treating it as a star of callers rather than an ordered chain.

## Completeness scale

- **Full:** all 7 spine stops, in order — socket read → transport absorb → message parse →
  message-handler pull (`PollMessage`) → `ProcessMessage` dispatch → type-match → per-type
  handler — with cites, naming the cross-thread handoff and the terminal per-type branch (e.g.
  `NetMsgType::TX` at `net_processing.cpp:4460`).
- **Partial:** ≥5 of 7 in roughly correct order, but missing either the socket/parse front end
  OR the cross-thread `ProcessMessages`→`ProcessMessage` handoff OR the terminal per-type
  branch.
- **Miss:** only the `ProcessMessage` dispatch half, an out-of-order chain, or no account of how
  bytes become a `CNetMessage`.

## Model answer

A peer message starts in the socket thread: `CConnman::ThreadSocketHandler` (`net.cpp:2264`)
calls `SocketHandler()` (`net.cpp:2271`) → `SocketHandlerConnected` (`net.cpp:2147`), which
`Recv`s bytes (`net.cpp:2206`) and passes them to `pnode->ReceiveMsgBytes(...)` (`net.cpp:2210`).
`CNode::ReceiveMsgBytes` (`net.cpp:668`) feeds them to `m_transport->ReceivedBytes`
(`net.cpp:677`); once `m_transport->ReceivedMessageComplete()` (`net.cpp:681`) it extracts a
`CNetMessage` via `m_transport->GetReceivedMessage(...)` (`net.cpp:685`), i.e. raw bytes become a
transport-agnostic parsed message held on the node. The dispatch happens on a separate thread:
`CConnman::ThreadMessageHandler` (`net.cpp:3193`) calls `m_msgproc->ProcessMessages(*pnode, ...)`
(`net.cpp:3214`); `PeerManagerImpl::ProcessMessages` (`net_processing.cpp:5141`) pulls the next
message with `node.PollMessage()` (`net_processing.cpp:5179`) and calls
`ProcessMessage(peer, node, msg.m_type, msg.m_recv, msg.m_time, interruptMsgProc)`
(`net_processing.cpp:5201`). `PeerManagerImpl::ProcessMessage` (`net_processing.cpp:3603`)
matches `msg_type` against `NetMsgType` constants (`protocol.h:65/92/117/127`) and runs the
matching branch — e.g. `if (msg_type == NetMsgType::TX)` (`net_processing.cpp:4460`)
deserializes the transaction (`vRecv >> TX_WITH_WITNESS(ptx)`, `net_processing.cpp:4473`), which
is where the per-type handler logic finally executes.
