# L5 reference answer key — a submitted transaction's journey to peer relay (cross-cutting)

Pinned source: `experiment/repos/bitcoin` (SHA 6d8e15dff015d3df8e87de63e650a5aee32ff12f, C++). All cites re-verified.

Threads three subsystems: the RPC/broadcast entry (`rpc/mempool.cpp`, `node/transaction.cpp`),
mempool acceptance (`validation.cpp` — `ChainstateManager::ProcessTransaction`,
`AcceptToMemoryPool`, `MemPoolAccept`), and P2P relay (`net_processing.cpp` —
`PeerManagerImpl`). The deferred per-peer queue-then-flush indirection is the crux.

## Required spine (gradeable core — must hit all)

1. **RPC accepts the raw transaction** — the `sendrawtransaction` RPC handler (registered in
   `static RPCMethod sendrawtransaction()`, `rpc/mempool.cpp:47`) decodes the hex and calls
   `BroadcastTransaction(node, tx, err_string, max_raw_tx_fee, method, /*wait_callback=*/true)`
   (`rpc/mempool.cpp:127`), choosing `TxBroadcast::MEMPOOL_AND_BROADCAST_TO_ALL` as the method
   (`rpc/mempool.cpp:124-125`). (`submitpackage` at `rpc/mempool.cpp:1467` and the chain
   interface at `node/interfaces.cpp:690` are sibling callers.)
2. **Submit to mempool acceptance** — `BroadcastTransaction` (`node/transaction.cpp:32`) first
   checks the tx is not already confirmed/known, then calls
   `node.chainman->ProcessTransaction(tx, /*test_accept=*/false)` (`node/transaction.cpp`, the
   `MEMPOOL_AND_BROADCAST_TO_ALL` branch around `:78-88`).
   `ChainstateManager::ProcessTransaction` (`validation.cpp:4455`) calls
   `AcceptToMemoryPool(active_chainstate, tx, GetTime(), /*bypass_limits=*/false, test_accept)`
   (`validation.cpp:4463`). `AcceptToMemoryPool` (`validation.cpp:1781`) constructs
   `MemPoolAccept::ATMPArgs::SingleAccept(...)` (`validation.cpp:1791`) and invokes
   `MemPoolAccept(pool, active_chainstate).AcceptSingleTransactionAndCleanup(tx, args)`
   (`validation.cpp:1792`), whose core is
   `MemPoolAccept::AcceptSingleTransactionInternal` (`validation.cpp:1323`).
3. **Checks that must pass to enter the mempool** — inside the acceptance pipeline
   `MemPoolAccept::PreChecks` (`validation.cpp:786`) runs `CheckTransaction(tx, state)`
   (`validation.cpp:802`, context-free sanity), then `IsStandardTx(...)` (`validation.cpp:812`,
   policy standardness), with `MemPoolAccept`'s script-verification stage calling
   `CheckInputScripts(tx, state, m_view, scriptVerifyFlags, ...)` (`validation.cpp:1150`, the
   consensus+policy signature/script checks against the UTXO view). Only on
   `MempoolAcceptResult::ResultType::VALID` is the transaction inserted into the mempool
   (`AcceptSingleTransactionInternal` finalizes on success, `validation.cpp:1323` onward).
4. **Acceptance triggers relay (queued, not sent inline)** — back in `BroadcastTransaction`,
   once `ProcessTransaction` returned `VALID`, the `MEMPOOL_AND_BROADCAST_TO_ALL` branch calls
   `node.peerman->InitiateTxBroadcastToAll(txid, wtxid)` (`node/transaction.cpp:137`), having
   also recorded the unbroadcast via `node.mempool->AddUnbroadcastTx(txid)`
   (`node/transaction.cpp:~91`). This is the relay trigger — but it does **not** send bytes; it
   only queues per-peer announcements.
5. **Queue per-peer (deferred)** — `PeerManagerImpl::InitiateTxBroadcastToAll(txid, wtxid)`
   (`net_processing.cpp:2272`) iterates all peers (`net_processing.cpp:2274`), and for each peer
   with tx relay enabled inserts the wtxid into `tx_relay->m_tx_inventory_to_send`
   (`net_processing.cpp:2291`), gated on the version handshake being complete
   (`m_next_inv_send_time != 0s`, `net_processing.cpp:2287`). The set
   `m_tx_inventory_to_send` (`net_processing.cpp:306`) is held per-peer under
   `m_tx_inventory_mutex` — the deferred queue.
6. **Flush the queue as inventory messages to peers** — later, on the message-handler thread,
   `PeerManagerImpl::SendMessages` (`net_processing.cpp:5792`) drains
   `m_tx_inventory_to_send` (`net_processing.cpp:6113-6114`) into `vInvTx`, sorts/heapifies for
   privacy/priority (`net_processing.cpp:6121-6123`), and for each candidate builds
   `CInv{MSG_WTX, wtxid.ToUint256()}` or `CInv{MSG_TX, txinfo.tx->GetHash().ToUint256()}`
   (`net_processing.cpp:6145-6146`), erasing it from the to-send set (`net_processing.cpp:6135`)
   and pushing it into `vInv` (`net_processing.cpp:6157`); when `vInv` fills (`MAX_INV_SZ`) or
   the section ends it emits `MakeAndPushMessage(node, NetMsgType::INV, vInv)`
   (`net_processing.cpp:6160`/`:6172`).
   That INV is the outbound announcement; a peer's subsequent GETDATA is answered with the TX
   message body (the `ProcessMessage`/`getdata` path, not required here).

## Common misses / wrong turns

- Claiming `BroadcastTransaction` (or the RPC) sends the TX directly to peers — it does not; the
  indirection through `InitiateTxBroadcastToAll` → per-peer `m_tx_inventory_to_send` →
  `SendMessages` flush is the whole point and a likely miss.
- Stopping at mempool acceptance without the relay trigger (`InitiateTxBroadcastToAll`) — the
  prompt explicitly spans to "bytes announcing it going out to peers."
- Stopping at `InitiateTxBroadcastToAll` without the `SendMessages` flush (spine 6) — the queued
  announcement would never leave the node in that account.
- Missing the deferred queue-then-flush pattern (spine 4-5 vs 6) — i.e. not explaining that
  acceptance queues per-peer and the actual INV is emitted later by `SendMessages`.
- Conflating with the inbound path (a peer sending us a TX) rather than the locally-submitted
  outbound journey the prompt describes.
- Citing `ProcessNewBlock`/block relay instead of the transaction relay machinery — wrong
  subsystem.

## Completeness scale

- **Full:** all 6 spine elements connected — RPC `sendrawtransaction`→`BroadcastTransaction`,
  `ProcessTransaction`→`AcceptToMemoryPool`→`MemPoolAccept` with `CheckTransaction`/
  `IsStandardTx`/`CheckInputScripts` entry checks, `InitiateTxBroadcastToAll` relay trigger,
  per-peer `m_tx_inventory_to_send` queueing, and the `SendMessages` flush producing
  `CInv`/`NetMsgType::INV` bytes to peers — with cites, and explicit acknowledgement of the
  queue-then-flush indirection.
- **Partial:** ≥4 of 6, but missing either the acceptance-checks stage OR the
  `InitiateTxBroadcastToAll`/`SendMessages` flush bridge (i.e. an account that doesn't explain
  how a accepted tx actually reaches peer-bound bytes).
- **Miss:** a direct "RPC → peers" story, or only the mempool-acceptance half, or only the
  inbound peer-to-peer TX path.

## Model answer

A locally submitted transaction enters via the `sendrawtransaction` RPC (`rpc/mempool.cpp:47`),
which calls `BroadcastTransaction(node, tx, ..., TxBroadcast::MEMPOOL_AND_BROADCAST_TO_ALL, ...)`
(`rpc/mempool.cpp:127`). `BroadcastTransaction` (`node/transaction.cpp:32`) submits it to the
mempool by calling `node.chainman->ProcessTransaction(tx, /*test_accept=*/false)`
(`node/transaction.cpp`, `MEMPOOL_AND_BROADCAST_TO_ALL` branch), which runs
`AcceptToMemoryPool` (`validation.cpp:1781`) → `MemPoolAccept::AcceptSingleTransactionInternal`
(`validation.cpp:1323`). The acceptance pipeline (`MemPoolAccept::PreChecks`, `validation.cpp:786`)
runs the context-free `CheckTransaction` (`validation.cpp:802`), the policy `IsStandardTx`
(`validation.cpp:812`), and the script-verification `CheckInputScripts` (`validation.cpp:1150`)
against the UTXO view; only on `ResultType::VALID` is the tx inserted into the mempool. Back in
`BroadcastTransaction`, that success triggers relay: `node.peerman->InitiateTxBroadcastToAll(txid,
wtxid)` (`node/transaction.cpp:137`) — but this only *queues*, it does not send.
`PeerManagerImpl::InitiateTxBroadcastToAll` (`net_processing.cpp:2272`) inserts the wtxid into
each eligible peer's `tx_relay->m_tx_inventory_to_send` set (`net_processing.cpp:2291`), held
per-peer under `m_tx_inventory_mutex` (`net_processing.cpp:306`). The actual emission is
deferred to the message-handler thread: `PeerManagerImpl::SendMessages`
(`net_processing.cpp:5792`) drains `m_tx_inventory_to_send` (`net_processing.cpp:6113-6114`),
builds `CInv{MSG_WTX,...}`/`CInv{MSG_TX,...}` (`net_processing.cpp:6145-6146`), and emits
`MakeAndPushMessage(node, NetMsgType::INV, vInv)` (`net_processing.cpp:6160`/`:6172`) — the
inventory announcement that travels to peers, after which a peer's GETDATA pulls the full
transaction. So the journey is: RPC → mempool acceptance (consensus+policy checks) → per-peer
queued announcement → flushed INV bytes to peers, with the queue-then-flush indirection between
acceptance and the outbound announcement.
