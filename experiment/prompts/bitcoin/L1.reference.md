The entity is `class Coin` in `src/coins.h` — the structure the coins cache holds for a single
unspent transaction output.

## Required spine (gradeable core — must hit all)

1. **The container is `class Coin`** — `coins.h:34` (`class Coin`), introduced by the comment
   `* A UTXO entry.` at `coins.h:28`. This is the per-UTXO record carried through the
   `CCoinsView` caching hierarchy.
2. **`out` = the actual output payload** — `coins.h:38` (`CTxOut out;`), commented

   spent-able output.
3. **`fCoinBase : 1` = coinbase origin flag** — `coins.h:41` (`bool fCoinBase : 1;`), read via

   coinbase (relevant to maturity rules).
4. **`nHeight : 31` = height of the creating block** — `coins.h:44` (`uint32_t nHeight : 31;`),
   commented `//! at which height this containing transaction was included in the active block
   chain` (`coins.h:43`).
5. **Spentness is derived, not a flag — `IsSpent()` checks `out.IsNull()`** — `coins.h:83-84`
   (`bool IsSpent() const { return out.IsNull(); }`), with the class doc at `coins.h:30-31`
   noting a spent coin's `Coin` is "nulled out" (`Clear()` sets `out.SetNull()`, `coins.h:53-55`).
   The gradeable distinction: spentness lives in `out`, while provenance lives in
   `fCoinBase`/`nHeight` — two separate concerns in one struct.
6. **Height and coinbase are packed together on disk** — serialization at `coins.h:66-67`:
   `uint32_t code{(uint32_t{nHeight} << 1) | uint32_t{fCoinBase}}; ::Serialize(s,
   VARINT(code));` (deserialized symmetrically at `coins.h:74`), matching the format documented
   in the class header comment `coins.h:29` (`VARINT((height << 1) | (coinbase ? 1 : 0))`).

## Common misses / wrong turns

- Answering with `CTxOut` (the output itself) or `COutPoint` (the outpoint identifier) instead
  of `Coin` — those are the payload / the locator, not the cache container that also carries
  height + coinbase.
- Conflating spentness with the `fCoinBase`/`nHeight` bits — `IsSpent()` does NOT read those;
  it reads `out`. The prompt explicitly asks how spentness is told apart from provenance.
- Omitting the on-disk packing (`(nHeight<<1)|fCoinBase`) — the prompt names "how that height
  and coinbase flag are packed together when persisted," and it is a single VARINT code.
- Pointing at `CCoinsViewCache`/`CCoinsCacheEntry` instead of `Coin` — those hold/flag the
  coin, they are not the per-UTXO record itself.

## Model answer

A single UTXO in the coins cache lives in `class Coin` (`coins.h:34`, "A UTXO entry" at
`coins.h:28`). Its first member `out` (`coins.h:38`, `CTxOut`) is the actual output — the value
and scriptPubKey someone can spend. Alongside it, two provenance bitfields record where the coin
came from: `fCoinBase : 1` (`coins.h:41`, exposed as `IsCoinBase()` at `coins.h:59-61`) says the
creating transaction was a coinbase, and `nHeight : 31` (`coins.h:44`) is the height of the block
that included it — both feed maturity and consensus checks. Whether the coin *still exists* is a
separate question answered not by those bits but by the payload: `IsSpent()`
(`coins.h:83-84`) returns `out.IsNull()`, and a spent coin is represented by nulling `out` (see
`Clear()`, `coins.h:53-55`, and the doc at `coins.h:30-31`). When persisted, the height and
coinbase flag are not stored as separate fields — they are packed into one VARINT as
`(nHeight << 1) | fCoinBase` (`coins.h:66-67`, format documented at `coins.h:29`), with the
`CTxOut` serialized afterwards via `TxOutCompression`. So `Coin` is the value payload plus two
provenance bits, with spentness derived from the payload's nullness rather than a dedicated flag.
