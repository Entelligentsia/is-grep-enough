## Required spine (gradeable core ‚Äî must hit all)

1. **The routine is `CheckTransaction(tx, state)`** ‚Äî def `consensus/tx_check.cpp:11`
   (decl `consensus/tx_check.h:18`). It performs the **context-free** transaction sanity checks
   (no UTXO set, no chain tip consulted) and reports failure through `TxValidationState` with
   result `TxValidationResult::TX_CONSENSUS`.
2. **What it actually checks (so the caller's "invalid" is concrete):** empty `vin`/`vout`
   (`bad-txns-vin-empty`, `bad-txns-vout-empty`, `tx_check.cpp:14-16`); serialized size within
   block weight (`bad-txns-oversize`, `tx_check.cpp:18-21`); non-negative, in-range, non-overflowing
   output values (`bad-txns-vout-negative`/`-toolarge`/`txouttotal-toolarge`, `tx_check.cpp:24-34`);
   no duplicate inputs (`bad-txns-inputs-duplicate`, `tx_check.cpp:39-42`, CVE-2018-17144);
   coinbase `scriptSig` length 2..100 (`bad-cb-length`, `tx_check.cpp:45-47`); non-coinbase inputs
   have non-null prevout (`bad-txns-prevout-null`, `tx_check.cpp:49-52`).
3. **Caller A ‚Äî mempool acceptance:** `MemPoolAccept::PreChecks` (`validation.cpp:786`) calls
   `if (!CheckTransaction(tx, state)) { return false; }` at `validation.cpp:802-803` ("state
   filled in by CheckTransaction"). A failure rejects the **loose transaction** from the mempool;
   the immediately following lines then special-case coinbase (`validation.cpp:805-807`) and
   standardness (`validation.cpp:812`), so `CheckTransaction` is the first gate on the ATMP path.
4. **Caller B ‚Äî block checking:** `CheckBlock` (`validation.cpp:3925`) loops over `block.vtx`
   and calls `if (!CheckTransaction(*tx, tx_state))` at `validation.cpp:3968`; on failure it
   **rejects the whole block**, translating the tx-level state into a block-level
   `BlockValidationResult::BLOCK_CONSENSUS` rejection carrying the tx's reject reason
   (`validation.cpp:3970-3972`, comment "CheckBlock() does context-free validation checks"). The
   CVE-2018-17144 note at `validation.cpp:3966` flags duplicate-input checking as the load-bearing
   reason this is called per-tx in the block path.
5. **The synthesis ‚Äî same bool+state, different consequence:** both callers consume the same
   `TxValidationState`/bool contract, but `PreChecks` maps a failure to a single rejected
   transaction (`TX_CONSENSUS`, loose-tx reject) whereas `CheckBlock` maps it to a rejected
   **block** (`BLOCK_CONSENSUS`) ‚Äî the gradeable point is that the routine is context-free and
   shared, while the *unit of rejection* (one tx vs the whole block) differs per caller.

- Naming `CheckInputs`/`CheckInputScripts` (`validation.cpp:148`/`:2067`) as the focal routine ‚Äî
  those are the UTXO-dependent, *contextual* checks; `CheckTransaction` is the context-free one
  the prompt describes ("before any chain context is consulted").
- Stating the callers but not the per-caller consequence (loose-tx reject vs block reject) ‚Äî

- Claiming `CheckTransaction` rejects the block directly ‚Äî it does not; it fills `state`, and

   layer run after `CheckTransaction` in `PreChecks` (`validation.cpp:812`), not the

## Model answer

The context-free sanity check is `CheckTransaction(tx, state)` (`consensus/tx_check.cpp:11`). It
consults neither the UTXO set nor the chain ‚Äî it verifies the transaction's own structure: non-
empty `vin`/`vout`, weight within block limits, non-negative/non-overflowing output values, no
duplicate inputs (CVE-2018-17144), valid coinbase `scriptSig` length, and non-null prevouts on
non-coinbase inputs ‚Äî flagging any failure as `TxValidationResult::TX_CONSENSUS` in the
`TxValidationState` it fills. Two validation paths call it. During **mempool acceptance**,
`MemPoolAccept::PreChecks` (`validation.cpp:786`) runs `if (!CheckTransaction(tx, state)) return
false;` at `validation.cpp:802` as the first gate; a failure rejects that one **loose
transaction** from entering the mempool (with coinbase/standardness handled in the lines that
follow at `validation.cpp:805-812`). During **block checking**, `CheckBlock`
(`validation.cpp:3925`) iterates `block.vtx` and calls `CheckTransaction(*tx, tx_state)` at
`validation.cpp:3968`; a failure is translated into a `BlockValidationResult::BLOCK_CONSENSUS`
rejection of the **entire block**, carrying the tx's reject reason (`validation.cpp:3970-3972`).
So the routine is shared and context-free, but its callers differ in what a "fail" means: one
dropped transaction (mempool) versus a rejected block (validation).
