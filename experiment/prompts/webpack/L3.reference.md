# L3 reference answer key — one entry dependency → built, graph-wired module

Pinned source: `experiment/repos/webpack` (SHA ff19f174…). All cites re-verified.

A single ordered chain across `lib/EntryPlugin.js`, `lib/Compilation.js`, and
`lib/NormalModuleFactory.js` plus `lib/NormalModule.js`. Order is gradeable.

## Required spine (gradeable core — must hit all, in order)

1. **Make-phase entry: `EntryPlugin` taps `compiler.hooks.make` and calls
   `compilation.addEntry`.** — `lib/EntryPlugin.js:47` (`compiler.hooks.make.tapAsync(
   PLUGIN_NAME, (compilation, callback) => {`), with the `addEntry` call at
   `lib/EntryPlugin.js:48` (`compilation.addEntry(context, dep, options, (err) => {
   callback(err); });`). The entry is an `EntryDependency` built by
   `EntryPlugin.createDependency` (`lib/EntryPlugin.js:58`), and
   `EntryDependency → normalModuleFactory` was registered on the compilation at
   `lib/EntryPlugin.js:34-44`.
2. **`addEntry` → `_addEntryItem` → `addModuleTree`** — `addEntry`
   (`lib/Compilation.js:2645`) → `_addEntryItem` (`lib/Compilation.js:2682`, call at
   `2652`) → `this.addModuleTree({ context, dependency: entry, … })`
   (`lib/Compilation.js:2730`). `_addEntryItem` records the entry and fires
   `hooks.addEntry`; `addModuleTree` is where the factory is chosen.
3. **`addModuleTree` picks the factory and calls `handleModuleCreation`** —
   `addModuleTree` (`lib/Compilation.js:2591`) gets the factory with
   `this.dependencyFactories.get(Dep)` (`lib/Compilation.js:2604`) — for an entry
   this resolves to the `NormalModuleFactory` — then calls
   `this.handleModuleCreation({ factory: moduleFactory, dependencies: [dependency],
   originModule: null, … })` (`lib/Compilation.js:2613`).
4. **`handleModuleCreation` → `factorizeModule` → `_factorizeModule` →
   `factory.create`** — `handleModuleCreation` (`lib/Compilation.js:2298`) calls
   `this.factorizeModule({ factory, dependencies, factoryResult: true, … })`
   (`lib/Compilation.js:2315`), which enqueues on `this.factorizeQueue`
   (`lib/Compilation.js:2272`); the queue processor `_factorizeModule`
   (`lib/Compilation.js:2171`) calls `factory.create({ contextInfo, resolveOptions,
   context, dependencies })` (`lib/Compilation.js:2190`).
5. **`NormalModuleFactory.create` resolves and instantiates the module** —
   `create(data, callback)` (`lib/NormalModuleFactory.js:1011`) builds the
   `resolveData`, then runs the hook waterfall `beforeResolve`
   (`lib/NormalModuleFactory.js:1052`) → `factorize` (called at
   `lib/NormalModuleFactory.js:1090`; the `factorize` tap at
   `lib/NormalModuleFactory.js:469` drives `resolve` → `afterResolve` →
   `createModule`) → `createModule` (`lib/NormalModuleFactory.js:512`) →
   `createModuleClass`/`new NormalModule(createData)` (`lib/NormalModuleFactory.js:528`).
   The resolver tap itself is `this.hooks.resolve.tapAsync({stage:100}, …)` at
   `lib/NormalModuleFactory.js:546`. The factory returns a `{ module,
   fileDependencies, … }` result back to `_factorizeModule`'s callback
   (`lib/NormalModuleFactory.js:1093`).
6. **`handleModuleCreation` adds the module to the compilation and connects it in
   the graph** — back in `handleModuleCreation`, `this.addModule(newModule, …)`
   (`lib/Compilation.js:2362`) (def `lib/Compilation.js:1585`) inserts it, then for
   each dependency `this.moduleGraph.setResolvedModule(originModule, dependency,
   module)` (`lib/Compilation.js:2404`, or the `unsafeCache` branch at
   `lib/Compilation.js:2388`) wires the edge from the entry dependency to the
   resolved module.
7. **`buildModule` → `module.build` (loaders + parse)** — `handleModuleCreation`
   then calls `this.buildModule(module, …)` (`lib/Compilation.js:2500`); `buildModule`
   (`lib/Compilation.js:1660`) enqueues on `this.buildQueue`
   (`lib/Compilation.js:1661`) whose processor `_buildModule`
   (`lib/Compilation.js:1670`) calls `module.build(options, compilation, resolver, fs,
   cb)` (`lib/Compilation.js:1697`). For a normal module that is
   `NormalModule.prototype.build` (`lib/NormalModule.js:1779`), which initializes
   `buildMeta` (`lib/NormalModule.js:1788`) / `buildInfo` (`lib/NormalModule.js:1794`) and calls `_doBuild`
   (`lib/NormalModule.js:1820`). `_doBuild` (`lib/NormalModule.js:1468`) runs the
   loaders via `runLoaders({ resource, loaders, … })` (`lib/NormalModule.js:1567`)
   and then parses the result with `this.parser.parse(this._ast || source, {…})`
   (`lib/NormalModule.js:1975`).
8. **Outgoing dependencies are queued for the same treatment** — after the build,
   `processModuleDependencies(module, …)` (`lib/Compilation.js:1745`, enqueuing on
   `this.processDependenciesQueue` at `lib/Compilation.js:1746`) walks the module's
   `dependencies`/`blocks` (`_processModuleDependencies` `lib/Compilation.js:1780`,
   the `processDependenciesBlock` walk at `lib/Compilation.js:1759`), and for each
   unresolved dependency it calls `this.handleModuleCreation(item, …)` again
   (`lib/Compilation.js:1832`) — which is what recursively builds the rest of the
   module graph.

## Acceptable extras (credit, never required)

- The queue hierarchy: `buildQueue` has `parent: this.factorizeQueue`
  (`lib/Compilation.js:1236`) and `processDependenciesQueue` is the parent of the
  factorize/build queues (`lib/Compilation.js:1215-1236`) — noting the
  queue-parent/child relationship explains the parallelism/scheduling. Plus, never
  required.
- The `factoryResult: true` path that also returns `fileDependencies` /
  `contextDependencies` / `missingDependencies` to the compilation
  (`lib/Compilation.js:2215-2222`) — credit.
- The unsafe-cache fast path in `handleModuleCreation`
  (`lib/Compilation.js:2370-2404`) that can `restoreFromUnsafeCache` and skip a
  rebuild. Mentioning it is a plus; never required.
- The parser hook (`NormalModule.getCompilationHooks`, the `parser.parse` options
  at `lib/NormalModule.js:1976-1985`) — noting the parser is what creates the
  module's dependency edges during build. Credit.
- The `moduleGraph.setIssuerIfUnset(module, originModule)` step
  (`lib/Compilation.js:2412`) recording who caused the module to be reached.

## Common misses / wrong turns

- Stopping at `addEntry` or `addModuleTree` and never reaching the factory/build
  (skips spine 4-7) — the prompt asks "until it has become a fully built module
  wired into the module graph with its own outgoing dependencies queued."
- Confusing the factory hook order (`beforeResolve → factorize → resolve →
  afterResolve → createModule → module`). The live dispatch order is `create`
  calls `beforeResolve` (`NormalModuleFactory.js:1052`), which calls `factorize`
  (`1090`); the `factorize` *tap* (`469`) then calls `resolve` (`475`), then
  `afterResolve` (`493`), then `createModule` (`512`).
- Naming `processModuleDependencies` as a *separate* path rather than the recursive
  re-entry into `handleModuleCreation` (`Compilation.js:1832`). The closing of the
  loop (deps → same treatment) is the terminal gradeable point.
- Citing `addModuleChain` (`Compilation.js:2578`) instead of `addModuleTree`
  (`2591`) — `addModuleChain` just delegates to `addModuleTree`; the entry path
  goes through `addModuleTree`.
- Treating it as a star of callers of `build` rather than an ordered chain.

## Completeness scale

- **Full:** all 8 spine stops, in order — make/addEntry → addModuleTree →
  handleModuleCreation → factorizeModule/factory.create → resolve+new NormalModule
  → addModule+setResolvedModule → buildModule/`module.build` (runLoaders+parse) →
  processModuleDependencies (recursion) — with cites; the terminal recursion into
  `handleModuleCreation` named.
- **Partial:** ≥5 of 8 in roughly correct order, but missing either the factory
  resolution half (spine 4-5) OR the build half (spine 6-7) OR the closing
  dependency-queuing step (spine 8).
- **Miss:** only the `addEntry`/make front end, or only the `module.build` internals
  without the connecting chain, or an out-of-order/incorrect chain.

## Model answer

When the compiler's `make` hook fires, `EntryPlugin`'s tap
(`lib/EntryPlugin.js:47`) calls `compilation.addEntry(context, dep, options, cb)`
(`lib/EntryPlugin.js:48`), handing in an `EntryDependency` (`EntryPlugin.js:58`).
`addEntry` (`lib/Compilation.js:2645`) → `_addEntryItem` (`2682`) records the entry,
fires `hooks.addEntry`, and calls `addModuleTree` (`lib/Compilation.js:2730`).
`addModuleTree` (`lib/Compilation.js:2591`) resolves the factory via
`dependencyFactories.get(EntryDependency)` → the `NormalModuleFactory`
(`Compilation.js:2604`) and calls `handleModuleCreation({factory, dependencies,
originModule: null, …})` (`Compilation.js:2613`). `handleModuleCreation`
(`Compilation.js:2298`) calls `factorizeModule` (`Compilation.js:2315`), which
queues onto `factorizeQueue` (`Compilation.js:2272`); the queue's
`_factorizeModule` (`Compilation.js:2171`) calls `factory.create({contextInfo,
context, dependencies, …})` (`Compilation.js:2190`). Inside
`NormalModuleFactory.create` (`lib/NormalModuleFactory.js:1011`) the hook waterfall
runs: `beforeResolve` (`1052`) → `factorize` (`1090`, whose tap at `469` drives
`resolve` `475` → `afterResolve` `493` → `createModule` `512`) → `createModule`
→ `new NormalModule(createData)` (`NormalModuleFactory.js:528`). Back in
`handleModuleCreation`, `addModule(newModule, …)` (`Compilation.js:2362`) inserts
the module and `moduleGraph.setResolvedModule(originModule, dependency, module)`
(`Compilation.js:2404`) wires the edge. Then `buildModule(module, …)`
(`Compilation.js:2500`); `buildModule` (`1660`) queues on `buildQueue` (`1661`)
whose `_buildModule` (`1670`) calls `module.build` — for a normal module,
`NormalModule.build` (`lib/NormalModule.js:1779`) sets up `buildMeta`/`buildInfo`
(`1788`/`1794`) and calls `_doBuild` (`1820`), which `runLoaders` (`1567`) and then
`this.parser.parse` (`1975`). Finally `processModuleDependencies(module, …)`
(`Compilation.js:1745`) walks the module's `dependencies`/`blocks`
(`_processDependenciesBlock` at `1759`) and, for each unresolved dependency, calls
`handleModuleCreation` again (`Compilation.js:1832`) — so the new module's outgoing
dependencies get the same factorize→build→wire treatment, recursively building the
graph.
