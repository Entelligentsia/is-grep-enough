# L1 reference answer key — the in-memory module node (`Module`)

Pinned source: `experiment/repos/webpack` (SHA ff19f174…). All cites re-verified.

The entity is the `Module` class in `lib/Module.js`. It is the universal in-memory
object standing for one module during a compilation, and it is also a node in the
module graph (its edges come from the base class).

## Required spine (gradeable core — must hit all)

1. **The node is `class Module`** — `lib/Module.js:243` (`class Module extends
   DependenciesBlock`). Subclasses (`NormalModule`, `ContextModule`, …) extend it;
   the base carries the graph-independent module identity.
2. **`type` = the module kind** — `lib/Module.js:254` (`this.type = type;`), typed
   `ModuleTypes`. This is the field that names *what kind* of module it is
   (e.g. `"javascript/esm"`, `"javascript/dynamic"`, `"css"`, …). (Constructor
   signature at `lib/Module.js:250`.)
3. **`context` / `layer` = origin context and layer** — `lib/Module.js:256`
   (`this.context = context;`) and `lib/Module.js:258` (`this.layer = layer;`).
4. **`buildMeta` + `buildInfo` = what the build step leaves behind** —
   `lib/Module.js:285` (`this.buildMeta = undefined;`) and `lib/Module.js:287`
   (`this.buildInfo = undefined;`). `buildMeta` is the public, cache-affecting
   result (exports type, strictness, …); `buildInfo` is the private per-build
   bookkeeping (cacheable, fileDependencies, hash, snapshot, …). A correct answer
   must say both are populated *by the build*, and distinguish meta (public/cache)
   from info (private).
5. **Edges live on the base class `DependenciesBlock`** — `Module extends
   DependenciesBlock` (`lib/Module.js:243`); the base supplies `this.dependencies`
   (`lib/DependenciesBlock.js:30`, `[]`), `this.blocks`
   (`lib/DependenciesBlock.js:32`, `[]` — the `AsyncDependenciesBlock` async
   boundaries), and `this.parent` (`lib/DependenciesBlock.js:34`). The header
   comment at `lib/DependenciesBlock.js:20-25` (class def `:27`) states
   `DependenciesBlock` is the base class for all `Module` classes and describes
   `dependencies` as edges ("an 'edge' pointing to another 'node' on module
   graph", `DependenciesBlock.js:58`).
   A correct answer must locate the dependency edges on the *base*, not on `Module`
   directly.

## Common misses / wrong turns

- Pointing at `NormalModule` / a specific module subclass as "the" module object,
  instead of the universal `Module` base. The prompt asks for the *kind* of object
  that is one module node; `NormalModule` is one specialization.
- Putting the dependency edges on `Module` itself. They are inherited from
  `DependenciesBlock`; missing the base class misses spine element 5.
- Conflating `buildMeta` and `buildInfo` (treating them as one bag). The prompt
  asks what the object "carries away from its build"; the meta-vs-info distinction
  is the central gradeable point.
- Describing `Compilation`/`ChunkGraph` bookkeeping (where modules are *held*) as
  if it were the module object itself. Those are containers; `Module` is the node.

## Completeness scale

- **Full:** identifies `class Module extends DependenciesBlock` and explains all 5

  as the build's output (with the meta-vs-info distinction), and the
  `dependencies`/`blocks`/`parent` edges on `DependenciesBlock` — with cites.
- **Partial:** names `Module` and ≥3 of 5 elements but blurs `buildMeta` vs

- **Miss:** points at a subclass or at `Compilation`/`ChunkGraph` instead of the

## Model answer

A single module during a webpack compilation is a `Module` instance — `class Module
extends DependenciesBlock` (`lib/Module.js:243`). Its `type` field
(`lib/Module.js:254`) records what kind of module it is (`"javascript/esm"`,
`"css"`, …), and `context` (`lib/Module.js:256`) / `layer` (`lib/Module.js:258`)
record its resolving context and layer. The build step writes its results into two
fields: `buildMeta` (`lib/Module.js:285`) — the public, cache-relevant facts such
as exports type and strictness — and `buildInfo` (`lib/Module.js:287`) — the
private per-build bookkeeping such as `cacheable`, `fileDependencies`, `snapshot`,
and `hash`; so `buildMeta` is the stable summary and `buildInfo` is the
instance-local detail. The module's edges to other modules are not on `Module`
itself but on its base class `DependenciesBlock` (`lib/DependenciesBlock.js:20-25`,
class def `:27`): `this.dependencies` (`DependenciesBlock.js:30`) is the list of
dependency edges to other modules, `this.blocks` (`DependenciesBlock.js:32`) holds
`AsyncDependenciesBlock` async boundaries (the seeds of code-splitting), and
`this.parent` (`DependenciesBlock.js:34`) links a block back to its enclosing
block. So the node carries its identity (`type`/`context`), the build's output
(`buildMeta`/`buildInfo`), and — via the base — the graph edges
(`dependencies`/`blocks`/`parent`).
