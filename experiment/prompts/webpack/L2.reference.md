# L2 reference answer key — the entry-seeding routine and its callers

Pinned source: `experiment/repos/webpack` (SHA ff19f174…). All cites re-verified.

The central routine is `Compilation.prototype.addEntry`. It is the single funnel
through which a build's *entry* dependencies enter a compilation during the make
phase; everything reached transitively from an entry comes in through the
dependency-processing path, not through `addEntry`. The question is the routine
plus the plugin callers that feed it.

## Required spine (gradeable core — must hit all)

1. **The routine: `addEntry(context, entry, optionsOrName, callback)`** —
   `lib/Compilation.js:2645`. It normalizes the options (`lib/Compilation.js:2647`)
   and delegates to `_addEntryItem(context, entry, "dependencies", options, callback)`
   (`lib/Compilation.js:2652`). `_addEntryItem` (`lib/Compilation.js:2682`) records
   the dependency in `this.entries`/`this.globalEntry` (`lib/Compilation.js:2684`-
   `2726`), fires `this.hooks.addEntry.call(entry, options)` (`lib/Compilation.js:2728`), then calls `this.addModuleTree({context, dependency: entry, …})`
   (`lib/Compilation.js:2730`) — which is the bridge into the module-creation path.
   A correct answer names `addEntry` as the funnel and notes its
   `addModuleTree`/`handleModuleCreation` continuation is the *downstream* side, not
   another caller.
2. **It is driven from the compiler's `make` hook.** Each caller below taps
   `compiler.hooks.make` (or a make-phase equivalent) and, inside that tap, calls
   `compilation.addEntry(...)`. The canonical caller is `EntryPlugin`, which taps
   `compiler.hooks.make.tapAsync(PLUGIN_NAME, (compilation, callback) => {
   compilation.addEntry(context, dep, options, (err) => callback(err)); })` —
   `lib/EntryPlugin.js:47-50` (the `addEntry` call at `lib/EntryPlugin.js:48`). It
   also wires `EntryDependency → normalModuleFactory` on the compilation
   (`lib/EntryPlugin.js:34-44`) and builds the `EntryDependency` via
   `EntryPlugin.createDependency` (`lib/EntryPlugin.js:23`/`58`).
3. **Dynamic / runtime-computed entries: `DynamicEntryPlugin`** —
   `lib/DynamicEntryPlugin.js:74` calls `compilation.addEntry(this.context,
   EntryPlugin.createDependency(entry, options), …)` inside a per-import
   `Promise`/`new Promise` block (`lib/DynamicEntryPlugin.js:72-78`). The entry
   string is obtained at build time (an async `import()` of an entry-defining module
   or a function), so multiple entries can be added as they resolve. This contrasts
   with `EntryPlugin`'s static, construction-time entry string.
4. **DLL entries: `DllEntryPlugin`** — `lib/dll/DllEntryPlugin.js:54` calls
   `compilation.addEntry(this.context, new DllEntryDependency(…), …)` from a
   `compiler.hooks.make.tapAsync(PLUGIN_NAME, …)` (`lib/dll/DllEntryPlugin.js:52`).
   The dependency is a `DllEntryDependency` (not an `EntryDependency`), mapping to a
   DLL-specific factory.
5. **Module-federation / container entries: `ContainerPlugin`** —
   `lib/container/ContainerPlugin.js:89` calls
   `compilation.addEntry(compilation.options.context, dep, …)`, where `dep` is a
   container dependency (`ContainerEntryDependency`) built from the federated
   exposes/remotes configuration (the `dep.loc = { name: this.options.name }` line
   at `lib/container/ContainerPlugin.js:87`).
6. **Auxiliary entries: `HtmlModulesPlugin`** — `lib/html/HtmlModulesPlugin.js:246`
   calls `compilation.addEntry(context, EntryPlugin.createDependency(entry.request,
   {…}), …)` inside a `new Promise` (`lib/html/HtmlModulesPlugin.js:244-248`) to
   feed HTML-referenced modules as additional entries.

## Acceptable extras (credit, never required)

- The `_addEntryItem` "include" variant: `addInclude` (`lib/Compilation.js:2663`)
  routes through `_addEntryItem(..., "includeDependencies", …)` — same routine,
  different target bag. Mentioning it is a plus.
- The `dependencyFactories` map: `addModuleTree`/`addModuleChain` resolve the
  factory from `this.dependencyFactories.get(Dep)` (`lib/Compilation.js:2604`); the
  `EntryPlugin` registration of `EntryDependency → normalModuleFactory`
  (`lib/EntryPlugin.js:34-44`) is what makes a normal entry resolve through the
  `NormalModuleFactory`. This is the hinge between L2 and L3; credit, not required.
- The `hooks.addEntry` / `hooks.addEntry`SyncHook declaration
  (`lib/Compilation.js:860`) — noting the routine announces itself via that hook.
- Other `addEntry` callers in less-central plugins if verifiable; the five above are
  the load-bearing ones.

## Common misses / wrong turns

- Listing callers of `addModuleChain` / `handleModuleCreation` / `factorizeModule`
  as if they were callers of `addEntry`. Those are the *downstream* continuation;
  the prompt is about who seeds entries, i.e. callers of `addEntry`.
- Treating the dependency-processing recursion (`processModuleDependencies`,
  `lib/Compilation.js:1745`) as an entry source. That reaches *transitive*
  dependencies, not entries — the prompt explicitly contrasts "entry points get
  seeded" with "modules reached later are discovered."
- Naming only `EntryPlugin` and missing the dynamic/DLL/container/html variants —
  that under-counts the spine (≥4 of the 5 callers needed for full).
- Confusing `addEntry` with `addInclude` as different routines; they share
  `_addEntryItem`.
- Pointing at `EntryOptionPlugin` (`lib/EntryOptionPlugin.js`) which *applies* the
  entry plugins rather than calling `addEntry` itself — it is one hop further out
  than the spine wants.

## Completeness scale

- **Full:** identifies `addEntry` (`Compilation.js:2645`) as the funnel, explains

- **Partial:** names `addEntry` and 2-3 callers, or gets the routine but blurs the

- **Miss:** points at the make/build chain (L3 territory) without identifying

## Model answer

The routine that seeds a build's entry points is `Compilation.prototype.addEntry`
(`lib/Compilation.js:2645`). It normalizes the options and calls `_addEntryItem`
(`lib/Compilation.js:2682`), which records the entry dependency in
`this.entries`/`this.globalEntry`, fires `hooks.addEntry`, then continues
downstream via `addModuleTree` (`lib/Compilation.js:2730`) into the module-creation
path — so `addEntry` is the *funnel*, and everything reached transitively from an
entry arrives through dependency processing, not through `addEntry` again. The
callers are plugins that tap the compiler's `make` hook and, inside it, hand an
entry dependency to `compilation.addEntry`. The canonical one is `EntryPlugin`
(`lib/EntryPlugin.js:48`, from its `make` tap at `lib/EntryPlugin.js:47`), which
also registers `EntryDependency → normalModuleFactory` (`lib/EntryPlugin.js:34-44`)
and builds the dependency via `createDependency` (`lib/EntryPlugin.js:58`).
`DynamicEntryPlugin` (`lib/DynamicEntryPlugin.js:74`) feeds runtime-computed
entries, resolving the entry string at build time inside a `Promise` and adding
each via `EntryPlugin.createDependency`. `DllEntryPlugin`
(`lib/dll/DllEntryPlugin.js:54`) feeds `DllEntryDependency` entries for DLL builds.
`ContainerPlugin` (`lib/container/ContainerPlugin.js:89`) feeds
`ContainerEntryDependency` entries for module-federation exposes. `HtmlModulesPlugin`
(`lib/html/HtmlModulesPlugin.js:246`) feeds HTML-referenced modules as extra
entries. So one routine, `addEntry`, is reached by several entry-producing plugins,
each supplying a different kind of entry dependency.
