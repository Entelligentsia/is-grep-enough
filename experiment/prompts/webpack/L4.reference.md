# L4 reference answer key — module graph → chunk graph (sealing / code-splitting)

Pinned source: `experiment/repos/webpack` (SHA ff19f174…). All cites re-verified.

A bounded subsystem: how `Compilation.seal` and `buildChunkGraph` cooperate to turn
the finished module graph into a chunk graph — entrypoints/chunk-groups/chunks
created and connected, and async boundaries (`AsyncDependenciesBlock`) split off
into their own chunks based on which modules are already available in parent
chunks. Several paths, one cohesive area.

## Required spine (gradeable core — must hit all)

1. **Entry to the subsystem: `Compilation.seal`** — `lib/Compilation.js:3473`
   (`seal(callback) {`). It clears the make-phase queues (`Compilation.js:3480-3484`),
   fires `this.hooks.seal.call()` (`Compilation.js:3496`), then runs the
   optimize-dependencies / before-chunks preamble before constructing the chunks
   (`Compilation.js:3498-3506`, `hooks.beforeChunks.call()` at `Compilation.js:3506`).
2. **Entrypoints and chunks are created and connected to entry modules.** In the
   `for (const [name, {dependencies, …}] of this.entries)` loop
   (`lib/Compilation.js:3510-3511`): `const chunk = this.addChunk(name)`
   (`Compilation.js:3512`), `const entrypoint = new Entrypoint(options)`
   (`Compilation.js:3516`), `entrypoint.setRuntimeChunk(chunk)` when there is no
   `dependOn`/`runtime` (`Compilation.js:3518`), `entrypoint.setEntrypointChunk(chunk)`
   (`Compilation.js:3520`), the entrypoint is registered in `this.entrypoints` /
   `this.chunkGroups` (`Compilation.js:3523-3525`), and each entry module is wired with
   `this.chunkGraph.connectChunkAndEntryModule(chunk, module, entrypoint)`
   (`Compilation.js:3541`). The `chunkGraphInit` map (`Compilation.js:3521`,
   populated at `Compilation.js:3546-3550`) is the input handed to `buildChunkGraph`.
   `Entrypoint extends ChunkGroup` (`lib/Entrypoint.js:22`); `setRuntimeChunk` at
   `lib/Entrypoint.js:60`, `setEntrypointChunk` at `lib/Entrypoint.js:81`.
3. **`buildChunkGraph` is the core traversal** — invoked as
   `buildChunkGraph(this, chunkGraphInit)` (`lib/Compilation.js:3656`); defined as
   `const buildChunkGraph = (compilation, inputEntrypointsAndModules) => { … }`
   (`lib/buildChunkGraph.js:1425`, exported at `lib/buildChunkGraph.js:1484`). Its
   engine is `visitModules(logger, compilation, inputEntrypointsAndModules,
   chunkGroupInfoMap, blockConnections, …)` (`lib/buildChunkGraph.js:294`).
4. **The walk distinguishes entry/sync blocks from async blocks.**
   - `processEntryBlock(block)` (`lib/buildChunkGraph.js:886`) handles blocks reached
     from an entrypoint — referenced modules are enqueued with
     `ADD_AND_ENTER_ENTRY_MODULE` / `PROCESS_BLOCK` actions
     (`lib/buildChunkGraph.js:906-910`), keeping them in the entry chunk.
   - `processBlock(block)` (`lib/buildChunkGraph.js:791`) handles ordinary
     (synchronous) dependency blocks, adding their modules to the *current* chunk
     (same chunk group).
   - `iteratorBlock(b)` (`lib/buildChunkGraph.js:564`) is what, when it meets an
     `AsyncDependenciesBlock`, creates a *new* `ChunkGroup` + `Chunk` for the async
     boundary and records the `blockConnections` edge — this is where code-splitting
     actually happens. (The async block type itself is `class AsyncDependenciesBlock
     extends DependenciesBlock`, `lib/AsyncDependenciesBlock.js:21`, constructor
     `lib/AsyncDependenciesBlock.js:27`; these are the `this.blocks` entries on a
     module from L1.)
   The dispatcher is `processQueue()` (`lib/buildChunkGraph.js:922`) which pops
   queue items and dispatches by `action`.
5. **Available-modules tracking decides what a new async chunk must contain.**
   `calculateResultingAvailableModules(chunkGroupInfo)` (`lib/buildChunkGraph.js:1014`)
   computes the set of modules already available in a chunk group (a bitmask of
   module ordinals, `getModuleOrdinal` at `lib/buildChunkGraph.js:322`), and
   `processConnectQueue()` (`lib/buildChunkGraph.js:1033`) reconciles chunk-group
   connections, so a module already present in a parent chunk is *not* duplicated
   into a child async chunk. The `ChunkGroupInfo` typedef
   (`lib/buildChunkGraph.js:40-51`) holds `minAvailableModules`,
   `availableModulesToBeMerged`, `resultingAvailableModules`, `children`,
   `availableSources`, `availableChildren` — the bookkeeping that lets a split
   chunk omit modules its parents already ship.
6. **The chunk graph itself is `ChunkGraph`** — `class ChunkGraph` at
   `lib/ChunkGraph.js:281` (with per-module `ChunkGraphModule` `lib/ChunkGraph.js:226`
   and per-chunk `ChunkGraphChunk` `lib/ChunkGraph.js:250`). It records which modules
   sit in which chunks: `connectChunkAndModule(chunk, module)` (`lib/ChunkGraph.js:392`)
   and `connectChunkAndEntryModule(chunk, module, entrypoint)`
   (`lib/ChunkGraph.js:1173`). After `buildChunkGraph` returns, `seal` fires
   `this.hooks.afterChunks.call(this.chunks)` (`lib/Compilation.js:3657`) — the
   subsystem boundary.

## Common misses / wrong turns

- Treating "seal" as just the optimize/codegen/emit tail and skipping
  `buildChunkGraph` entirely — that misses the spine (the chunk graph is *built* in
  `buildChunkGraph`, not in the optimize stages).
- Saying chunks are created in `createChunkAssets`. They are created in the seal
  entry loop (`addChunk` at `Compilation.js:3512`) and the `buildChunkGraph` async
  walk; `createChunkAssets` only *renders* already-decided chunks to assets.
- Missing the async-split mechanism — i.e. describing entry/sync placement but not
  that `AsyncDependenciesBlock` (`iteratorBlock`, `buildChunkGraph.js:564`) spawns a
  new chunk group. The prompt explicitly asks how dynamic imports split off.
- Missing the available-modules tracking (`calculateResultingAvailableModules`
  `buildChunkGraph.js:1014` / `processConnectQueue` `1033`) that prevents
  duplicating a module already in a parent chunk — the prompt asks how "which
  modules are already available in a parent chunk decides what goes into a newly
  split async chunk."
- Conflating the module graph (`ModuleGraph`, edges = dependencies) with the chunk
  graph (`ChunkGraph`, edges = chunk-group parent/child, modules-per-chunk). They
  are distinct graphs; the subsystem converts one into the other.

## Completeness scale

- **Full:** all 6 spine elements — `seal` entry, entrypoint/chunk creation +
  `connectChunkAndEntryModule`, `buildChunkGraph`/`visitModules` core, the
  entry-vs-async block distinction (`processEntryBlock`/`processBlock`/`iteratorBlock`
  with `AsyncDependenciesBlock`), available-modules tracking
  (`calculateResultingAvailableModules`/`processConnectQueue`), and `ChunkGraph` as
  the result — with cites, ending at `hooks.afterChunks`.
- **Partial:** ≥4 of 6, but missing the async-split mechanism OR the
  available-modules tracking OR confusing module graph with chunk graph.
- **Miss:** only the seal entry loop (spine 2) without `buildChunkGraph`, or only
  the optimize/asset tail, or a description that never names `buildChunkGraph` /
  `ChunkGraph`.

## Model answer

During sealing `Compilation.seal` (`lib/Compilation.js:3473`) fires `hooks.seal`
(`3496`) and `hooks.beforeChunks` (`3510`), then for each entry creates a chunk
(`this.addChunk(name)`, `Compilation.js:3512`) and an `Entrypoint` (`new
Entrypoint(options)`, `Compilation.js:3516`; `Entrypoint extends ChunkGroup`,
`lib/Entrypoint.js:22`), sets its runtime/entrypoint chunks
(`Compilation.js:3518,3520`), and connects each entry module with
`this.chunkGraph.connectChunkAndEntryModule(chunk, module, entrypoint)`
(`Compilation.js:3541`; def `lib/ChunkGraph.js:1173`). It collects these into
`chunkGraphInit` and passes it to `buildChunkGraph(this, chunkGraphInit)`
(`Compilation.js:3656`). `buildChunkGraph` (`lib/buildChunkGraph.js:1425`) runs
`visitModules` (`buildChunkGraph.js:294`), driven by a queue
(`processQueue` `922`). Entry/sync blocks are handled by `processEntryBlock`
(`886`) and `processBlock` (`791`), which add their referenced modules to the
*current* chunk. When the walk meets an `AsyncDependenciesBlock`
(`lib/AsyncDependenciesBlock.js:21` — the `this.blocks` entries a module carries
from L1), `iteratorBlock` (`buildChunkGraph.js:564`) creates a *new* chunk group and
chunk for that async boundary and records the parent→child connection — the actual
code-split. Whether a module is duplicated into a child async chunk is governed by
available-modules tracking: `calculateResultingAvailableModules`
(`buildChunkGraph.js:1014`) and `processConnectQueue` (`buildChunkGraph.js:1033`)
maintain, per chunk group, the bitmask of modules already available in its parents
(`ChunkGroupInfo.minAvailableModules`/`resultingAvailableModules`,
`buildChunkGraph.js:40-51`; `getModuleOrdinal` `322`), so a module a parent already
ships is not re-added. The result is recorded in `ChunkGraph`
(`class ChunkGraph` `lib/ChunkGraph.js:281`) via `connectChunkAndModule`
(`ChunkGraph.js:392`) and `connectChunkAndEntryModule` (`ChunkGraph.js:1173`). When
`buildChunkGraph` returns, `seal` fires `hooks.afterChunks` (`Compilation.js:3657`)
— the module graph has been converted into a connected chunk graph, with sync
dependencies inside their entry chunk and dynamic-import boundaries split into
their own chunk groups.
