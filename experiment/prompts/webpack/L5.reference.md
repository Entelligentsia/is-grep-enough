# L5 reference answer key — one build run, end to end

Pinned source: `experiment/repos/webpack` (SHA ff19f174…). All cites re-verified.

Whole-system: the full journey of one `compiler.run` invocation, threading the
`Compiler` lifecycle, the `Compilation` make/finish/seal phases, module
graph → chunk graph → code generation → hashing → assets, and finally emission to
disk. Multiple subsystems, end to end.

## Required spine (gradeable core — must hit all, in order)

1. **`Compiler.run` — run lifecycle hooks, then `compile`.** `run(callback)`
   (`lib/Compiler.js:539`) guards against concurrent runs, builds the `onCompiled`
   callback, then `run()` (inner) calls `this.hooks.beforeRun.callAsync(this, …)`
   (`lib/Compiler.js:651`) → `this.hooks.run.callAsync(this, …)`
   (`lib/Compiler.js:654`) → `this.readRecords(…)` (`lib/Compiler.js:657`) →
   `this.compile(onCompiled)` (`lib/Compiler.js:660`). (Watch mode instead enters via
   `watchRun`; not required.)
2. **`compile` creates params and a new `Compilation`.** `compile(callback)`
   (`lib/Compiler.js:1403`) calls `newCompilationParams()` (`Compiler.js:1404`)
   which builds a `NormalModuleFactory` (`createNormalModuleFactory`
   `Compiler.js:1374`) and a `ContextModuleFactory`; fires `hooks.beforeCompile`
   (`Compiler.js:1405`) and `hooks.compile` (`Compiler.js:1408`); then
   `newCompilation(params)` (`Compiler.js:1410`, def `Compiler.js:1361`) which calls
   `createCompilation` (`new Compilation(this, params)`, `Compiler.js:1353`) and
   fires `hooks.thisCompilation` and `hooks.compilation` (`Compiler.js:1365-1366`).
3. **The `make` phase builds the module graph from entries.** `compile` then fires
   `this.hooks.make.callAsync(compilation, …)` (`lib/Compiler.js:1415`).
   `EntryPlugin.apply` tapped `compiler.hooks.make` (`lib/EntryPlugin.js:47`) and
   calls `compilation.addEntry(context, dep, options, cb)` (`lib/EntryPlugin.js:48`).
   From there the L3 chain runs — `addEntry` → `addModuleTree` →
   `handleModuleCreation` → `factorizeModule`/`NormalModuleFactory.create` →
   `addModule` + `moduleGraph.setResolvedModule` → `buildModule`/`NormalModule.build`
   (runLoaders + parse) → `processModuleDependencies`, which recursively re-enters
   `handleModuleCreation` (`lib/Compilation.js:1832`) until every reachable module is
   built and wired into `Compilation.modules` / `moduleGraph`. (See the L3 key for
   per-stop cites; the L5 spine needs the make→entry→build→recurse shape, not every
   inner cite.) On completion `compile` fires `hooks.finishMake` (`Compiler.js:1420`)
   then `hooks.afterCompile` (`Compiler.js:1436`), and invokes `onCompiled`.
4. **`finish` runs the finish-modules phase.** `Compilation.finish(callback)`
   (`lib/Compilation.js:3171`) clears the factorize queue, then
   `this.hooks.finishModules.callAsync(modules, …)` (`lib/Compilation.js:3381`)
   runs plugins' last-chance module edits; afterward it extracts and reports module
   warnings/errors (`Compilation.js:3384-3460`). The module graph is frozen
   (`moduleGraph.freeze("dependency errors")`, `Compilation.js:3385`).
5. **`seal` converts the module graph into chunks, generates code, hashes, and
   produces assets.** `Compilation.seal(callback)` (`lib/Compilation.js:3473`) fires
   `hooks.seal` (`Compilation.js:3496`) and `hooks.beforeChunks` (`Compilation.js:3510`),
   creates entrypoints/chunks and connects entry modules (`Compilation.js:3512-3541`),
   then `buildChunkGraph(this, chunkGraphInit)` (`Compilation.js:3656`) builds the
   chunk graph (see the L4 key). It then runs the optimize stages
   (`hooks.optimize`, `optimizeModules`, `optimizeChunks`, `optimizeTree`,
   `optimizeChunkModules` — `Compilation.js:3661-3693`), assigns module/chunk ids
   (`Compilation.js:3696-3712`) and runtime ids (`Compilation.js:3716`), runs
   `optimizeCodeGeneration` (`Compilation.js:3737`), hashes modules
   (`createModuleHashes` `Compilation.js:3742`), runs `codeGeneration(…)`
   (`Compilation.js:3748`, def `Compilation.js:3911`), processes runtime
   requirements (`Compilation.js:3757`), and `createHash()`
   (`Compilation.js:3763`, def `Compilation.js:4797`). It then creates assets:
   `createModuleAssets` (`Compilation.js:3785`), `createChunkAssets(…)`
   (`Compilation.js:3841`), and `hooks.processAssets.callAsync(this.assets, …)`
   (`Compilation.js:3790`), followed by `hooks.afterProcessAssets`
   (`Compilation.js:3796`) and `hooks.afterSeal` (`Compilation.js:3826`). The
   `compilation.assets` map is now frozen (`Compilation.js:3810`).
6. **The compiler emits assets to the filesystem and reports done.** Back in
   `onCompiled` (`lib/Compiler.js:576`), the compiler calls
   `this.emitAssets(compilation, …)` (`lib/Compiler.js:595`; def `Compiler.js:746`).
   `emitAssets` fires `hooks.emit.callAsync(compilation, …)` (`lib/Compiler.js:1089`)
   and writes each asset to the output filesystem (the per-asset `assetEmitted` hook
   is `Compiler.js:201`), then `hooks.afterEmit` (after the emit loop). Control
   returns to `onCompiled`, which calls `this.emitRecords(…)` (`Compiler.js:623`,
   def `Compiler.js:1130`), builds a `new Stats(compilation)` and fires
   `this.hooks.done.callAsync(stats, …)` (`Compiler.js:607` in the shouldEmit branch
   / `Compiler.js:644` in the normal branch), stores build dependencies
   (`Compiler.js:637`), and finally `finalCallback(null, stats)`
   (`Compiler.js:641`) fires `hooks.afterDone` (`Compiler.js:563`) and invokes the
   user callback.

- Watch mode (`Watching`, `compiler.hooks.watchRun`, `hooks.invalid`) as the
  alternative entry to step 1. Plus.
- Records read/write (`readRecords` `Compiler.js:1130`-area, `emitRecords`
  `Compiler.js:1130`) and the `recordModules`/`recordChunks`/`recordHash`/`record`

  unsafe-cache path) threading make/build — credit, it is cross-cutting but not on
  the spine.

## Common misses / wrong turns

- Stopping at `seal` and never reaching `emitAssets` / the `done` hook (skips spine
  6) — the prompt explicitly says "until the final assets are written out."
- Treating `Compiler.run` and `Compilation.seal` as one blob and missing the
  `compile → newCompilation → make → finish → seal → emit` ordering. Order is
  gradeable.
- Putting code generation / hashing *before* `buildChunkGraph`. The chunk graph is
  built first (`Compilation.js:3656`), ids assigned, then codegen
  (`Compilation.js:3748`) and hashing (`Compilation.js:3763`).
- Confusing `createChunkAssets` (`Compilation.js:3841`, in-memory asset creation in
  `compilation.assets`) with `emitAssets` (`Compiler.js:746`, writing to disk). The
  spine distinguishes the two.
- Missing the make-phase bridge: `hooks.make` (`Compiler.js:1415`) → `EntryPlugin`
  → `addEntry` → the build chain. Without that, the "module graph from entries"
  step is hand-waved.
- Citing `Compiler.compile` as where compilation *runs*; it creates the
  compilation and fires `make`, but the actual building is driven by the make-hook
  taps (EntryPlugin et al.).

## Completeness scale

- **Full:** all 6 spine stages, in order — run/hooks → compile/newCompilation →
  make/EntryPlugin/addEntry/build chain → finish/finishModules →
  seal/buildChunkGraph/optimize/codegen/hash/assets → emitAssets/done — with cites,
  naming both the `Compiler`-side and `Compilation`-side owners of each stage and
  the hook that hands off between them.
- **Partial:** ≥4 of 6 in roughly correct order, but missing either the make/build
  bridge (spine 3) OR the emit/done tail (spine 6), or with the seal internal
  ordering wrong (codegen before chunk graph, etc.).
- **Miss:** only "Compiler.run creates a Compilation and outputs files" without the
  staged hook chain, or only one side (Compiler lifecycle OR Compilation seal)
  without the connection, or an out-of-order chain.

## Model answer

When `compiler.run(callback)` is called (`lib/Compiler.js:539`), it fires the run
lifecycle hooks — `hooks.beforeRun` (`Compiler.js:651`), `hooks.run`
(`Compiler.js:654`) — reads records, then calls `this.compile(onCompiled)`
(`Compiler.js:660`). `compile` (`lib/Compiler.js:1403`) builds the module/context
factories, fires `hooks.beforeCompile`/`hooks.compile` (`Compiler.js:1405-1408`),
creates a fresh `Compilation` via `newCompilation` (`Compiler.js:1361`/`1410`,
firing `hooks.thisCompilation` and `hooks.compilation`), and then fires
`hooks.make.callAsync(compilation, …)` (`Compiler.js:1415`). The make phase builds
the module graph: `EntryPlugin`'s make tap (`lib/EntryPlugin.js:47`) calls
`compilation.addEntry` (`lib/EntryPlugin.js:48`), which runs the L3 chain —
`addEntry` → `addModuleTree` → `handleModuleCreation` →
`factorizeModule`/`NormalModuleFactory.create` → `addModule` +
`moduleGraph.setResolvedModule` → `buildModule`/`NormalModule.build` (runLoaders +
parse) → `processModuleDependencies`, re-entering `handleModuleCreation`
(`Compilation.js:1832`) until the whole reachable graph is built. `compile` then
fires `hooks.finishMake`/`hooks.afterCompile` and calls `onCompiled`, which (after
`Compilation.finish` — `Compilation.js:3171` — runs `hooks.finishModules`
`Compilation.js:3381` and reports errors) proceeds to `Compilation.seal`
(`Compilation.js:3473`). `seal` fires `hooks.seal` (`Compilation.js:3496`), creates
entrypoints/chunks and connects entry modules (`Compilation.js:3512-3541`), calls
`buildChunkGraph(this, chunkGraphInit)` (`Compilation.js:3656`) to convert the
module graph into a chunk graph (async boundaries split into chunk groups), runs
the optimize stages and id assignment (`Compilation.js:3661-3716`), runs
`codeGeneration` (`Compilation.js:3748`) and `createHash` (`Compilation.js:3763`),
then produces assets via `createModuleAssets`/`createChunkAssets`
(`Compilation.js:3785`/`3841`) and `hooks.processAssets` (`Compilation.js:3790`),
freezing `compilation.assets`. Control returns to `onCompiled`, which calls
`this.emitAssets(compilation, …)` (`Compiler.js:595`; def `Compiler.js:746`),
firing `hooks.emit` (`Compiler.js:1089`) and writing each asset to the output
filesystem, then `this.emitRecords` (`Compiler.js:623`), builds a `Stats`, fires
`hooks.done` (`Compiler.js:607`/`644`), and `finalCallback(null, stats)`
(`Compiler.js:641`) fires `hooks.afterDone` and invokes the user callback. So one
build run is: `run` hooks → `compile`/new `Compilation` → `make`/entry build chain
→ `finish` → `seal` (chunk graph → codegen → hash → assets) → `emitAssets` →
`done`.
